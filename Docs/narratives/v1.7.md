## 1. 从 `mysql_source_delay` 倒推 IaC 完整性

**触发点**：`inventory/hosts.yml:49` 的 `mysql_source_delay: 3600`

**发现的问题**：按命名直觉应控制 MySQL 延迟复制窗口，但全仓库 grep 后没有任何 role / template 实际消费它。线上确实是延迟复制状态，但配置来自 v1.2 升级期间的手工初始化，自从 v1.3 把变量写进 inventory 之后，这个值就只起文档作用、不起代码作用。从这一行倒推，识别出 21 个同类 IaC 完整性问题——共性是线上服务正常，但新机器从零跑 `playbooks/site.yml` 起不来或起来后状态不一致。

**已解决 / 解决方法**：

- V1.7 → V1.8 准备期已修 10 条（规则文件去硬编码、服务地址从 group 派生、role 边界整理、`setup_backup.yml` 接入主 `site.yml` 等）
- 剩 11 条按严重度分级，排进 v1.8 / v1.9 / v1.10 三个版本路线

**决策分析**：线上正常 ≠ IaC 完整。项目早期路径是"先实现，再 IaC 化"——v1.1-v1.5 优先解决"配置下发能跑通"，IaC 完整性留白；v1.7 复盘期才系统性显式化这部分债务。修复不一次做完是因为按 ROI 分级：影响"site.yml 在新机器能跑通"的归严重必须修，影响"HA 切换正确性"的归中等，其他归轻量可不修。

**面试一句话版**（30s 内讲完）

> "我项目 inventory 里有一行 `mysql_source_delay: 3600`，按命名直觉应当控制 MySQL 延迟复制窗口，但 v1.7 全量复盘时我发现，全仓库没有任何 role 消费它——线上确实是延迟复制状态，但配置来自 v1.2 期间的手工初始化，这个值只起文档作用、不起代码作用。从这一行倒推，我在 v1.7 → v1.8 准备期一共识别并处理了 21 个同类 IaC 完整性问题，10 个两周内陆续修了，剩 11 个按 ROI 分级排进了 v1.8-v1.10 三个版本。这件事最大的收获是：**评估 IaC 完整性的锚点不是'线上服务是否正常'，而是'新机器从零跑 site.yml 能否还原现状'**。"

**可追问 Q&A**

| 追问 | 备答 |
|---|---|
| 你怎么保证以后不会再生长出第 22 条同类问题？ | v1.8 引入 `iac-prelint` Job 是基础门禁；更根本的是把"新机器能否跑通 site.yml"作为每次复盘的固定评估项。完整性不可能一次到位，但有了显式的评估锚点，新问题被识别的时机会从"全量复盘"前移到"每次复盘"。 |
| 为什么不一次性把 11 条全修了？ | 按 ROI 分级：影响"site.yml 能跑通"的归严重，前 3 条排 v1.8 优先做；影响"HA 切换正确性"的中等，排 v1.9；剩下 4 条线上无影响、属于"理论缺陷"，暂不修，等自然合并到 v1.11 节点接入时顺带解决。一份完整的 IaC 不是把所有状态都管起来，而是把"重建集群必须的状态"管起来。 |
| 这 10 条已修里最有代表性的是哪一条？ | 服务地址从手工硬编码改为 `hostvars[groups['db_master_nodes'][0]].ansible_host` 派生（commit `d175ec9`）。它解决的不是某一行配置，而是"事实有两份真相"这类隐性问题——节点 IP 调整时不再需要手动改服务地址，把"配置事实的唯一来源"原则落到 group_vars。 |
| 如果你是 leader，怎么避免新人在你的代码库里制造同类问题？ | 在 PR review checklist 里加一条"这次变更如果在新机器上跑 site.yml，是否还能起来"，并在 `iac-prelint` 阶段强校验；用门禁做兜底，不依赖个人记得做。 |

**可迁移的工程原则**

- **线上正常 ≠ IaC 完整**。评估 IaC 完整性的锚点是"新机器从零能否还原现状"，而不是"当前服务是否正常运行"。这两件事在 95% 时间里看起来一样，但在"换机器 / 故障重建"时差异致命。
- 一类隐性技术债往往不是单点失误，而是项目演进路径的副产品（本项目里是"先实现、再 IaC 化"的 V1.0→V1.4 顺序）；识别这类债务的方式是从一个具体细节倒推、不是从抽象规范出发。

## 2. IaC 自身缺 CI 质量门禁

**触发点**：每次请 AI 帮做代码 review，反复看到它自报"现在我来检查敏感内容" / "现在我来进行敏感关键词检查"——追踪后定位到 `.cursor/rules/60-git-and-commits.mdc` 里那段提交前手工 `grep` 自检条款，与 AI 进一步交互后意识到：这是一条规则文件里写好的"君子约定"，而不是"做不到就被拦下来"的强制门禁，本质等于把防线交给"提交者（人或 AI）记得执行"。

**发现的问题**：从这一条往上倒推 CI 链路，应用代码侧 CI 完整覆盖 build → push → deploy → smoke；IaC 代码侧只有 deploy + smoke，**缺 build 之前的 lint 与 scan**。`60-git-and-commits.mdc` 里那段防凭据泄漏的手工 `grep` 之所以"看起来像有防"，是因为它写在规则文件里——但规则文件描述的是"应该怎么做"，不是"做不到就被拦下来"，两者性质不同，混淆后就成了一道伪门禁。

**已解决 / 解决方法**：计划新增 Jenkins Job `iac-prelint`，覆盖 `yamllint` + `ansible-lint` + `gitleaks` + vault 加密状态校验（首行 `$ANSIBLE_VAULT;1.1;AES256`），任一失败即阻断 Pipeline，飞书通知作者。

**决策分析**：`ansible --check` 不是质量门禁——它验证的是"在目标节点上模拟执行 playbook 会改什么"，校验**部署结果**，不校验**代码本身质量**，两件事不能混。工具选 gitleaks 而非 detect-secrets：前者 Go 单二进制无依赖，CI 内运行更轻量。这件事的工程意义有两层：一是显式打破"IaC 不算代码"的隐性假设——很多中小厂运维实践里这条是缺的；二是显式区分"规则文件描述"与"CI 强制门禁"，前者是君子约定可以被疏忽跳过，后者是不可绕过的硬拦截，**写在规则里 ≠ 落到门禁上**。

**面试一句话版**（30s 内讲完）

> "我们 docker-infra 仓库的 CI 一开始覆盖的是'应用镜像构建 → 部署 → Smoke Test'这条链路，但 IaC 代码自身的提交质量是空白——靠规则文件里写的一段手工 grep 命令防泄漏，本质上等于没防。我做 v1.7 复盘时，把三条 Pipeline 的覆盖范围列在一起对照，才发现这个缺口，在 Jenkins 上加了一条 `iac-prelint` Job：`yamllint` + `ansible-lint` + `gitleaks` + vault 加密状态校验，任一失败就阻断 Pipeline、飞书通知作者。这件事的工程意义不在工具多复杂，而在**'IaC 也是代码、代码就要有质量门禁'，以及'规则文件描述 ≠ CI 强制门禁'——前者是君子约定可以被绕过，后者是不可绕过的硬拦截**。"

**可追问 Q&A**

| 追问 | 备答 |
|---|---|
| 为什么是 gitleaks 不是 detect-secrets？ | gitleaks 是 Go 单二进制无依赖；detect-secrets 是 Python 包，CI 里需要更多环境管理。两者规则能力接近，差异在部署体感——CI 内运行 gitleaks 更轻量。 |
| 为什么不用 pre-commit hook 在本地拦截？ | 本地 hook 可被 `--no-verify` 跳过（虽然规则禁止此参数，但属于君子约定）；CI 阶段是不可绕过的强制门禁。本地 hook 可以作为锦上添花的辅助，但不能取代 CI 门禁。 |
| 白名单怎么管？ | vault 文件本身、`*.example`、明确标注 placeholder 的位置走白名单；白名单也入 Git，由 PR review 把关；新增白名单条目需要 commit message 说清理由。 |
| 为什么不直接用 `ansible-playbook --syntax-check`？ | `--syntax-check` 只查 playbook 语法，不查 role 最佳实践（如未声明变量、handler 误用、`when` 表达式风险等）；`ansible-lint` 才覆盖最佳实践。两者不是替代关系，是互补关系。 |
| 你怎么发现这个缺口的？ | 不是在做架构设计时发现的，是反复观察到 AI review 自报"现在我来检查敏感内容"，好奇这条行为是被哪条规则触发的——追踪后定位到一段"伪门禁"。这条线索很特殊：是 AI 的执行行为暴露了规则与门禁的混淆。 |

**可迁移的工程原则**

- **规则文件描述 ≠ CI 强制门禁**：前者写的是"应该这么做"（君子约定），后者写的是"做不到就被拦下来"（硬拦截）。把君子约定写在显眼位置反而比"承认没有"更危险——它会让维护者潜意识里认为"已经有规范在管"。
- **IaC 也是代码，代码就要有质量门禁**。这一条在很多中小厂运维实践里是缺的——一旦把 IaC 仓库当成"配置文件集合"而不是"代码"，质量门禁就被默认豁免。

**索引**：`Docs/reviews/v1.7-iac-ci-quality-gate-gap.md`

## 3. 网关与业务前端的 volume 挂载耦合

**触发点**：`roles/nginx/templates/docker-compose.yml.j2:16` 的 `./ruoyi-ui-dist:/usr/share/nginx/ruoyi:ro`

**发现的问题**：网关容器 compose 文件持有业务知识——新增前端业务必须改这个 compose 并重启 Nginx。单业务时是合理简化，多业务接入后这层耦合会越来越显性。

**已解决 / 解决方法**：v1.12 网关迁移阶段**原样**搬迁到 gz-02，不在 v1.12 展开解耦；v1.13 K3s + Ingress 落地后，前端走"独立 Nginx 镜像 + Ingress"自然解耦。中间状态若有新增前端业务，按现有方式补 `conf.d/*.conf` + volume 即可，不为此提前重构网关 role。

**决策分析**：v1.12 本身已是大规模角色重整（gz-04 接副本、gz-02 接入口、gz-01 降 watchdog），网关迁移复杂度足够；硬塞解耦会扩大失败面。解耦最佳时机是 v1.13——前端容器化和 K3s 迁移天然是同一件事，再走一遍"前端独立 compose"中间状态没有 ROI。"识别问题但克制处理"比"为了显得专业越级重构"更能体现工程判断力。

**面试一句话版**（30s 内讲完）

> "我审 nginx role 时发现网关 compose 文件持有业务知识——新增前端业务必须改这个 compose 重启 Nginx。单业务时是合理简化，多业务接入后会越来越显性耦合。但我没有立刻解耦——v1.12 已经在做网关迁移和大规模角色重整，叠加解耦会扩大失败面；最佳解耦时机是 v1.13——K3s + Ingress 落地后前端走独立镜像 + Ingress 天然解耦。这个判断**比'识别问题'本身更重要**：识别问题谁都会，难在选择不修的时机——为了显得专业越级重构，对系统是负价值。"

**可追问 Q&A**

| 追问 | 备答 |
|---|---|
| v1.8-v1.12 中间状态如果新增前端业务怎么办？ | 按现有方式补 `conf.d/*.conf` + volume，不为此提前重构网关 role。同时把这条耦合记录到 nginx role 的 README / 待解耦清单，作为 v1.13 K3s 落地时的修复列表，避免遗漏。 |
| 为什么不能在 v1.12 就一起解耦？ | v1.12 本身已是大规模角色重整（gz-04 接 ruoyi-admin-2 副本、gz-02 接入口流量、gz-01 降 watchdog），网关迁移已经是复杂度上限。硬塞"compose 解耦"会扩大失败面，一旦回滚分不清是哪一层出问题。改造价值要和改造时机匹配。 |
| 你怎么判断"该修"和"该等"？ | 看下一次"自然合并到更大改造里"的时机什么时候到。v1.13 K3s 迁移本身就需要把前端容器化，再走一遍"前端独立 compose"作为中间状态没有 ROI——同样的代码会被写两遍丢一遍。 |
| 那如果 v1.13 K3s 落不下来怎么办？ | 这个 fallback 已经想过——如果 v1.13 延迟超过 6 个月，或者中间新增业务达到 3 个以上让耦合显性化，就提前在某个版本里独立做"前端 compose 解耦"。判断标准不是时间，是耦合实际造成的痛苦。 |

**可迁移的工程原则**

- **"识别问题但克制处理"是初级到中级工程师的关键判断**。越级重构看起来积极，实际是给系统增加失败面；理想时机是"问题被自然顺带解决"——前提是把这条耦合显式记录下来、不让它被遗忘。
- 改造价值要和改造时机匹配——同一个改造在 v1.12 是高风险叠加，在 v1.13 是顺带完成；技术上等价，工程上完全不同。

## 4. 从一次 Ruoyi 死 5 天无人感知，倒推入口探测的功能深度盲区

**触发点**：2026-05-20 21:25，用户登录 `https://ruoyi.jjmstart.com` 时前端弹出红框 "系统接口 502 异常"，但 Jenkins `ansible-deploy` 最近 5 天全绿、Prometheus `probe_success{service="ruoyi"}` 持续为 1、Alertmanager 没有任何告警。上节点执行 `docker ps | grep ruoyi` 才发现两个 `ruoyi-admin-*` 容器持续处于 `Restarting (1)` 且已经持续 5 天。

**发现的问题**：模板根因（commit `75d81a6`）很快定位——Jinja `{% for %}` 块在渲染 Compose 时吞掉换行，把 Redis Sentinel 端口和下一条环境变量拼到了同一行，Spring Boot 启动期 `NumberFormatException` 直接退出。但更值得复盘的是 **"为什么 CD 全绿、blackbox SUCCESS、5 天都没人感知"**：倒推发现 blackbox 探测的 `https://ruoyi.jjmstart.com/` 命中的是 Nginx 静态前端文件（`root /usr/share/nginx/ruoyi`），完全不经过后端 Java 进程；CD Smoke Test 又复用了这个探测——整条链路的"成功标准"停留在"入口层 HTTP 200"，从未触达"业务功能可用"。顺这条线索把 6 个 blackbox 探测目标全部过一遍，盘出三类盲区：① 前后端分离架构的天然盲区（仅 ruoyi）② proxy_pass 链路的伪健康（diary / jenkins / grafana）③ `/-/healthy` 端点的语义局限（prometheus / alertmanager）。另外发现一个放大器：当前只有主机级 node-exporter，没有 cAdvisor 等容器级指标采集，所以"容器持续 Restarting"这种本应秒级被告警捕获的故障，目前只能依赖入口探测间接发现——而入口探测又恰好命中第一类盲区，两个盲区相乘才出现"5 天无人感知"的现象。

**已解决 / 解决方法**：
- 模板根因已用 commit `75d81a6` 修复
- 三类盲区按 ROI 分级排进路线：
  - **P0**（v1.8 准备期）：给 ruoyi 增加业务级探测目标 `https://ruoyi.jjmstart.com/prod-api/captchaImage`（登录验证码接口，无需鉴权，必经过后端 + Redis），Smoke Test 查询改双源
  - **P1**（v1.9，与 HA 演练同期）：部署 cAdvisor，新增"容器 Restarting 持续 N 分钟"告警规则
  - **P2**（v2.0 可观测性增强）：jenkins / grafana / diary 各自补一个业务级探测路径
  - **P3**（**不修**）：Alertmanager dead-man-switch 与项目规模不匹配，明确写在 review § 4.3 暂不做的；如果未来真出过一次"告警链路全断没人发现"的事件再回头评估

**决策分析**：根因是 V1.4-V1.7 主线优先解决"流程从无到有"——Ansible 引入、CI/CD 引入、备份引入——每一步都聚焦"功能从无到有"，"流程的质量保障"留白。Smoke Test 复用 blackbox 本身是好原则（避免 Jenkins 直连业务节点 + 避免 CD 和监控告警结论不一致），但它隐含"入口探测语义 = 业务可用"——前后端分离稳定后这个前提已经失效，复用没有同步收紧。这是本审计与另外两份 v1.7 review 的边界互补：`v1.7-iac-completeness-audit.md` 是**产物层**问题（IaC 是否完整描述集群），`v1.7-iac-ci-quality-gate-gap.md` 是**流程层提交前**问题（代码是否经过 lint/scan），本审计是**流程层部署后**问题（部署是否真正可用）——三份合起来才是完整的 V1.7 流程盲区盘点。

**可迁移的工程原则**

- **每一道"返回 200"的探测都在验证某层壳的可达性，不是整条调用链的功能可用**。分清楚自己的探测在哪一层壳上，是决定告警链路有用还是假繁荣的关键。

## 5. Jenkins 探测 403 + meta refresh 的双层语义

**触发点**：自 2026-05-16 起 BlackboxProbeFailed (jenkins) 告警连续 5 天每隔几分钟向飞书推一条，但浏览器（无痕模式也一样）访问 `https://jenkins.jjmstart.com` 完全正常——告警与人工体验严重不一致。

**发现的问题**：curl 探测复现，发现 Jenkins 对匿名 GET `/` 返回的不是标准 `HTTP 302 Location: /login`，而是 **协议层 `HTTP/1.1 403 Forbidden` + 应用层 HTML body 里塞一段 `<meta http-equiv='refresh' content='1;url=/login?from=%2F'/>` + JS 兜底**。浏览器解析 body 跑 meta refresh，视觉上像被 302 重定向了；blackbox-exporter 只看协议层状态码，403 不在 `valid_status_codes: [200, 301, 302]` 白名单内 → `probe_success=0` → `for: 2m` 后告警 → 持续 firing。链路上每一层都"按设计行为正确"，但**协议层和应用层的语义被服务端故意拆成两份**，导致两类客户端看到的"事实"相反。

**已解决 / 解决方法**：探测目标从 `https://jenkins.jjmstart.com` 改为 `https://jenkins.jjmstart.com/login`（公开登录页，匿名 GET 直接 `200 OK`）。把根因——"403 + meta refresh 双层语义、为什么不放宽白名单"——以注释形式写在 `roles/monitor-stack/templates/prometheus.yml.j2` 中，留作未来维护者的现场上下文。Commit `49a644e`。这条本来在 `Docs/reviews/v1.7-ingress-probe-functional-depth-gap.md` § 4.1 P2 排在 v2.0；因告警风暴污染告警体系信任度，提前到 v1.7 修复期回填，review § 6 后续追踪表同步更新。

**决策分析**：三个备选方案中明确选了 `/login`，对比意图本身比选项更值得讲：

| 方案 | 表面效果 | 真实代价 |
|---|---|---|
| 把 403 加进 `valid_status_codes` 白名单 | 一行改完，告警立刻消失 | **等于把"未授权"也算成功**——未来 Jenkins 权限插件挂掉 / LDAP 断开真正返回 403 时告警就静默了。"为消除噪音而放宽告警语义"是典型反模式 |
| 给 blackbox 加 bearer/basic auth 绕过 403 | 探测真实首页 | 要管 token、写进 vault、轮换；并且不解决根本动机——我们只想知道"入口是否健康"，不需要"某 Job 能不能跑"（那是 v2.0 业务级探测） |
| **探测目标换 `/login`** | 一行改完，告警消失且语义合理 | 探测语义从"根路径可达"收紧到"登录入口可用"——更聚焦但仍覆盖 Jenkins 进程活 + Nginx 反代通 + TLS 有效 |

至于 Jenkins 为什么要把语义拆成双层：它的根路径同时承载 Web UI、REST API（`/api/json`）、CLI 调用。如果统一发标准 302 跳登录页，所有默认 follow-redirect 的 API 客户端（curl `-L`、Python `requests`）会被带到 200 + HTML 登录页，脚本认为请求成功，开始按 JSON parse HTML → 报莫名其妙错误。社区争议了十几年但从没改过，因为会破坏全世界 CI 脚本兼容性。

**面试一句话版**（30s 内讲完）

> "我项目里 Jenkins 的入口探测连续 5 天每隔几分钟告警一次，但浏览器访问 Jenkins 一切正常——告警和人工体验严重对不上。倒推根因发现 Jenkins 对匿名访问根路径返回的是 **协议层 HTTP 403 + 应用层 HTML body 里塞 meta refresh + JS 跳 /login** 的双层语义——浏览器解析 body 跟跳所以看着像 302 重定向，但 blackbox 只看协议层状态码，403 不在白名单里就持续告警。Jenkins 这种设计是故意的：根路径同时承载 Web UI 和 REST API，统一发 302 会让 follow-redirect 的 API 客户端把 HTML 当 JSON 解析。我没有把 403 加进白名单（**那等于把"未授权"也算成功，是为消除噪音而污染告警语义的反模式**），而是把探测目标改成 `/login`——公开登录页，匿名 GET 直接 200，探测语义从"根路径可达"收紧到"登录入口可用"，覆盖范围更聚焦。可迁移原则是：**HTTP 状态码和响应体是两层独立事实，便于人用的服务经常让两层互相矛盾——浏览器自动掩盖了，机器探测看不到。监控误报时先问'我是用什么读这封信的'**。"

**可追问 Q&A**

| 追问 | 备答 |
|---|---|
| 你怎么发现 403 + meta refresh 这套机制的？ | 第一轮怀疑 `Accept` 头协商——curl 加 `Accept: text/html` 实测还是 403，假设否定。第二轮 curl 把 body 拿下来看，发现里面是 `<meta http-equiv='refresh' content='1;url=/login?from=%2F'/>` + redirect.js，瞬间想通——浏览器解析 HTML 跟跳，blackbox 只看协议层。这种"行为正常但语义错位"的 bug 单靠看状态码定位不出来，必须把 body 也拿下来对照。 |
| 为什么不直接把 403 加进 `valid_status_codes`？ | 那等于把"未授权"也算成功。未来 Jenkins 真的因为权限插件挂掉、LDAP 断开、Realm 配置出错返回 403，告警就静默了——典型的"为消除噪音而牺牲语义"。**告警系统的价值不在于触发的次数少，在于触发的每一次都对应一个真实需要响应的事件**。 |
| 为什么不加 bearer/basic auth 让 blackbox 用账密登录探真实页面？ | 复杂度过高：要管 token、写进 vault、轮换；并且不解决根本动机——我们只想知道"Jenkins 入口是否健康"，不需要知道"某个 Job 是否能跑"。后者是 v2.0 业务级探测的事（review § 4.1 P2），跟入口探测分两层职责。 |
| Jenkins 这设计是不是 bug？为什么不发标准 302？ | 不是 bug，是兼容性历史包袱。根路径同时承载 Web UI、REST API（`/api/json`）、CLI 调用。如果统一发 302 跳登录页，curl/requests 的默认 follow-redirect 行为会把脚本带到 200 + HTML 登录页，脚本认为请求成功后按 JSON parse 报错。社区争议过几次都没改，因为会破坏全世界 CI 脚本。理解这一层后能反推：**遇到对 API 客户端友好但对浏览器需要"美化"的服务，状态码和 body 大概率有分层**。 |
| 这件事和叙事点 #4 (ruoyi 入口探测深度) 是一回事吗？ | 不是。#4 是"探测够浅 + 静态文件直返 = 看不到后端死活"——属于**探测目标位置选错**；#5 是"探测命中正确目标但服务把语义拆成协议层+应用层两份"——属于**探测协议层不够'懂'**。两个加起来才是入口探测盲区的完整图景：**选位置和选层次都不能错**。 |

**可迁移的工程原则**

- **HTTP 的"状态码"和"响应体"是两层独立事实，发件人可以让两层互相矛盾**。便于人用的服务（Jenkins、SSO 网关、某些反代鉴权、Spring Security 部分配置）经常这么干——浏览器自动掩盖了，机器探测看不到。监控误报时先问"我是用什么读这封信的"。
- **"为消除告警噪音而放宽告警条件"是反模式**。要么换探测目标到真公开的入口、要么让探测器学会读懂应用层语义；不能为了清屏污染语义——告警的价值不在触发次数少，在每次触发都对应真实需要响应的事件。

## 6. prometheus reload 6 版本潜伏失效 → IaC 验证心智

**触发点**：修复完叙事点 #5 的探测目标后，宿主机 `prometheus.yml` 确认更新，`ansible-playbook` 输出 `changed=1`、`reload prometheus` handler 也调用成功（`http_code=200`），但 5 分钟后再查 `probe_success{service="jenkins"}` 仍是 0，告警未消失。

**发现的问题**：`stat` 对比宿主机和容器内的同一文件：

```
宿主机 inode: 264071  Links: 1  Modify: 2026-05-21
容器内 inode: 263795  Links: 0  Modify: 2026-05-15
```

`Links: 0` 是核心信号——这个 inode 在宿主机文件系统已被 `unlink` 删除，但容器进程还持有它的句柄。根因是三件各自正确的设计叠加：

1. **Docker 单文件 bind mount 是 inode 级的**：`docker-compose.yml.j2:17` 用的是 `./prometheus/prometheus.yml:/etc/prometheus/prometheus.yml:ro`
2. **`ansible.builtin.template` 写文件是"写临时文件 → atomic rename"**：原子写保证半途崩溃不留半张配置，但 rename 会换 inode
3. **Prometheus `wget /-/reload` 让进程从自己的视角重读配置**：容器视角里还是旧 inode 的文件描述符，读到的是 5 天前的内容，"reload 成功"但什么都没变

隐性后果：**过去 6 个版本（v1.5 引入监控栈以来）所有 `prometheus.yml.j2` 改动都没真正生效过**，只是大家以为生效了。同 role 里 `rules/` 目录的 reload 不受影响，因为 `docker-compose.yml.j2:20` 挂的是 `./prometheus/rules` 整个**目录**，目录 bind mount 自动跟随子文件变化。

**已解决 / 解决方法**：拆 `reload prometheus` handler 为两个语义不同的：`reload prometheus rules`（轻量热 reload，给 rules 目录改动用，正确工作）和 `restart prometheus`（`community.docker.docker_compose_v2` recreate 容器，给主配置改动用，让 mount 跟新 inode）。`tasks/main.yml` 里两条 template 任务分别 notify 对应 handler。两条 handler 上方写了详细注释，包括"为什么 rules 不受影响"、"为什么不能用 unsafe_writes"，避免未来维护者重蹈覆辙。Commit `8b60f5b`。

**决策分析**：备选方案对比：

| 方案 | 思路 | 代价 |
|---|---|---|
| **拆 handler，主配置改用 restart** | 单文件 mount 不变，handler 改重建容器 | 保留 atomic 写；prometheus 重启窗口 ~5s。学习项目流量极低，acceptable |
| 把 `prometheus.yml` 也挂成目录 mount | 改 `docker-compose.yml.j2:17` 为 `./prometheus/:/etc/prometheus/:ro` | `prometheus/` 子目录下还有 `prometheus/data/` TSDB 数据库，配置 `:ro` 和 TSDB 写入冲突；要把 `data/` 挪出去，涉及备份脚本路径、role 文件结构连锁修改 |
| `ansible template` 加 `unsafe_writes: yes` | in-place 写文件，保留 inode | 失去原子写保证；半张配置文件可能让 Prometheus 启动失败 |

最终选 handler 拆分，因为改动范围最小、风险最可控。**再深一层的工程意义**：这条 bug 已经潜伏 6 个版本没被发现，不是因为运气好，而是因为没人系统验证过"我刚改的配置真的进容器了吗"——`ansible 输出 changed`、`reload 返回 200` 是"流程跑通"信号，不是"语义生效"信号。大部分 `prometheus.yml` 改动都是"加新 target / 调 scrape_interval"这种"加东西"类操作，没生效时表现是"指标不出现"，容易被误判为"还在抓取中"或"路径写错"；这次能抓到是因为它表现成"告警不消失"，比"指标不出现"显眼得多。**P0 真故障的副产品是会逼出长期潜伏的 latent bug**——快速修完故障然后收工，价值会少一半。

**面试一句话版**（30s 内讲完）

> "我修完 Jenkins 探测目标，跑了 ansible playbook，也手动触发了 prometheus reload，宿主机文件确认更新，但告警还在。stat 一查发现容器内 `/etc/prometheus/prometheus.yml` 还是 5 天前的 inode，`Links=0`——这个 inode 在宿主机已经被 unlink 删除了，容器进程还持有它的句柄。根因是 **Docker 单文件 bind mount + ansible template atomic rename + Prometheus wget reload** 三件凑一块的经典坑：bind mount 是 inode 级的、template 写临时文件再 rename 会换 inode、reload 只让 Prometheus 从自己的视角重读文件——**三件事各自都对，叠加起来导致过去 6 个版本所有 prometheus.yml 改动都没真正生效过，只是大家以为生效了**。修复是把 reload handler 拆成两个：rules 目录 mount 不受影响继续热 reload，主配置改用 docker compose restart 让 mount 跟新 inode。这件事最大的教训是：**绿色的 ansible 输出和 200 的 reload 调用是"流程跑通"信号，不是"语义生效"信号；验证 IaC 改动是否生效要看目标状态有没有出现，不是看动作执行成功**。"

**可追问 Q&A**

| 追问 | 备答 |
|---|---|
| 为什么不把 `prometheus.yml` 也改成目录 mount？ | `prometheus/` 子目录里有 `prometheus/data/` TSDB 数据库，配置 `:ro` 和 TSDB 写入冲突。改成目录 mount 要把 `data/` 挪出 `prometheus/` 子目录，对 role 文件结构、备份脚本路径、`.gitignore` 都有连锁影响——单点修一个 handler 拆分性价比明显更高。 |
| 为什么不用 `unsafe_writes: yes` 保留 inode？ | 失去原子写。in-place 写到一半 ansible 崩溃或网络断开，会留下半张配置文件，Prometheus reload 时直接 panic 或拒绝加载，监控栈彻底瘫。学习项目流量低不算严重，但 5s 重启 vs 半张配置的风险，5s 显然更可控。 |
| 这个 bug 6 个版本都没被人发现，怎么解释？ | 大部分 `prometheus.yml` 改动都是"加新 target / 调 scrape_interval"这种"加东西"类操作，没生效时表现是"指标不出现"，容易被误判为"还在抓取中"或"路径写错"，不会立刻怀疑 reload 机制。这次能抓到是因为这个 bug 表现成"告警不消失"，比"指标不出现"显眼得多。这也是 P0 真故障最有价值的副产品——**会逼出长期潜伏的 latent bug**。 |
| 你怎么验证修复真的生效？ | `stat` 容器内文件 inode 对比宿主机，`grep` 容器内文件内容确认是新内容，最后看行为——`probe_success{instance=".../login"}=1`。`ansible 输出 changed` 和 `prometheus reload 返回 200` 都不算验证：前者只能证明文件被写入宿主机，后者只能证明 prometheus 进程响应了请求。**要看的是"目标状态"，不是"动作的退出码"**。 |
| 这条经验能迁移到哪里？ | 任何"配置文件 + 热 reload + 单文件 mount"模式都要小心：Nginx reload、Envoy reload、HAProxy reload 同样有这个坑。一旦改成目录 mount 或 K8s ConfigMap，问题自然消失——K8s ConfigMap 实际是 `..data` 符号链接 swap 而不是文件覆盖，每次刷新都重新 symlink，绕开了 inode 缓存问题。这也解释了为什么 K8s 社区谈到"ConfigMap 热更新"时心智模型和 Docker 单文件 mount 完全不同。 |

**可迁移的工程原则**

- **绿色的 IaC 输出不等于语义生效**。验证 IaC 改动是否真正生效，应该看"目标状态有没有出现"而不是"动作执行成功"。`changed=1` + `reload 200` 全绿可能只意味着"动作跑完了"，与"配置在运行时进程里真正生效了"是两件事；无法直接区分时必须有正交手段（`stat` inode、`grep` 容器内容、行为观察）兜底。
- **多个各自正确的设计叠加，可能产生没人预料的失效模式**。bind mount 是 inode 级 + atomic rename 换 inode + 进程缓存文件描述符——任何一条单独看都没问题，三件凑一起就成了 6 版本潜伏 bug。**设计 review 时不仅看每个组件，还要看组件之间的接缝**。
- **真故障是发现 latent bug 的最佳触发器**。如果不是 Jenkins 告警这么显眼的 P0，prometheus reload 失效可能再潜伏好几个版本。把每次 P0 故障当成"做一次广角排查"的机会，比"快速修完了事"价值更高——一次故障如果只修了一个 bug，是浪费。

## 7. Jenkins CD 触发条件路径过滤：选层级的两次试错

**触发点**：2026-05-21 20:22 推完 `f97fa24 docs(workflow): centralize subdir specs in rules; trim READMEs to index` 后看 `ansible-deploy` Job 主页，build #65 正在跑 `PLAY [Deploy Redis master]`——一个**只动 `Docs/` 与 `.cursor/` 的纯文档 commit** 触发了全量 Ansible Check + Deploy + Smoke Test。回看最近 50 条 commit，`docs/style/(docs)` 前缀占 60%+，每条都跑同样的部署流程。识别出"docs commit 占 60%+ 但每条都跑部署"的噪音问题后**先选 SCM 层过滤作为方案 A**——20:38 在 Jenkins Job UI 配 Excluded Regions `Docs/.*` 等五条路径；20:43 推 `f90cefe` 做验证，本以为零构建记录，**结果 build #66 仍被触发**。打开 GitHub Hook Log 一行行看，才发现 SCM 层根本没起作用。

**发现的问题**：双层。

**第一层 — 触发粒度过粗**：CD 流水线对"任何 push 到 main 都跑全量部署"的策略让 60%+ 对运行时零影响的 commit（`Docs/`、`CHANGELOG.md`、`README.md`、`.cursor/` 等）都走完整 Ansible Check + Deploy + Smoke Test。两类成本叠加：① **构建历史信噪比恶化**——真正需要复盘的部署被淹没在 docs commit 触发的绿色构建里，真出问题时第一时间定位到对应 build 更慢；② **集群资源浪费**——每次都对所有节点发起 SSH + 全量 playbook 评估 + 30s smoke sleep，平均 2–3 分钟。

**第二层 — Jenkins SCM 层路径过滤在 webhook + ls-remote 模式下静默失效**。Build #66 的 GitHub Hook Log 给出了 smoking gun：

```
> git ls-remote -h -- git@github.com:jjmstart/docker-infra.git
[poll] Latest remote head revision on refs/heads/main is: f90cefe...
Done. Took 11 sec
Changes found
```

整个 polling 只调用了 `git ls-remote -h`，目的是对比"远端 HEAD SHA 和上次构建时是否一致"。它**没有**执行 `git log f97fa24..f90cefe --name-only` 这类拉 commit 列表的操作——所以 Excluded Regions 那五条正则没有任何东西可以匹配，自然没起作用。Jenkins Git plugin 在"webhook 触发 + 轻量 polling"模式下只比对 HEAD SHA，不下载 commit 对象，看不到改动了哪些文件。**更关键的是 UI 上配置时 Jenkins 不给任何警告**——它静静地接受、静静地不消费，让你以为生效了。这是典型的"配置存在 ≠ 行为发生"陷阱，和叙事点 #2 "规则文件描述 ≠ CI 强制门禁"是同一类思维盲点：把"我配了"等同于"它工作了"。

**已解决 / 解决方法**：方案 A 被实测否决后退回**方案 B — Pipeline 层早退 stage**。在 `Jenkinsfile` 的 `Checkout` 之后插入新 stage `Path Filter`，核心逻辑：

```groovy
stage('Path Filter') {
  steps {
    script {
      def prevSha = env.GIT_PREVIOUS_SUCCESSFUL_COMMIT
      if (!prevSha?.trim()) { return }   // fail-safe：首次构建放行

      def diffOut = sh(
        script: "git diff --name-only ${prevSha} HEAD 2>/dev/null || true",
        returnStdout: true
      ).trim()
      if (!diffOut) { return }            // fail-safe：拿不到 diff 放行

      def docPatterns = [
        ~/^Docs\/.*/, ~/^\.cursor\/.*/,
        ~/^CHANGELOG\.md$/, ~/^README\.md$/, ~/^\.gitignore$/,
      ]
      def files = diffOut.split('\n').findAll { it }
      def nonDocs = files.findAll { f -> !docPatterns.any { p -> f ==~ p } }

      if (nonDocs.isEmpty()) {
        currentBuild.result = 'NOT_BUILT'
        currentBuild.description = 'Skipped: docs-only change'
        env.SKIP_PIPELINE = 'true'
      }
    }
  }
}
```

后续 `Verify Ansible` / `Ansible Check` / `Deploy` / `Smoke Test` 四个 stage 的 `when` 块全部加 `expression { env.SKIP_PIPELINE != 'true' }` 门禁。docs-only push 走完 Path Filter 后所有后续 stage 全部 skip，整次构建 5 秒结束、显示灰色 `NOT_BUILT`，`post.success` / `post.failure` 都不触发（NOT_BUILT 不是 SUCCESS 也不是 FAILURE），不发飞书。Jenkins UI 上之前配的 Excluded Regions 同步删除——避免"两套机制并存、未来排查时不知道哪个在生效"。比对基准选 `GIT_PREVIOUS_SUCCESSFUL_COMMIT` 而非 `GIT_PREVIOUS_COMMIT`：如果上次构建是错误触发的 docs commit 部署，基准仍然停留在最后一次真正成功的状态，多个 docs commit 串联也能正确判定。

**决策分析**：本质问题不是"docs commit 要不要跑 Pipeline"，而是"消除 CI 噪音应该在哪一层做"。三个层级对比，加上**本次实测代价**列：

| 层级 | 实现方式 | 理论代价 | 本次实测代价 |
|---|---|---|---|
| SCM 触发层（方案 A） | Job 配置 Excluded Regions | 理论上零构建记录；Jenkins UI 配置漂移风险 | **静默失效**：webhook + ls-remote 轻量 polling 看不到改动文件 |
| **Pipeline 执行层（方案 B，最终采用）** | Jenkinsfile 加 Path Filter stage + `NOT_BUILT` | 仍产生构建记录（变灰但占位）；逻辑在 Git 仓库可审计 | 5 秒内退出；噪音从绿色变灰色、视觉降级一档；与 v1.10 IaC 方向一致 |
| Commit 约定层 | `[skip ci]` 关键字 | 依赖人为标注；和 Conventional Commits 叠加 | 未试 — 高频场景不适合走君子约定 |

回头看为什么方案 A 直觉上"明显更优"却失败，根因是把"Jenkins UI 提供这个配置项"等同于"它在我的配置下会生效"——这种"配置存在 = 行为发生"的假设在 Jenkins 这种插件化系统里特别危险，因为 UI 不会告诉你某条配置在当前触发模式下是否被消费。方案 B 的代价（保留灰色构建记录）实测下来反而是优点：① **逻辑在 Git 里可审计**——未来排查"为什么这条 commit 没触发部署"直接看 Jenkinsfile，不用进 Jenkins UI 翻 Job 设置；② **天然 IaC**——和 `Docs/scheme/phase-1-architecture-upgrade.md` v1.10 任务"CI/CD 状态 IaC 化"方向一致，不再制造新的"隐性 Jenkins UI 状态"；③ **可演进**——白名单想加路径直接改 Jenkinsfile + commit，比改 UI 留痕清晰。**和叙事点 #5（不能为消除告警噪音放宽白名单）形成对照**：那次噪音来自"探测目标真有问题但状态码反映不了"，放宽白名单等于把"未授权"也算成功——污染语义；本次噪音来自"docs 变更天然不需要 CI 验证"，过滤它们不污染任何东西。**判断噪音可否消除的锚点不是噪音多寡，是"被过滤掉的事件是否需要被响应"**——前者答案是"是"，所以必须修真问题；后者答案是"否"，所以可以且应当消除。

**面试一句话版**（30s 内讲完）

> "我项目 Jenkins ansible-deploy 流水线最近 50 条 commit 里 60%+ 是 docs 改动，每条都跑全量 Ansible Check + Deploy + Smoke Test，对运行时零影响。第一反应是在 SCM 层做路径过滤——Jenkins Job UI 上把 `Docs/.*`、`\.cursor/.*` 这些列进 Excluded Regions，配上去之后理论上 webhook 一到 Jenkins 就被拦下、零构建记录。结果推完 docs commit build #66 还是被触发，打开 GitHub Hook Log 一看 smoking gun——Jenkins 在 webhook 触发模式下只跑了 `git ls-remote -h` 比对 HEAD SHA，根本没拉 commit 列表，所以 Excluded Regions 那五条路径根本没东西可以匹配，**Jenkins UI 配上了但静默失效**。退回 Pipeline 层在 Jenkinsfile 里加 Path Filter stage，用 `GIT_PREVIOUS_SUCCESSFUL_COMMIT` 做基准、`git diff --name-only` 拿改动列表、五条 doc 路径正则全部命中就把 `currentBuild.result` 设成 `NOT_BUILT`、后续 stage 全部 skip。最大的收获是两层：**第一，CI 系统的健康指标是'信号/噪声比'，不是'构建绿色率'**——绿色但无意义的构建仍是噪音；**第二，'配置存在 ≠ 行为发生'**——尤其在插件化系统里，UI 让你配某项不代表那项在你的触发模式下会被消费，必须用实测验证生效，不能停在'我配了'就以为搞定了。"

**可追问 Q&A**

| 追问 | 备答 |
|---|---|
| 你怎么发现 SCM 层失效的？ | 推完 docs commit 看 Jenkins Job 主页，build #66 仍然出现——按预期不应该有这一行。然后进左侧 "GitHub Hook Log" 看 polling 详情，整段日志只有 `git ls-remote -h` 一个 git 命令，没有 `git log` 也没有 `git fetch`——拉 commit 列表的动作完全没发生。这时候才意识到 Excluded Regions 那五条正则没东西可比，UI 上配了等于没配。 |
| 为什么 Jenkins 不警告"这条配置在当前触发模式下不会生效"？ | 因为 Excluded Regions 是 Git plugin 的通用配置，理论上**在 SCM 轮询模式下**是生效的——只是大多数现代项目都用 webhook 触发，webhook + ls-remote 是轻量 polling，省去 fetch commit 的开销。Git plugin 不知道你打算用 webhook 还是 polling，所以不能在 UI 层报警。这是插件化系统的典型陷阱：**单个组件按设计行为正确，组合起来才暴露能力空洞**。 |
| 那为什么不强制 Jenkins 用"workspace-based polling"让 Excluded Regions 起作用？ | 一是 Git plugin 不同版本对这条选项的实现有差异，调通了未来升级可能失效；二是即使调通，每次 polling 都要在 Jenkins 控制器上跑 git fetch，比 ls-remote 慢且占磁盘；三是这条 UI 配置仍然不在 Git 里，与 phase-1 v1.10 任务"CI/CD 状态 IaC 化"方向相反。综合下来 Pipeline 层方案没那么"优雅"但更可靠也更顺向。 |
| 混合 commit（既改 `Docs/` 又改 `roles/`）怎么办？ | 正常触发。Path Filter 的语义是"所有改动文件**全部命中**才跳过"，混合提交里有任何非 docs 文件都会走完整 Pipeline，符合"动了基础设施就该验证"的直觉。 |
| 路径白名单怎么选才不会漏？ | 锚点是"该路径变更是否影响新机器跑 `site.yml` 的结果"。`Docs/` 是项目交付文档、`.cursor/` 是 AI 协作规则、`CHANGELOG.md` / `README.md` 是项目元数据、`.gitignore` 只影响 git 行为——都不被任何 role / playbook 引用。反例：`ansible.cfg`、`Jenkinsfile`、`inventory/`、`roles/`、`playbooks/`、`vault/` 都不能进——它们要么改变 Ansible 行为、要么改变 CI 逻辑本身、要么是运行时配置。 |
| 用 `[skip ci]` commit 约定不更灵活吗？ | 灵活但不可靠——它依赖**提交者记得标注**，是君子约定不是硬性机制。和叙事点 #2 "规则文件描述 ≠ CI 强制门禁"是同一类思维——一旦判断标准能被机械化（路径完全可枚举），就别留给人来记。`[skip ci]` 适合"极少数边界情况下的逃生口"，不适合处理 60%+ 的高频场景。 |
| 为什么基准选 `GIT_PREVIOUS_SUCCESSFUL_COMMIT` 而非 `GIT_PREVIOUS_COMMIT`？ | 防"基准被中间错误构建带歪"。如果上次构建是个错触发的 docs commit 部署，`GIT_PREVIOUS_COMMIT` 会指向那次，下一次 docs commit diff 只能看到自己改的文件，过滤判定仍正确；但如果中间有 N 次错误构建串联，基准会一路向后挪。`GIT_PREVIOUS_SUCCESSFUL_COMMIT` 始终锚定"最后一次真正成功的状态"——多个 docs commit 串联也能稳定地全部判为 docs-only，不会因为基准漂移而漏过。 |
| 这件事和叙事点 #5"不能为消除告警噪音而放宽白名单"矛盾吗？ | 不矛盾，是对照面。#5 那里"噪音"来自"探测目标真有问题但状态码反映不了"，放宽白名单等于把未授权也算成功——污染语义；本次"噪音"来自"docs 变更天然不需要 CI 验证"，过滤它们不污染任何东西。**判断噪音可不可以消除的锚点不是噪音多寡，是"被过滤掉的事件是否需要被响应"**——前者答案是"是"所以不能消，后者答案是"否"所以可以也应该消。两个原则方向相反但判断标准一致。 |

**可迁移的工程原则**

- **CI 系统的健康指标是"信号/噪声比"，不是"构建绿色率"**。每一次构建都应当对应一个**真实需要被验证的变更**；绿色但无意义的构建会污染历史、消耗资源、让真正失败时的告警更难被注意到。一条不该跑的绿色构建和一条该跑没跑的红色构建对系统价值同等负面。
- **"配置存在 ≠ 行为发生"**。尤其在插件化系统（Jenkins、Kubernetes admission controller、Nginx module、Spring Security filter 链等）里，UI / config 让你设某项不代表那项在你的运行模式下会被消费——单个组件按设计行为正确，组合起来才会暴露能力空洞。**配置完必须用实测验证生效**，不能停在"我配了"。和叙事点 #2 "规则文件描述 ≠ CI 强制门禁"、叙事点 #6 "`reload 200` ≠ 配置真正生效"是同一族陷阱——**判断"是否生效"的锚点永远是行为，不是配置存在或动作完成**。
- **判断噪音能否消除的锚点是"被过滤的事件是否需要被响应"**。是 → 必须修真问题（叙事点 #5 路径）；否 → 大胆消除噪音（本条路径）。**两者方向相反但判断标准一致**——不能把"消除噪音"或"保留告警"任何一条做成无条件原则。
- **在最合适的层级解决问题，但层级选择要被实测纠偏**。同一类问题在不同层（SCM 触发层 / Pipeline 执行层 / Commit 约定层 / 节点执行层）解决，代价、可靠性、可观测性、未来迁移成本完全不同。选层的初判依据是"判定能否机械化 + 是否需要保留可审计记录 + 频率"，但最终选哪一层要被实测验证——本次本来按这个判断标准最优选项是 SCM 层（机械化判定、不需要审计、高频），实测下来不可用、退到次优但可靠的 Pipeline 层，是正确的工程姿态。
