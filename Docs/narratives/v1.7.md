## 1. 从 `mysql_source_delay` 倒推 IaC 完整性

**触发点**：`inventory/hosts.yml:49` 的 `mysql_source_delay: 3600`

**发现的问题**：按命名直觉应控制 MySQL 延迟复制窗口，但全仓库 grep 后没有任何 role / template 实际消费它。线上确实是延迟复制状态，但配置来自 v1.2 升级期间的手工初始化，自从 v1.3 把变量写进 inventory 之后，这个值就只起文档作用、不起代码作用。从这一行倒推，识别出 21 个同类 IaC 完整性问题——共性是线上服务正常，但新机器从零跑 `playbooks/site.yml` 起不来或起来后状态不一致。

**已解决 / 解决方法**：

- V1.7 → V1.8 准备期已修 10 条（规则文件去硬编码、服务地址从 group 派生、role 边界整理、`setup_backup.yml` 接入主 `site.yml` 等）
- 剩 11 条按严重度分级，排进 v1.8 / v1.9 / v1.10 三个版本路线

**决策分析**：线上正常 ≠ IaC 完整。项目早期路径是"先实现，再 IaC 化"——v1.1-v1.5 优先解决"配置下发能跑通"，IaC 完整性留白；v1.7 复盘期才系统性显式化这部分债务。修复不一次做完是因为按 ROI 分级：影响"site.yml 在新机器能跑通"的归严重必须修，影响"HA 切换正确性"的归中等，其他归轻量可不修。

**面试一句话版**（30s 内讲完）

> "我项目 inventory 里有一行 `mysql_source_delay: 3600`，按命名直觉应当控制 MySQL 延迟复制窗口，但 v1.7 全量复盘时我发现，全仓库没有任何 role 消费它——线上确实是延迟复制状态，但配置来自 v1.2 期间的手工初始化，这个值只起文档作用、不起代码作用。从这一行倒推，我在 v1.7 → v1.8 准备期一共识别并处理了 21 个同类 IaC 完整性问题，10 个两周内陆续修了，剩 11 个按 ROI 分级排进了 v1.8-v1.10 三个版本。这件事最大的收获是：**评估 IaC 完整性的锚点不是'线上服务是否正常'，而是'新机器从零跑 site.yml 能否还原现状'**。"

**可追问 Q&A**

| 追问 | 备答 |
|---|---|
| 你怎么保证以后不会再生长出第 22 条同类问题？ | v1.8 引入 `iac-prelint` Job 是基础门禁；更根本的是把"新机器能否跑通 site.yml"作为每次复盘的固定评估项。完整性不可能一次到位，但有了显式的评估锚点，新问题被识别的时机会从"全量复盘"前移到"每次复盘"。 |
| 为什么不一次性把 11 条全修了？ | 按 ROI 分级：影响"site.yml 能跑通"的归严重，前 3 条排 v1.8 优先做；影响"HA 切换正确性"的中等，排 v1.9；剩下 4 条线上无影响、属于"理论缺陷"，暂不修，等自然合并到 v1.11 节点接入时顺带解决。一份完整的 IaC 不是把所有状态都管起来，而是把"重建集群必须的状态"管起来。 |
| 这 10 条已修里最有代表性的是哪一条？ | 服务地址从手工硬编码改为 `hostvars[groups['db_master_nodes'][0]].ansible_host` 派生（commit `d175ec9`）。它解决的不是某一行配置，而是"事实有两份真相"这类隐性问题——节点 IP 调整时不再需要手动改服务地址，把"配置事实的唯一来源"原则落到 group_vars。 |
| 如果你是 leader，怎么避免新人在你的代码库里制造同类问题？ | 在 PR review checklist 里加一条"这次变更如果在新机器上跑 site.yml，是否还能起来"，并在 `iac-prelint` 阶段强校验；用门禁做兜底，不依赖个人记得做。 |

**可迁移的工程原则**

- **线上正常 ≠ IaC 完整**。评估 IaC 完整性的锚点是"新机器从零能否还原现状"，而不是"当前服务是否正常运行"。这两件事在 95% 时间里看起来一样，但在"换机器 / 故障重建"时差异致命。
- 一类隐性技术债往往不是单点失误，而是项目演进路径的副产品（本项目里是"先实现、再 IaC 化"的 V1.0→V1.4 顺序）；识别这类债务的方式是从一个具体细节倒推、不是从抽象规范出发。

## 2. IaC 自身缺 CI 质量门禁

**触发点**：每次请 AI 帮做代码 review，反复看到它自报"现在我来检查敏感内容" / "现在我来进行敏感关键词检查"——追踪后定位到 `.cursor/rules/60-git-and-commits.mdc` 里那段提交前手工 `grep` 自检条款，与 AI 进一步交互后意识到：这是一条规则文件里写好的"君子约定"，而不是"做不到就被拦下来"的强制门禁，本质等于把防线交给"提交者（人或 AI）记得执行"。

**发现的问题**：从这一条往上倒推 CI 链路，应用代码侧 CI 完整覆盖 build → push → deploy → smoke；IaC 代码侧只有 deploy + smoke，**缺 build 之前的 lint 与 scan**。`60-git-and-commits.mdc` 里那段防凭据泄漏的手工 `grep` 之所以"看起来像有防"，是因为它写在规则文件里——但规则文件描述的是"应该怎么做"，不是"做不到就被拦下来"，两者性质不同，混淆后就成了一道伪门禁。

**已解决 / 解决方法**：计划新增 Jenkins Job `iac-prelint`，覆盖 `yamllint` + `ansible-lint` + `gitleaks` + vault 加密状态校验（首行 `$ANSIBLE_VAULT;1.1;AES256`），任一失败即阻断 Pipeline，飞书通知作者。

**决策分析**：`ansible --check` 不是质量门禁——它验证的是"在目标节点上模拟执行 playbook 会改什么"，校验**部署结果**，不校验**代码本身质量**，两件事不能混。工具选 gitleaks 而非 detect-secrets：前者 Go 单二进制无依赖，CI 内运行更轻量。这件事的工程意义有两层：一是显式打破"IaC 不算代码"的隐性假设——很多中小厂运维实践里这条是缺的；二是显式区分"规则文件描述"与"CI 强制门禁"，前者是君子约定可以被疏忽跳过，后者是不可绕过的硬拦截，**写在规则里 ≠ 落到门禁上**。

**面试一句话版**（30s 内讲完）

> "我们 docker-infra 仓库的 CI 一开始覆盖的是'应用镜像构建 → 部署 → Smoke Test'这条链路，但 IaC 代码自身的提交质量是空白——靠规则文件里写的一段手工 grep 命令防泄漏，本质上等于没防。我做 v1.7 复盘时，把三条 Pipeline 的覆盖范围列在一起对照，才发现这个缺口，在 Jenkins 上加了一条 `iac-prelint` Job：`yamllint` + `ansible-lint` + `gitleaks` + vault 加密状态校验，任一失败就阻断 Pipeline、飞书通知作者。这件事的工程意义不在工具多复杂，而在**'IaC 也是代码、代码就要有质量门禁'，以及'规则文件描述 ≠ CI 强制门禁'——前者是君子约定可以被绕过，后者是不可绕过的硬拦截**。"

**可追问 Q&A**

| 追问 | 备答 |
|---|---|
| 为什么是 gitleaks 不是 detect-secrets？ | gitleaks 是 Go 单二进制无依赖；detect-secrets 是 Python 包，CI 里需要更多环境管理。两者规则能力接近，差异在部署体感——CI 内运行 gitleaks 更轻量。 |
| 为什么不用 pre-commit hook 在本地拦截？ | 本地 hook 可被 `--no-verify` 跳过（虽然规则禁止此参数，但属于君子约定）；CI 阶段是不可绕过的强制门禁。本地 hook 可以作为锦上添花的辅助，但不能取代 CI 门禁。 |
| 白名单怎么管？ | vault 文件本身、`*.example`、明确标注 placeholder 的位置走白名单；白名单也入 Git，由 PR review 把关；新增白名单条目需要 commit message 说清理由。 |
| 为什么不直接用 `ansible-playbook --syntax-check`？ | `--syntax-check` 只查 playbook 语法，不查 role 最佳实践（如未声明变量、handler 误用、`when` 表达式风险等）；`ansible-lint` 才覆盖最佳实践。两者不是替代关系，是互补关系。 |
| 你怎么发现这个缺口的？ | 不是在做架构设计时发现的，是反复观察到 AI review 自报"现在我来检查敏感内容"，好奇这条行为是被哪条规则触发的——追踪后定位到一段"伪门禁"。这条线索很特殊：是 AI 的执行行为暴露了规则与门禁的混淆。 |

**可迁移的工程原则**

- **规则文件描述 ≠ CI 强制门禁**：前者写的是"应该这么做"（君子约定），后者写的是"做不到就被拦下来"（硬拦截）。把君子约定写在显眼位置反而比"承认没有"更危险——它会让维护者潜意识里认为"已经有规范在管"。
- **IaC 也是代码，代码就要有质量门禁**。这一条在很多中小厂运维实践里是缺的——一旦把 IaC 仓库当成"配置文件集合"而不是"代码"，质量门禁就被默认豁免。

**索引**：`Docs/reviews/v1.7-iac-ci-quality-gate-gap.md`

## 3. 网关与业务前端的 volume 挂载耦合

**触发点**：`roles/nginx/templates/docker-compose.yml.j2:16` 的 `./ruoyi-ui-dist:/usr/share/nginx/ruoyi:ro`

**发现的问题**：网关容器 compose 文件持有业务知识——新增前端业务必须改这个 compose 并重启 Nginx。单业务时是合理简化，多业务接入后这层耦合会越来越显性。

**已解决 / 解决方法**：v1.12 网关迁移阶段**原样**搬迁到 gz-02，不在 v1.12 展开解耦；v1.13 K3s + Ingress 落地后，前端走"独立 Nginx 镜像 + Ingress"自然解耦。中间状态若有新增前端业务，按现有方式补 `conf.d/*.conf` + volume 即可，不为此提前重构网关 role。

**决策分析**：v1.12 本身已是大规模角色重整（gz-04 接副本、gz-02 接入口、gz-01 降 watchdog），网关迁移复杂度足够；硬塞解耦会扩大失败面。解耦最佳时机是 v1.13——前端容器化和 K3s 迁移天然是同一件事，再走一遍"前端独立 compose"中间状态没有 ROI。"识别问题但克制处理"比"为了显得专业越级重构"更能体现工程判断力。

**面试一句话版**（30s 内讲完）

> "我审 nginx role 时发现网关 compose 文件持有业务知识——新增前端业务必须改这个 compose 重启 Nginx。单业务时是合理简化，多业务接入后会越来越显性耦合。但我没有立刻解耦——v1.12 已经在做网关迁移和大规模角色重整，叠加解耦会扩大失败面；最佳解耦时机是 v1.13——K3s + Ingress 落地后前端走独立镜像 + Ingress 天然解耦。这个判断**比'识别问题'本身更重要**：识别问题谁都会，难在选择不修的时机——为了显得专业越级重构，对系统是负价值。"

**可追问 Q&A**

| 追问 | 备答 |
|---|---|
| v1.8-v1.12 中间状态如果新增前端业务怎么办？ | 按现有方式补 `conf.d/*.conf` + volume，不为此提前重构网关 role。同时把这条耦合记录到 nginx role 的 README / 待解耦清单，作为 v1.13 K3s 落地时的修复列表，避免遗漏。 |
| 为什么不能在 v1.12 就一起解耦？ | v1.12 本身已是大规模角色重整（gz-04 接 ruoyi-admin-2 副本、gz-02 接入口流量、gz-01 降 watchdog），网关迁移已经是复杂度上限。硬塞"compose 解耦"会扩大失败面，一旦回滚分不清是哪一层出问题。改造价值要和改造时机匹配。 |
| 你怎么判断"该修"和"该等"？ | 看下一次"自然合并到更大改造里"的时机什么时候到。v1.13 K3s 迁移本身就需要把前端容器化，再走一遍"前端独立 compose"作为中间状态没有 ROI——同样的代码会被写两遍丢一遍。 |
| 那如果 v1.13 K3s 落不下来怎么办？ | 这个 fallback 已经想过——如果 v1.13 延迟超过 6 个月，或者中间新增业务达到 3 个以上让耦合显性化，就提前在某个版本里独立做"前端 compose 解耦"。判断标准不是时间，是耦合实际造成的痛苦。 |

**可迁移的工程原则**

- **"识别问题但克制处理"是初级到中级工程师的关键判断**。越级重构看起来积极，实际是给系统增加失败面；理想时机是"问题被自然顺带解决"——前提是把这条耦合显式记录下来、不让它被遗忘。
- 改造价值要和改造时机匹配——同一个改造在 v1.12 是高风险叠加，在 v1.13 是顺带完成；技术上等价，工程上完全不同。

**索引**：`Docs/scheme/phase-1-architecture-upgrade.md` v1.12 任务背景与面试叙事点（commit `5f84734`）

## 4. 从一次 Ruoyi 死 5 天无人感知，倒推入口探测的功能深度盲区

**触发点**：2026-05-20 21:25，用户登录 `https://ruoyi.jjmstart.com` 时前端弹出红框 "系统接口 502 异常"，但 Jenkins `ansible-deploy` 最近 5 天全绿、Prometheus `probe_success{service="ruoyi"}` 持续为 1、Alertmanager 没有任何告警。上节点执行 `docker ps | grep ruoyi` 才发现两个 `ruoyi-admin-*` 容器持续处于 `Restarting (1)` 且已经持续 5 天。

**发现的问题**：模板根因（commit `75d81a6`）很快定位——Jinja `{% for %}` 块在渲染 Compose 时吞掉换行，把 Redis Sentinel 端口和下一条环境变量拼到了同一行，Spring Boot 启动期 `NumberFormatException` 直接退出。但更值得复盘的是 **"为什么 CD 全绿、blackbox SUCCESS、5 天都没人感知"**：倒推发现 blackbox 探测的 `https://ruoyi.jjmstart.com/` 命中的是 Nginx 静态前端文件（`root /usr/share/nginx/ruoyi`），完全不经过后端 Java 进程；CD Smoke Test 又复用了这个探测——整条链路的"成功标准"停留在"入口层 HTTP 200"，从未触达"业务功能可用"。顺这条线索把 6 个 blackbox 探测目标全部过一遍，盘出三类盲区：① 前后端分离架构的天然盲区（仅 ruoyi）② proxy_pass 链路的伪健康（diary / jenkins / grafana）③ `/-/healthy` 端点的语义局限（prometheus / alertmanager）。另外发现一个放大器：当前只有主机级 node-exporter，没有 cAdvisor 等容器级指标采集，所以"容器持续 Restarting"这种本应秒级被告警捕获的故障，目前只能依赖入口探测间接发现——而入口探测又恰好命中第一类盲区，两个盲区相乘才出现"5 天无人感知"的现象。

**已解决 / 解决方法**：
- 模板根因已用 commit `75d81a6` 修复
- 三类盲区按 ROI 分级排进路线：
  - **P0**（v1.8 准备期）：给 ruoyi 增加业务级探测目标 `https://ruoyi.jjmstart.com/prod-api/captchaImage`（登录验证码接口，无需鉴权，必经过后端 + Redis），Smoke Test 查询改双源
  - **P1**（v1.9，与 HA 演练同期）：部署 cAdvisor，新增"容器 Restarting 持续 N 分钟"告警规则
  - **P2**（v2.0 可观测性增强）：jenkins / grafana / diary 各自补一个业务级探测路径
  - **P3**（**不修**）：Alertmanager dead-man-switch 与项目规模不匹配，明确写在 review § 4.3 暂不做的；如果未来真出过一次"告警链路全断没人发现"的事件再回头评估

**决策分析**：根因是 V1.4-V1.7 主线优先解决"流程从无到有"——Ansible 引入、CI/CD 引入、备份引入——每一步都聚焦"功能从无到有"，"流程的质量保障"留白。Smoke Test 复用 blackbox 本身是好原则（避免 Jenkins 直连业务节点 + 避免 CD 和监控告警结论不一致），但它隐含"入口探测语义 = 业务可用"——前后端分离稳定后这个前提已经失效，复用没有同步收紧。这是本审计与另外两份 v1.7 review 的边界互补：`v1.7-iac-completeness-audit.md` 是**产物层**问题（IaC 是否完整描述集群），`v1.7-iac-ci-quality-gate-gap.md` 是**流程层提交前**问题（代码是否经过 lint/scan），本审计是**流程层部署后**问题（部署是否真正可用）——三份合起来才是完整的 V1.7 流程盲区盘点。

**面试一句话版**（30s 内讲完）

> "我项目里有一次 ruoyi 后端两个实例全死了 5 天没人发现——CD 全绿、Prometheus 全绿，用户登录时才报 502。倒推根因：CD Smoke Test 复用了 blackbox 入口探测，而 blackbox 探测的 `https://ruoyi.jjmstart.com/` 命中的是 Nginx 静态前端文件，根本不经过后端。整条链路的'成功标准'停留在'入口层 HTTP 200'，没触达'业务功能可用'。我顺这条线索把 6 个 blackbox 探测目标全过了一遍，盘出三类盲区——前后端分离的天然盲区、proxy_pass 链路的伪健康、`/-/healthy` 的语义局限——按 ROI 分级排进 v1.8 / v1.9 / v2.0 三个版本，**P0 只动一行：给 ruoyi 加一个 `/prod-api/captchaImage` 业务级探测**，就能消掉本次故障类型 80% 的风险。这件事最大的教训不是'监控没用'，而是**每一道'返回 200'的探测，本质上都在验证某层壳的可达性，不是整条调用链的功能可用——分清楚是什么，决定了告警是真有用还是假繁荣**。"

**可追问 Q&A**

| 追问 | 备答 |
|---|---|
| 为什么不一次性把所有 blackbox 目标都改成业务级探测？ | 按 ROI 分级。前后端分离架构是 ruoyi 独有的天然盲区，性价比最高；其他三个域名是"功能内部异常"才会漏，触发概率明显低，且每个业务级探测路径都需要单独评估（鉴权、副作用、幂等性），不适合一次性铺开。延后做的都记进了 phase-1 路线，不会被忘掉。 |
| 容器都死了 5 天，难道 docker daemon 不告警吗？ | docker daemon 本身没有"容器持续 Restarting 触发告警"的能力——它只按 `restart: unless-stopped` 反复拉起。要捕获这类故障需要容器级指标采集（cAdvisor 或 docker_exporter），我项目目前只有主机级 node-exporter。这是 P1 修复项，排进 v1.9 和 HA 演练一起做。 |
| 你说 Smoke Test 复用入口探测有"隐含前提"，那当初为什么要复用？ | 复用本身是对的——避免 Jenkins 直连业务节点带来的"网络问题误判为业务故障"，也避免 Smoke Test 和监控告警结论不一致。问题在于"复用什么样的探测"——v1.6 落地时探测目标是 v1.5 留下的入口首页，那个目标的语义在 v1.5 阶段还可以等同于"业务可用"，但到了前后端分离稳定的 v1.7 已经不成立，复用没有同步收紧。正确的演进是**复用 Prometheus 这个聚合点不变，但 blackbox 探测目标要随架构演进调整**。 |
| 既然识别出 3 类盲区，为什么 P3 直接判 "不修"？ | "不修"不是"忽视"，是和项目规模、求职定位的匹配判断。Alertmanager dead-man-switch 需要第二个独立健康检查源（或第二套 Alertmanager），对初级运维项目复杂度过高、面试讲不深；与其搭一套讲不清的东西，不如把"不修的理由 + 触发条件"明确写进 review，等真出过一次"告警链路全断没人发现"再评估。**克制处理比为了显得专业越级修复更有工程价值。** |
| 这件事和你 v1.7-iac-completeness-audit 是一回事吗？ | 不是。IaC 完整性审计关注"新机器从零跑 site.yml 能不能起来"，是**产物层**问题；CI 质量门禁审计关注"已提交代码有没有被 lint/scan"，是**流程层提交前**问题；本审计关注"已部署代码有没有被验证业务可用"，是**流程层部署后**问题。三份合起来才是 V1.7 期间识别的完整流程盲区——根因都是"V1.4-V1.7 主线优先解决流程从无到有，质量保障留白"。 |

**可迁移的工程原则**

- **每一道"返回 200"的探测都在验证某层壳的可达性，不是整条调用链的功能可用**。分清楚自己的探测在哪一层壳上，是决定告警链路有用还是假繁荣的关键。
- **复用基础设施的同时要随架构演进同步重新评估其隐含前提**。"复用 X 做 Y" 是好原则，但 X 的语义可能随架构演进失效；复用不是一次性决策，是需要随时回头复查的承诺。
- **盲区往往不是单点失误，是放大器与根因的叠加**。本次"5 天无人感知"是"入口探测深度不够"（根因）和"容器级监控缺失"（放大器）的乘积；只修任一层都不够，但根因的优先级永远高于放大器。
