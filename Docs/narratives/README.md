# narratives/ — 面试叙事素材集

本目录沉淀**从 retrospectives / reviews / drills 等深度复盘文档中抽取的可面试叙事切片**。每个版本一份文件，聚合该版本期间产出的所有"可在面试中讲清工程判断力"的叙事点。与 reviews / retrospectives 不同，本目录的写作视角是**面试现场**——每条叙事都给出一句话版本、可追问 Q&A 与可迁移的工程原则，便于直接演练。

## 与其他文档的边界

| 维度 | `retrospectives/vX.Y-retrospective.md` | `reviews/v{触发版本}-{topic}.md` | `narratives/vX.Y.md` |
|---|---|---|---|
| 写作视角 | 复盘单次演进 | 诊断跨版本盲点 | 面向面试现场表达 |
| 触发时机 | 单次架构升级完成后 | 发现盲点时 | reviews / retrospectives 落盘后整理 |
| 组织维度 | 按版本 | 按主题 | 按版本聚合多个叙事 |
| 是否独立结论 | 是 | 是 | 否——所有结论都来自 reviews / retrospectives，本目录只做切片与重写 |
| 颗粒度 | 一份文档对应一次演进 | 一份文档对应一个主题 | 一份文档聚合 3-7 条叙事点 |

`narratives/` 不引入新的事实——所有素材都来自 reviews / retrospectives / drills；本目录的价值在"重写为面试现场可用的语态"。

## 文件命名

`vX.Y.md`

- 每个架构版本一份文件，聚合该版本期间产出的所有叙事点
- 不按主题拆分（即使一个版本产出 5+ 条叙事，也放在同一文件内，方便整体演练）
- 与 reviews/ 的"一主题一文件"约定不同，narratives/ 是"一版本一文件"

示例：

- `v1.7.md`
- `v1.8.md`（待 v1.8 复盘后产出）

## 单篇文档建议结构

```markdown
# VX.Y 面试叙事点

> 配套使用：作为 phase-2-job-hunting.md 任务 1（vX.Y 复盘交付物验收）的录音自述清单

## 速查表（顶部）

| # | 叙事点 | 重量级 | 体现的工程能力 | 触发场景 |

## 1. <叙事点 1 标题>

**触发点**：（从哪一行 / 哪个具体细节开始）
**发现的问题**：（顺着触发点倒推出的问题）
**已解决 / 解决方法**：（实际怎么处理的）
**决策分析**：（为什么这么处理，不那么处理的代价）
**面试一句话版**：（30s 内能脱稿讲完的精炼版本）
**可追问 Q&A**：（提前演练的 2-4 个追问与备答，表格化）
**可迁移的工程原则**：（脱离本项目语境也成立的一般化结论）
**索引**：（来源 review / retrospective / drill 路径）
```

## 写作约束

- **不引入新事实**：narratives/ 是切片与重写，所有事实都应能在 reviews / retrospectives / drills 里找到出处
- **面试一句话版必须是脱稿口语**：写成书面语就失去意义；避免长定语从句、避免书面专有名词堆叠
- **Q&A 不是装饰**：每条追问都应当是面试中实际可能被问到的，备答要能在 30s 内讲完
- **可迁移的工程原则**要去本项目化：写"评估 IaC 完整性的锚点是新机器能否还原现状"，不要写"我项目里 mysql_source_delay 这个变量"
- 一条叙事如果无法被压缩成 30s 一句话版本，说明它要么是两个叙事被混在一起、要么本身不够清晰——回头去 reviews / retrospectives 重新拆解

## 与 phase-2-job-hunting.md 的衔接

`Docs/scheme/phase-2-job-hunting.md` 任务 1（v1.X 复盘交付物验收）的"录音自述"流程以本目录为清单来源：

- 每条叙事至少录一遍 2-3 分钟版本（面试一句话版 → 最可能被追问的 Q&A → 可迁移的工程原则）
- 录音回放时如果出现"嗯…那个…文档里写了…"卡壳，回原始 review / retrospective 补完后再录第二遍
