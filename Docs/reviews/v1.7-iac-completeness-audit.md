## 1. 问题识别的起点

V1.7 全量复盘期间，重新阅读 `inventory/hosts.yml` 时注意到 `bj-01` 主机变量段中有一行：

```yaml
mysql_source_delay: 3600
```

按命名直觉，这应当是控制 MySQL 延迟复制窗口的变量。但在 `roles/mysql-replica/`、`roles/mysql-master/`、`playbooks/` 全仓库搜索，**没有任何 task 或 template 消费它**：

- 既不写进 `docker-compose.yml.j2`（合理：它本来就不是 mysqld 启动参数，而是复制通道参数 `CHANGE REPLICATION SOURCE TO ... SOURCE_DELAY=3600`）
- 也不写进任何 SQL 任务（不合理：当前 role 没有任何 SQL 任务）

紧接着登录 `bj-01` 执行 `SHOW REPLICA STATUS\G`，确认线上 `SQL_Delay=3600` 确实生效。说明：

- 这条复制通道是**早期手工初始化**的（V1.2 升级期间，见 `Docs/runbooks/v1.1-to-v1.2.md`）
- `mysql_source_delay` 这个变量自 V1.3 进入 inventory 后，**只起文档作用，不起代码作用**
- 当前 IaC 实质上不能保证"新机器从零复现现在的集群状态"

从这一行变量倒推下去，逐个 role / template 排查后，识别出 11 个**仍待修复**的同类盲点（详见 §2）；同时回顾 V1.7 发布到本审计期间（2026-05-07 → 2026-05-19，约 1.5 周）的 commit 历史，发现已经有 10 个同类 IaC 完整性问题被陆续修复（详见 §3）。两部分合计 21 个 IaC 完整性问题在同一时间窗口被识别 / 处理。它们的共性是：**线上服务正常运行，但若不处理，新机器重建集群时 Ansible 不能保证还原现状**。

## 2. 系统性盘点

按"新机器跑完 `playbooks/site.yml` 后服务能否起来"作为严重度划分依据。

### 2.1 严重（新机器跑 site.yml 后服务起不来）

| 编号 | 盲点 | 现象 | 影响范围 |
|------|------|------|---------|
| #1 | MySQL 用户与 schema 不在 Ansible 管 | `roles/mysql-master/tasks/main.yml` 只渲染目录 + `.env` + `docker-compose.yml`，没有任何 SQL 任务 | 应用 / Exporter / ProxySQL 全连不上 |
| #2 | MySQL 复制通道不在 Ansible 管 | `mysql_source_delay` 未消费；没有 `CHANGE REPLICATION SOURCE TO ...` 流程 | 新从库即使建出来也接不上主库 |
| #3 | Docker 网络 `global_gateway` 没人创建 | 几乎所有 compose 都 `external: true`，但全仓库无 `docker network create global_gateway` | 新节点 `docker compose up` 第一步就报 `network global_gateway not found` |
| #4 | Jenkins 内部状态完全不在 IaC | 自定义镜像 `jenkins-ansible:latest` 的 Dockerfile 不在仓库；插件、Credentials、Job、`jenkins_home` 都没纳入备份链路 | `bj-01` 一旦挂了，整个 CD 流水线要从零重建 |

**#1 细化清单**（应建未建的 MySQL 用户与数据库）：

| 对象 | 出现位置 | 用途 |
|------|---------|------|
| `ry-vue` 数据库 | `roles/ruoyi/templates/docker-compose.yml.j2:16` | 业务库；应用直连 `jdbc:mysql://${DB_HOST}:${DB_PORT}/ry-vue` |
| `repl_user` | `Docs/runbooks/v1.1-to-v1.2.md` 多处 `CHANGE REPLICATION SOURCE TO ... SOURCE_USER='repl_user'` | 复制账号，需要 `REPLICATION SLAVE` 权限 |
| `mysql_exporter_user` | `roles/mysqld-exporter/templates/mysqld-exporter.cnf.j2:2` | mysqld_exporter 采集 MySQL 指标 |
| `proxysql_monitor_user` | `roles/proxysql/templates/proxysql.cnf.j2:22` | ProxySQL 探活后端 MySQL |
| `ruoyi_datasource_username` | `roles/proxysql/templates/proxysql.cnf.j2:61` | 应用通过 ProxySQL 访问 MySQL 时的账号 |

### 2.2 中等（影响 HA / 切换正确性）

| 编号 | 盲点 | 现象 | 影响范围 |
|------|------|------|---------|
| #5 | Redis `replicaof` 静态化 | `roles/redis-replica/templates/redis.conf.j2:8` 硬编码 `replicaof {{ redis_master_host }}` | Sentinel 故障切换后任何一次 redis role 重跑都会把新 master 错误降级 |
| #6 | Sentinel 运行时状态被覆盖 | `sentinel.conf` 通过 bind mount 挂载，Sentinel 启动后会向 conf 文件追加 `sentinel known-replica`、`current-epoch` 等运行时状态，Ansible 重渲染会清掉 | 一般 Sentinel 能恢复，但故障切换正在进行中重跑就有风险 |
| #7 | ProxySQL 配置改动推不到 runtime | `proxysql.cnf` 只在 `datadir` 首次启动时被读取，之后所有配置在 SQLite runtime DB 中；当前 role 只渲染 cnf + `docker_compose_v2 recreate`，cnf 改动不会生效 | 修改 `mysql_servers` / `mysql_users` / `mysql_query_rules` 全推不动 |

### 2.3 轻量（容易补、影响可控）

| 编号 | 盲点 | 现象 |
|------|------|------|
| #8 | `mysql-restore-test` 恢复演练只是手动流程 | V1.7 retrospective 提到 34 秒 RTO 是手动操作；没有 Ansible playbook 或 Jenkins Job 自动化 |
| #9 | Prometheus / Grafana 数据卷与 `jenkins_home` 不在备份链路 | V1.7 备份只覆盖 MySQL `ry-vue`；Grafana 仪表板、Prometheus TSDB、`jenkins_home`（最关键）都没备份 |
| #10 | 节点系统时区 / NTP 未显式管理 | `mysql_backup.sh` cron `0 2 * * *` 按主机本地时区执行；`docker-daemon` role 不管 timezone，依赖云厂商默认设置；多云节点可能各自有偏差 |
| #11 | 云资源层（Tailscale 接入 / OSS Bucket / DNS / 安全组）依赖控制台手工配置 | 不属于 Ansible 自然管辖范围，但要做"一套配置复现集群"，必须正面回答：用 Terraform？还是文档化"云资源前置依赖清单"？ |

## 3. V1.7 → V1.8 准备期已完成的 IaC 治理优化

§2 列出的是 V1.7 全量复盘时**仍待修复**的盲点。在 V1.7 发布到本审计完成期间（2026-05-07 → 2026-05-19），仓库其实已经陆续修复了 10 个同类问题。把它们与 §2 放在同一份审计文档里，是为了呈现完整的"识别 — 已修 — 待修 — 排期"闭环，也为后续 commit 链路追溯留底。

### 3.1 规则层 IaC 治理（去硬编码节点事实、建立动态查询入口）

| Commit | 解决的同类盲点 |
|---|---|
| `e97fe64` | 新增 `.cursor/rules/15-current-facts.mdc` 作为动态事实查询入口，AI 不再把规则文件里的示例（节点 IP / 角色 / 服务分布）当作权威事实，统一指向 `inventory/hosts.yml` 与最新架构快照 |
| `0c9581b` | `00-project-context.mdc` / `10-docs-workflow.mdc` 去除硬编码的四节点表与 IP / 规格 / 延迟，改为 `<region>-<NN>` 命名约定 + inventory 指针 |
| `d578515` | `01 / 20 / 30 / 40 / 50` 五份规则文件统一去硬编码，红线与示例不再钉在某一版本快照 |

这一组的本质是把"规则文件"也视作 IaC 的一部分。规则文件中硬编码节点事实，等同于"配置事实有两份真相"，是隐性 IaC 完整性问题。

### 3.2 配置层 IaC 治理（硬编码 → group / host_vars 派生）

| Commit | 解决的同类盲点 |
|---|---|
| `d175ec9` | `inventory/group_vars/all.yml` 中 MySQL / Redis / ProxySQL 等服务地址从手工硬编码改为按主机分组派生（`hostvars[groups['db_master_nodes'][0]].ansible_host`）。节点 IP 调整时不再需要手动改服务地址 |
| `dcc9669` | `roles/docker-daemon/templates/daemon.json.j2` 的 `insecure-registries` 改为按主机变量驱动，registry 宿主迁移时只需改 inventory 一处 |

这一组是把"事实的真相只放一处"原则贯彻到 `group_vars` / `host_vars`。

### 3.3 role 与 playbook 边界整理

| Commit | 解决的同类盲点 |
|---|---|
| `1e85509` | 把 `mysqld-exporter` 从 `monitor-stack` 中拆出独立 role。原先两个 exporter 混在监控栈里，违反"role 职责对应单一组件"约束 |
| `070e55d` | `setup_diary.yml` 从 `setup_ops.yml` 拆出，diary 部署链路独立，不再混在 ops 编排里 |
| `4c43171` | diary 静态资源（`memos_icon.png`）改由 diary role 自身分发；不再靠 nginx role 持有跨组件资产 |
| `f07f0e6` | `setup_backup.yml` 接入 `site.yml` 主流程。V1.7 新增的备份 playbook 之前游离在主流程外，存在"忘了跑"的风险 |

这一组的核心是 role / playbook 的职责边界对齐"谁拥有这个资源"。它和 §4 中"role 职责边界过窄"的根因分析对应——只是这次是边界**整理**，不是边界**扩展**。

### 3.4 局部 bug 修复

| Commit | 解决的同类盲点 |
|---|---|
| `22b2306` | `roles/nginx/handlers/main.yml` 修复 reload 在 compose 重启前执行的 handler 顺序 bug。属于"看似幂等、实际有时序依赖"的隐性问题 |

## 4. 根因分析

§2 的 11 个待修盲点与 §3 的 10 个已修问题，根因是共通的——不是某一次失误，而是项目演进路径与早期设计哲学的副产品。

### 4.1 项目演进路径：先实现，再 IaC 化

V1.0 → V1.4 是"先把服务跑起来，再把 Ansible 套进来管"的顺序：

- V1.0：手工部署所有服务，Docker Compose 直接在节点上跑
- V1.1：监控栈迁移，仍是手工操作
- V1.2：MySQL 主从 + ProxySQL，**所有 SQL 操作都是手工执行**（`CREATE USER`、`CHANGE REPLICATION SOURCE TO ...`、ProxySQL Admin SQL 灌入），运维成果以 runbook 形式保存，没进 Ansible
- V1.3：`/opt/docker` 纳入 Git（仅配置文件）
- V1.4：引入 Ansible，重构为 `roles/` + `inventory/` + `playbooks/`，**但 role 的职责边界定义为"管 compose 栈与配置文件"**

V1.4 的 IaC 切入点选择决定了：所有 V1.2 之前已经手工完成的"一次性数据层初始化"动作，从未进入 Ansible 的视野。

### 4.2 role 的职责边界过窄

各个 role 的 tasks 高度同构：

1. 创建目录
2. 渲染 `.env`
3. 渲染 `docker-compose.yml`
4. `community.docker.docker_compose_v2` 拉起

这套模式覆盖了 90% 的"无状态服务"管理需求，但对以下两类对象失效：

- **数据层期望状态**：MySQL 的用户、库、复制通道，本质上是 mysqld 进程内的状态，不在文件系统里
- **运行时配置 DB**：ProxySQL 的 SQLite、Redis Sentinel 的 conf 自动追加段、Jenkins 的 `jenkins_home`

`mysql_source_delay` 写进 inventory 但 role 不消费它，本质上是因为没有人新增过"管理 MySQL 复制通道期望状态"这一类 task。V1.6 retrospective 已经识别到 `mysql-replica` 不管复制线程状态这个问题，做了局部修复（追加 `START REPLICA` 自动恢复），但没把"复制通道本身的配置"纳入 IaC。§3.3 中 `mysqld-exporter` 拆 role 和 diary 模块化，本质上也是这个根因的局部修复——但只覆盖了"边界整理"，没覆盖"边界扩展"。

### 4.3 备份范围控制下的有意识取舍

V1.7 的 proposal 明确把范围限定在"MySQL 业务库逻辑备份 + 恢复演练"，不做：

- Binlog/PITR
- Redis 备份
- Jenkins 状态备份
- 监控数据备份

这个范围控制本身合理（v1.7 retrospective Q/A 已论证），但副作用是：**整个集群的"可重建性"被默认假设为"基础设施层 + MySQL 业务数据"，运维平台自身（Jenkins）、监控数据这两个维度被排除在外**，而这两个维度恰好是新机器重建时最难恢复的部分。

## 5. 修复方案与排期

修复路径已纳入 `Docs/scheme/phase-1-architecture-upgrade.md`，对应 v1.9 / v1.10 / v1.11 / v1.12 / v1.13 五个连续版本（版本号在 2026-05-26 因 DMS 插入新 v1.8 整体顺延一格）。下表给出盲点编号到修复版本的映射。

| 盲点编号 | 修复版本 | 修复方式 |
|---------|---------|---------|
| #1 | v1.9 | `roles/mysql-master/` 新增幂等 SQL 初始化任务（`CREATE DATABASE IF NOT EXISTS`、`CREATE USER IF NOT EXISTS`、`GRANT`），覆盖业务库 + 复制账号 + Exporter 账号 + ProxySQL 监控账号 + 应用账号 |
| #2 | v1.9 | `roles/mysql-replica/` 校验复制通道期望状态；新增受保护的 bootstrap playbook（要求 `mysql_replica_bootstrap_confirm=true`），用于新机器 / 空数据目录初始化复制通道，使用 `SOURCE_DELAY={{ mysql_source_delay \| default(0) }}` |
| #3 | v1.9 | `roles/docker-daemon/` 或独立基础设施 role 中管理 `global_gateway` 网络创建，幂等 |
| #5 #6 | v1.11 | 拆分"初始拓扑 bootstrap"与"运行期健康检查"两类任务；Ansible 重跑不无条件覆盖 Sentinel 已选出的主从拓扑 |
| #7 | v1.10 | `roles/proxysql/` 升级为"配置文件 + Admin SQL 双路径管理"，每次变更后执行 `LOAD ... TO RUNTIME` + `SAVE ... TO DISK` |
| #4 | v1.12 | 将 `jenkins-ansible:latest` Dockerfile 纳入 Git；引入 JCasC 或 Job DSL 管理三个核心 Job；`jenkins_home` 进入备份策略 |
| #9（部分） | v1.12 | `jenkins_home` 备份链路并入 |
| #8 | 暂不修 | 演练自动化收益低于其他优先项；保留为手动流程，runbook 即可 |
| #9（剩余） | 暂不修 | Grafana / Prometheus 数据卷价值远低于 `jenkins_home`，可重建；待后续按需补 |
| #10 | 暂不修 | 多云节点时区漂移目前未观察到实际偏差；待 v1.13 节点接入时统一管理 |
| #11 | 暂不修 | 云资源层超出 Ansible 自然管辖范围；待 v1.13 节点接入时以"云资源前置依赖清单"文档化，不引入 Terraform |

修复不修的"留白"理由：项目目标是"找到目标岗位 offer + 入职后能正常干活"，不是"做到 100% IaC 完整"。当前 11 个盲点中，前 7 个直接戳"集群可重建"这一 IaC 核心命题；剩余 4 个属于"线上无影响、面试可叙事化兜底"，按 ROI 不修。

## 6. 面试叙事点

这次审计本身就是高 ROI 的面试素材。可讲的三个角度：

### 6.1 如何从一个变量发现整个 IaC 体系的设计漏洞

> "我们项目里 inventory 中有一个 `mysql_source_delay: 3600` 变量，按命名直觉应当控制 MySQL 延迟复制窗口，但我在做 V1.7 全量复盘时发现，全仓库没有任何 role 消费它。线上确实是延迟复制状态，但配置来自早期手工初始化，自从 V1.3 把变量写入 inventory 后，这个值就只起文档作用、不起代码作用。从这一行倒推下去，我在 V1.7 → V1.8 准备期一共识别并处理了 21 个同类 IaC 完整性问题：其中 10 个在两周内陆续修了（包括规则文件去硬编码、服务地址从 group 派生、role 边界整理、setup_backup 接入主 site.yml 等），剩 11 个排进了 v1.9-v1.12 等版本的路线图。"

### 6.2 如何区分"线上影响"和"理论缺陷"

> "这 11 个待修盲点严格来说没有一个会影响线上服务运行——线上服务都跑得好好的、监控正常、备份恢复演练实测 34 秒。它们的影响只会在'换机器'或'故障重建'时暴露。但我把它们记下来，并按严重度排了序：影响 'site.yml 在新机器能否跑通'的归为严重；影响 'HA 切换正确性' 的归为中等；其他归为轻量可暂不修。"

### 6.3 如何给修复排期

> "这些问题我做了取舍：前 7 个排进了 v1.9 / v1.10 / v1.11 / v1.12 四个连续版本，分别是基础可复现性修复、ProxySQL HA 化、Redis Sentinel 边界修复、CI/CD 状态 IaC 化；剩下 4 个暂不修，理由是 ROI 不够。一份完整的 IaC 平台不是把所有状态都管起来，而是把'重建集群必须的状态'管起来。"
