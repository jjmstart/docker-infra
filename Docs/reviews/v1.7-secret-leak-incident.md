## 1. 问题识别的起点

本次事件发生在仓库公开前的最后一道安全检查：准备将 `docker-infra` 从 private 改为 public 前，先用 `gitleaks v8.21.2` 对 Git 全历史进行扫描。扫描结果显示，早期归档目录 `Archive/` 中保留了 3 处真实生产凭据；这些凭据不是废弃样例，而是截至发现时仍在生产环境使用。

泄漏位置集中在两类历史配置中：

- `Archive/backend/proxysql/proxysql.cnf`：包含 ProxySQL monitor 用户密码和 Ruoyi MySQL 业务用户密码。
- `Archive/backend/redis/conf/sentinel.conf`：包含 Redis Sentinel 连接 Redis 主从时使用的认证密码。

另外，`Docs/runbooks/v1.0-to-v1.1.md` 中 `curl -u` 示例使用了 `admin:your_password` 占位符，被 `gitleaks` 的 `curl-auth-user` 规则识别为疑似凭据。该项不是实际密码，但如果不处理，仓库公开后会持续产生噪音，削弱后续 secret scanning 告警的可信度。

本次处理按"先轮换、后清理历史、最后公开"执行：在生产凭据失效前不做 Git 历史重写，避免出现"历史已改但线上仍依赖旧密码"的中间风险。

## 4. 根因分析

### 4.1 `Archive/` 归档时没有经过敏感扫描

`Archive/` 的初衷是保留早期手工配置的演进证据，方便追溯 V1.0-V1.3 阶段从手工 Compose 到 Ansible IaC 的迁移过程。问题在于，早期配置文件是生产现场的真实配置，归档时只完成了"移动到 Archive"这一文件组织动作，没有执行 secret scan 和脱敏处理。

这说明"归档"和"可公开归档"不是同一件事：前者保留历史上下文，后者还必须满足凭据、IP、账号、运维入口等公开边界检查。`Archive/` 可以保留，但进入 Git 历史前必须先脱敏。

### 4.2 IaC 化前的明文配置被原样继承

V1.4 引入 Ansible 后，敏感信息逐步收敛到 `vault/secrets.yml`，但 V1.4 之前已经存在的明文配置并没有被系统性反扫。结果是：当前线上 role/template 已经遵循 vault 管理，但 Git 历史里仍保留着 IaC 化前的明文遗留物。

这类问题不会在日常部署中暴露，因为当前 HEAD 的生产模板可能已经是安全的；只有做全历史扫描、仓库公开或合规审计时才会被发现。

### 4.3 质量门禁停留在规则层，没有进入 CI 阻断层

`Docs/reviews/v1.7-iac-ci-quality-gate-gap.md` 已经指出：当前敏感信息检查主要依赖规则文件和人工自觉，而不是 Jenkins 中强制失败的 `gitleaks` stage。本次事件正是该盲点的真实样本。

如果 `gitleaks` 早在 `Archive/` 归档 commit 前作为 CI 门禁运行，真实凭据会在进入 Git 历史前被阻断；如果只在公开前才扫描，就只能通过轮换 + history rewrite 做事故响应。

## 6. 核心工程结论

- **仓库公开前必须做全历史 secret scan，而不是只扫 HEAD**。真实泄漏往往藏在早期 commit、归档目录或曾经删除的文件里；只扫当前工作树无法证明公开安全。
- **真实凭据事故的处置顺序是先轮换、后清理历史、最后公开**。history rewrite 只能降低未来暴露面，不能让已经在线使用的旧密码自动失效；轮换才是风险解除动作。
- **`Archive/` 目录可以保留，但不能成为敏感信息例外区**。归档文件的历史价值不等于可以保留明文；正确做法是脱敏后保留结构和语义。
- **误报也要治理**。持续存在的误报会让 secret scanning 变成噪音源，最终降低真实告警被认真处理的概率。
- **安全规则必须最终落到自动化门禁**。规则文件适合描述原则，CI 才能提供阻断；任何依赖人工记得执行的检查，都应视为临时防线。
