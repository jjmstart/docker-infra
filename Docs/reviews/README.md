# reviews/ — 主题型横向复盘

本目录用于沉淀**跨版本、按主题**展开的深度复盘。触发时机不再是"完成某次架构升级"，而是"在阅读代码 / 排障 / 评审过程中突然识别到一个之前没意识到的盲点"。这类复盘通常涉及多个版本、多个组件，按主题（IaC 完整性、状态管理边界、HA 设计、CI/CD 韧性等）组织，而不是按版本号。

## 与 `retrospectives/` 的边界

| 维度 | `retrospectives/vX.Y-retrospective.md` | `reviews/<version>-<topic>.md` |
|------|----------------------------------------|--------------------------------|
| 触发时机 | 单次架构升级**完成后** | 任意时刻，**发现盲点**时 |
| 组织维度 | 按版本 | 按主题 |
| 内容主线 | 该版本踩了什么坑、为什么这么选 | 当前架构有哪些隐性盲点、为什么之前没看见 |
| 时间范围 | 单一版本内 | 可横跨多个版本 |
| 是否随版本发布 | 是（v1.X 完成后写） | 否（偶发性，按需写） |

## 与 `proposals/` 的边界

- `proposals/` 是**面向未来**的设计草案，回答"下一步打算做什么"
- `reviews/` 是**面向当前**的诊断报告，回答"当前架构现在有哪些隐性问题"
- 一份 review 通常**会**催生若干 proposal；但 review 本身不写实施细节，只识别问题

## 文件命名

`v{触发版本}-{英文 slug}.md`

- `触发版本`：识别这次盲点时集群所处的最新版本号（标记时间锚点，便于追溯）
- `英文 slug`：英文短主题，便于 grep 与 CLI 操作

示例：

- `v1.7-iac-completeness-audit.md`
- `v1.8-ha-state-boundary-review.md`
- `v1.10-cicd-resilience-audit.md`

## 单篇文档建议结构

每篇 review 至少包含以下章节，可按主题增减：

### 章节 1：文档元数据（文件开头）

```markdown
> **触发时间**：YYYY-MM-DD
> **触发版本**：vX.Y
> **触发点**：一句话说清楚从什么具体细节开始倒推的
> **状态**：识别中 / 已规划修复 / 已部分修复 / 已完结
> **关联文档**：链接到对应的 proposals / runbooks / retrospectives / phase-1 路线版本号
```

### 章节 2：问题识别的起点

讲清楚"我从哪个具体细节开始注意到这个问题"。这一节是**故事化**的，叙事性强，是面试中讲清楚"工程判断力"的核心素材。

### 章节 3：系统性盘点

把同类问题在整个仓库里全部找出来。表格化，按严重程度排序。每条至少给出：

- 现象
- 影响范围
- 严重等级（严重 / 中等 / 轻量）

### 章节 4：根因分析

为什么这类盲点会存在？是设计阶段没考虑？是历史遗留？还是范围控制下的有意识取舍？根因要说清楚技术原理或决策过程，不能只写"配置错误"。

### 章节 5：修复方案与排期

哪些必须修、哪些可以延后、哪些不修的理由。引用对应的 `Docs/scheme/phase-1-architecture-upgrade.md` 路线版本号，避免与正式路线脱节。

### 章节 6：面试叙事点

本篇 review 能在面试中怎么讲。和 `retrospectives/` 的"技术决策 Q/A"风格一致，但更强调"如何识别盲点"而非"如何选型"。

### 章节 7：后续追踪

修复落地后回填实际结果；新发现的同类盲点追加。

## 写作约束

- **不写实施细节**：review 是诊断报告，不是 runbook 或 proposal
- **不重复 retrospective 的内容**：单版本踩坑放在 retrospective，跨版本盲点放在 review
- **不捏造**：所有盲点必须有具体文件 / 行号 / 命令依据，不允许"印象中应该是"
- **状态可流转**：review 的状态字段随修复进度更新，但**不删除历史内容**，让追溯链路保留
