## 1. 问题识别的起点

### 1.1 起点：CD 全绿、监控全绿、用户却看到 502

2026-05-20 21:25 左右登录 `https://ruoyi.jjmstart.com`，前端弹出红框 "系统接口 502 异常"。第一反应是去 Jenkins 看部署历史——`ansible-deploy` 最近 5 天没有失败 build，最近一次部署的飞书通知是 "✅ Smoke Test 通过"；再去 Prometheus / Alertmanager 看告警，`probe_success{service="ruoyi"}` 持续为 1，没有任何告警。所有自动化监控信号都说"健康"。

到节点上手工查才发现真相：

```
# gz-02
ruoyi-admin-2   100.118.69.78:5000/ruoyi:latest   Restarting (1) 53 seconds ago

# gz-03
ruoyi-admin-1   100.118.69.78:5000/ruoyi:latest   Restarting (1) 10 seconds ago
```

两个后端容器**全部死了 5 天**，`docker logs` 显示 Spring Boot 启动期 `NumberFormatException`，根因是 Compose 模板里 Jinja `{% for %}` 块在渲染时吞掉了换行，把 Redis Sentinel 端口和下一条环境变量拼到了同一行。模板根因已用 commit `75d81a6` 修复。

但更值得复盘的不是"模板写法"——是 **"为什么这次故障在 CD pipeline 通过、blackbox 持续 SUCCESS、5 天都没人感知"**。

### 1.2 顺着证据链一路倒推

`Jenkinsfile` 里 Smoke Test 查询的是：

```bash
curl -sf "http://100.117.7.75:9090/api/v1/query" \
  --data-urlencode 'query=min(probe_success{service="ruoyi"})'
```

`roles/monitor-stack/templates/prometheus.yml.j2` 里这个 service 对应的探测目标是：

```yaml
- targets:
    - https://ruoyi.jjmstart.com
  labels:
    service: ruoyi
    severity: critical
```

`roles/monitor-stack/templates/blackbox.yml.j2` 里 `http_2xx` 模块的成功标准是：

```yaml
valid_status_codes: [200, 301, 302]
```

`roles/nginx/templates/conf.d/ruoyi.conf` 里 `/` location 是：

```nginx
location / {
    root   /usr/share/nginx/ruoyi;
    index  index.html index.htm;
    try_files $uri $uri/ /index.html;
}
```

链路闭环：**blackbox 访问 `https://ruoyi.jjmstart.com/` 命中 Nginx 静态前端文件，根本不经过后端 Java 进程**。两个 ruoyi-admin 容器死光，只要 Nginx 进程在、`/usr/share/nginx/ruoyi/index.html` 还在磁盘上，blackbox 就返回 200，整条 CD + 监控链路一致认为 ruoyi 健康。用户看到的 502 来自前端登录页加载完之后的 `/prod-api/login` 接口请求——但那条路径**从来没有任何自动化探测覆盖过**。

至此整个盲点闭环清晰：**CD pipeline 的"成功标准"和 blackbox 探测的"健康定义"都停留在"入口层 HTTP 200"，没有触达"业务功能可用"这一层**。

## 2. 系统性盘点

### 2.1 6 个 blackbox 探测目标的功能深度矩阵

对照 `prometheus.yml.j2` 当前 `blackbox-http` 全部 6 个目标，按"进程死 / 进程活但功能死 / 进程活且功能可用"三层分析其覆盖能力：

| 探测目标 | 探测路径 | Nginx / 端点处理方式 | 后端 / 功能故障能发现吗 | 严重等级 |
|---|---|---|---|---|
| `ruoyi.jjmstart.com` | `/` | **静态文件直返**（`root /usr/share/nginx/ruoyi`） | 否，后端死光也是 200 | **严重（本次踩坑）** |
| `diary.jjmstart.com` | `/` | `proxy_pass {{ diary_host }}:5230` | 进程死能发现；进程活但内部异常未覆盖 | 中等 |
| `jenkins.jjmstart.com` | `/` | `proxy_pass {{ jenkins_host }}:8080` | 进程死能发现；登录页 200 ≠ 插件 / Job 功能可用 | 中等 |
| `monitor.jjmstart.com` | `/` | `proxy_pass {{ grafana_host }}:3000` | 进程死能发现；登录页 200 ≠ Prometheus 数据源连通 | 中等 |
| `prometheus` | `/-/healthy` | Prometheus 官方健康端点 | 进程死能发现；scrape targets 全掉线 / 规则加载失败不反映 | 中-低 |
| `alertmanager` | `/-/healthy` | Alertmanager 官方健康端点 | 进程死能发现；飞书 webhook 失效 / 路由规则错误不反映 | 中-低 |

### 2.2 三类不同的盲区

把上表抽象出三类共性，作为本审计的核心盘点结论：

| 类别 | 涉及目标 | 盲点本质 |
|---|---|---|
| **第一类：前后端分离架构的天然盲区** | `ruoyi` | 前端静态文件被 Nginx 直接托管，根 URL 不触达后端任何代码；当前架构里**唯一一个**符合"后端完全死光、入口探测依然 200"的服务 |
| **第二类：proxy_pass 链路的伪健康** | `diary` / `jenkins` / `grafana` | 进程死能被 502 捕获，但应用进程活着 + 关键功能挂掉（数据源断、插件失败、底层文件锁）时，登录页 / 首页本身不依赖那些功能，照样返回 200 |
| **第三类：`/-/healthy` 的语义局限** | `prometheus` / `alertmanager` | 官方健康端点的语义是"进程能响应请求"，不包括"采集链路全断"、"规则文件解析失败"、"出口 webhook 失效"、"路由匹配规则把告警送进黑洞" |

### 2.3 监控自身的盲区放大效应

`docker run` 选项里 `restart: unless-stopped` + 启动期失败的组合，会让容器进入"启动 → 报错 → 退出 → 等几秒 → 再启动"无限循环，宿主机 docker daemon 不告警；而当前监控栈：

- **node-exporter** 只看主机级 CPU / 内存 / 磁盘，不看容器状态
- **没有部署 cAdvisor 或等价的容器级指标采集**，因此 `docker_container_restart_count` / `docker_container_state` 这类指标根本不在 Prometheus 里
- 告警规则里没有任何针对"容器 Restarting 持续 N 分钟"的规则

所以一个本应秒级被告警捕获的故障（"两个生产容器在反复 Restarting"），在当前监控体系下**只能依赖入口层 blackbox 探测间接发现**——而入口层探测又因为 §2.1 第一类盲区命中不到这个故障。两个盲区叠加，故障持续了 5 天才被人工发现。

## 3. 根因分析

### 3.1 Smoke Test 复用入口探测的隐含前提

V1.6 引入 CD Smoke Test 时（见 `Docs/architecture/v1.6.md` 与 `.cursor/rules/30-jenkinsfile.mdc`），明确写了一条原则：**禁止 Jenkins 直接 curl 业务节点，必须复用 blackbox + Prometheus 判定健康**。这条原则本身是对的——避免"网络问题误判为业务故障"、避免 Smoke Test 和监控告警结论不一致——但它有一个隐含前提：**入口层探测的语义 = 业务功能可用**。

这个前提在 v1.5 / v1.6 阶段成立，因为当时还没有充分意识到 ruoyi 前后端分离 + Nginx 静态托管的特殊性；到 v1.7 阶段，前端 dist 目录已稳定挂载在 Nginx 容器里（见 `Docs/narratives/v1.7.md` § 3 网关耦合那条），这个隐含前提已经被破坏，但 Smoke Test 的实现没有同步收紧。

### 3.2 blackbox 探测目标选取的早期简化

V1.5 引入告警系统（`v1.5-alerting.md` proposal）时，blackbox 目标按"4 个对外域名 + 2 个监控自身健康端点"列出，业务侧只挑了首页 URL。这在"先把告警链路从无到有跑通"的早期是合理的简化，目标是验证 Prometheus → Alertmanager → 飞书这条通道；但随着前后端分离架构成型，"首页 200 = 业务可用"这个等式已经不再成立，这条简化没有同步演进。

### 3.3 与其他两份 v1.7 review 的边界

| Review | 关注层面 | 一句话定位 |
|---|---|---|
| `v1.7-iac-completeness-audit.md` | 产物层 | IaC 是否完整描述了集群期望状态——新机器能否复现 |
| `v1.7-iac-ci-quality-gate-gap.md` | 流程层（提交前） | 已提交的代码有没有被自动 lint / scan 校验过 |
| **本审计** | **流程层（部署后）** | **已部署的代码有没有被自动验证"业务功能可用"** |

三者根因相同——**V1.4-V1.7 主线都在解决"流程从无到有"，"流程的质量保障"留白**——但盲点位于不同层面，故拆为三份独立 review。本审计是对前两份的功能性补全：从"IaC 是否完整"和"代码是否经过 lint"，延伸到"部署是否真正可用"。

### 3.4 容器级监控空白是放大器、不是根因

§2.3 提到的"容器 Restarting 不被监控捕获"是放大器：如果容器级指标到位，本次故障从容器开始 Restarting 那一刻就会触发告警，根本等不到 5 天后用户手工发现。但**这条不是本审计的根因**——根因仍然是"入口探测的语义不够深"；容器级监控只是兜底层。即使补齐 cAdvisor，如果 ruoyi 探测路径不收紧到业务接口，"应用进程在 + 功能挂"的场景（比如后端进程活但 DB 连不上）仍然抓不到。

## 4. 修复方案与排期

### 4.1 修复方案（按 ROI 分级）

**优先级 P0：ruoyi 业务级探测（消掉本次故障类型，覆盖率提升最显著的一项）**

| 改动点 | 内容 |
|---|---|
| `prometheus.yml.j2` 新增 target | `https://ruoyi.jjmstart.com/prod-api/captchaImage`（ruoyi 登录前置接口，无需鉴权，但必经过后端 + Redis；任一组件挂掉都会立刻 5xx） |
| target 标签 | `service: ruoyi-backend`，`severity: critical` |
| `Jenkinsfile` Smoke Test | 查询改为 `min(probe_success{service=~"ruoyi\|ruoyi-backend"})`，两个目标任一失败即阻断部署 |
| `blackbox.yml.j2` 是否新增模块 | 暂不新增；继续复用 `http_2xx`，验证码接口返回 200 即可 |

**优先级 P1：容器级监控（消掉同类故障的"无人感知 5 天"问题）**

- 在 `roles/monitor-stack/` 或新建 `roles/cadvisor/` 中部署 cAdvisor 到所有 app_nodes
- Prometheus 新增 cAdvisor scrape job
- 新增告警规则：`container_restart_count_total{name=~"ruoyi-admin.*"} - container_restart_count_total{name=~"ruoyi-admin.*"} offset 10m > 3` → critical
- 修复排期建议：[`proxysql-ha`](../scheme/phase-1-architecture-upgrade.md#proxysql-ha) 主题同期落地——HA 演练本身就需要观察容器重启次数，cAdvisor 是配套基础设施

**优先级 P2：其他域名的功能级探测（解锁第二类盲区）**

- jenkins / grafana / diary 各自补一个业务级探测路径（例如 grafana 的 `/api/health`、jenkins 的 `/login` 而非 `/`、diary 的某个具体笔记 API）
- 修复排期建议：[`observability-integration`](../scheme/phase-1-architecture-upgrade.md#observability-integration) 主题，与"Dashboard 整合 + 告警阈值调优"一起做

**优先级 P3：Prometheus / Alertmanager 的功能性自检（第三类盲区）**

- 新增 dead-man-switch 告警：Alertmanager 每隔 N 分钟必须发出一条 "watchdog" 告警，飞书接收侧如果 N+M 分钟未收到则报警（需借助外部健康检查或第二个独立 Alertmanager）
- 此项成本与本项目规模严重不匹配，**当前版本不修**；如果未来真出过一次"告警链路全断没人发现"的事件再回头评估

### 4.2 排期总览

| 优先级 | 项 | 计划主题 | 备注 |
|---|---|---|---|
| P0 | ruoyi 业务级探测 + Smoke Test 双源 | **v1.7 修复期**（独立小任务，1 个工作日） | 不开 proposal / runbook，完成后回填本 review § 6 |
| P1 | cAdvisor + 容器 Restarting 告警 | [`proxysql-ha`](../scheme/phase-1-architecture-upgrade.md#proxysql-ha) | 与 ProxySQL HA 演练配套 |
| P2 | jenkins / grafana / diary 业务级探测 | [`observability-integration`](../scheme/phase-1-architecture-upgrade.md#observability-integration) | 纳入 "Dashboard 整合" 同期完成 |
| P3 | Alertmanager dead-man-switch | **已升级**为 [`dms-outlet-redundancy`](../scheme/phase-1-architecture-upgrade.md#dms-outlet-redundancy) 主题 | 原判"不修"在 2026-05-26 `v1.7-alerting-pipeline-spof-audit` 中升级为独立主题 |

### 4.3 暂不做的

- **不引入 APM（SkyWalking / Pinpoint 等）**：与初级运维岗求职定位不匹配，复杂度失控
- **不为每个域名都做"业务接口探测"**：diary 等单体应用，根 URL 状态 ≈ 业务状态，强行加一层徒增维护成本
- **不立刻切换到 Spring Boot `/actuator/health`**：actuator 暴露端点涉及鉴权 / 安全策略调整，单独评估；当前阶段用业务接口路径（验证码 / 登录前置）已经足够覆盖"后端进程 + Redis"两条关键链路

## 5. 面试叙事点

### 5.1 面试一句话版（30 秒内讲完）

> "我项目里有一次 ruoyi 后端两个实例全死了 5 天没人发现——CD 全绿、Prometheus 全绿，用户登录时才报 502。倒推根因：CD Smoke Test 复用了 blackbox 入口探测，而 blackbox 探测的 `https://ruoyi.jjmstart.com/` 命中的是 Nginx 静态前端文件，根本不经过后端。整条链路的'成功标准'停留在'入口层 HTTP 200'，没触达'业务功能可用'。我顺这条线索把 6 个 blackbox 探测目标全过了一遍，盘出三类盲区——前后端分离的天然盲区、proxy_pass 链路的伪健康、`/-/healthy` 的语义局限——按 ROI 分级排进 v1.7 修复期 / `proxysql-ha` 主题 / `observability-integration` 主题三档，**P0 只动一行：给 ruoyi 加一个 `/prod-api/captchaImage` 业务级探测**，就能消掉本次故障类型 80% 的风险。这件事的教训不是'监控没用'，而是**每一道'返回 200'的探测，本质上都在验证某层壳的可达性，不是整条调用链的功能可用——分清楚是什么决定了告警是真有用还是假繁荣**。"

### 5.2 可追问 Q&A

| 追问 | 备答 |
|---|---|
| 为什么不一次性把所有 blackbox 目标都改成业务级探测？ | 按 ROI 分级。前后端分离架构是 ruoyi 独有的天然盲区，性价比最高；其他三个域名是"功能内部异常"才会漏，触发概率明显低，并且每个业务级探测路径都需要单独评估（鉴权、副作用、幂等性），不适合一次性铺开。延后做的也都记进了 phase-1 路线，不会被忘掉。 |
| 容器都死了 5 天，难道 docker daemon 不告警吗？ | docker daemon 本身没有"容器持续 Restarting 触发告警"的能力——它只负责按 `restart: unless-stopped` 策略反复拉起。要捕获这类故障需要容器级指标采集（cAdvisor 或 docker_exporter），我项目目前只有主机级 node-exporter。这是 P1 修复项，排进 `proxysql-ha` 主题和 ProxySQL HA 演练一起做。 |
| 你说 Smoke Test 复用入口探测有"隐含前提"，那当初为什么要复用？ | 复用本身是对的——避免 Jenkins 直连业务节点带来的"网络问题误判为业务故障"，也避免 Smoke Test 和监控告警结论不一致。问题在于"复用什么样的探测"——v1.6 落地时探测目标是 v1.5 留下的入口首页，那个目标的语义在 v1.5 阶段还可以等同于"业务可用"，但到了前后端分离稳定的 v1.7 已经不成立，复用没有同步收紧。正确的演进是**复用 Prometheus 这个聚合点不变，但 blackbox 的探测目标要随架构演进调整**。 |
| 既然识别出 3 类盲区，为什么 P3 直接判 "不修"？ | "不修"不等于"忽视"，是和项目规模与求职定位的匹配判断。Alertmanager 的 dead-man-switch 需要第二个独立健康检查源（或第二套 Alertmanager），对初级运维项目来说复杂度过高、面试讲不深；与其搭一套讲不清的东西，不如把"不修的理由 + 触发条件"明确写进 review，等真正出过一次"告警链路全断没人发现"的事件再回头评估。**克制处理比为了显得专业越级修复更有工程价值。** |
| 这件事和你 v1.7-iac-completeness-audit 是一回事吗？ | 不是。IaC 完整性审计关注"新机器从零跑 site.yml 能不能起来"，是**产物层**问题；CI 质量门禁审计关注"已提交代码有没有被 lint/scan"，是**流程层提交前**问题；本审计关注"已部署代码有没有被验证业务可用"，是**流程层部署后**问题。三份合起来才是 V1.7 期间识别的完整流程盲区，根因都是"V1.4-V1.7 主线优先解决流程从无到有，质量保障留白"。 |

### 5.3 可迁移的工程原则

- **每一道"返回 200"的探测都在验证"某层壳的可达性"，不是"整条调用链的功能可用"**。分清楚自己的探测在哪一层壳上，是决定告警链路有用还是假繁荣的关键。
- **复用基础设施的同时要随架构演进同步重新评估其隐含前提**。"复用 blackbox + Prometheus 做 Smoke Test"是好原则，但它隐含"入口探测语义 = 业务可用"，架构演进后这个前提失效，复用也得跟着改。
- **盲区往往不是单点失误，是放大器与根因的叠加**。本次"5 天无人感知"是"入口探测深度不够"（根因）和"容器级监控缺失"（放大器）的乘积；只修任一层都不够。
