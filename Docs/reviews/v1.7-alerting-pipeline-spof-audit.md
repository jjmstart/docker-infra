## 1. 问题识别的起点

### 1.1 起点：一个"如果 bj-01 整机挂了"的随手提问

在与 AI 讨论 "如果 bj-01 整机挂了，我还能不能收到通知" 这个问题时，顺手回头核对自己在 `Docs/scheme/phase-1-architecture-upgrade.md` § 任务 8（v1.13 服务迁移）里写过的 watchdog 设计——原文是：

> **gz-01 降级为轻量 watchdog 监控节点**：保留一个极轻量的监控实例，只盯 gz-02：node-exporter 健康、Nginx 健康端点、Prometheus 健康端点，以及公网域名 blackbox probe；告警走现有 Alertmanager（bj-01），仅数据来源从 gz-01 降为 watchdog 采集；目的：防范监控主节点（gz-02）宕机后告警链路中断，解决"谁来监控监控系统"的问题。

定睛一看，问题就清楚了：**当时设计 watchdog 时，只考虑了"监控主节点（Prometheus 所在节点）宕机"这一层冗余，完全没考虑"告警出口节点（bj-01 上的 Alertmanager + prometheus-alert）宕机或容器崩溃"这一层。** watchdog 自己也是"告警走现有 Alertmanager（bj-01）"——一旦 bj-01 整机挂或 Alertmanager 容器 OOM 重启失败，watchdog 看到一切异常，但**说不出去**。

### 1.2 这是经典 "who watches the watcher" 在自己设计里再嵌套一层

watchdog 的设计语义是 "who watches the watcher"——给监控系统加一层冗余采集端，防止"采集系统自己挂了没人告警"。但落到 v1.7 现有架构里，告警链路是：

```
被监控对象 → Prometheus（gz-01）→ Alertmanager（bj-01:9093）→ prometheus-alert（bj-01 容器内）→ 飞书 webhook（公网）→ 飞书群
```

watchdog 解决了第一段（采集 / 评估）的冗余——主监控节点宕机时仍有备份采集源；但**第二段及以后（出口节点 / 链路 / 外部送达）的所有 SPOF 没有任何备份**，watchdog 自己产生的告警照样要经过 bj-01 这条单链路才能送出去。

这是经典 "who watches the watcher" 在我自己的设计里再嵌套了一层 **"who watches the watcher's mouth"**：watchdog 解决了"采集"的冗余，但没解决"送达"的冗余。两者性质不同，混在一起就成了一种"看起来已经做了 HA、实际上只覆盖了 1/4 失效域"的伪 HA。

意识到这一点后，本审计要回答的核心问题就清楚了：**告警链路的可用性必须按独立失效域分别审视，不能把"监控系统"作为一个整体来谈高可用。**

## 2. 系统性盘点

把当前告警链路按 "采集 / 评估 / 出口 / 送达" 四个独立失效域拆开，逐域审视冗余覆盖现状：

| # | 失效域 | 现状（v1.7） | 故障表现 | watchdog（顺延后 v1.14）是否覆盖 | 严重等级 |
|---|---|---|---|---|---|
| 1 | **采集 / 评估端**（Prometheus 所在主监控节点宕机） | 单点（gz-01；服务迁移后为 gz-02） | 指标无人采集、规则不再评估、所有 firing 告警在节点恢复前看不见 | ✅ watchdog 覆盖（备份采集源 + 独立评估） | 严重 |
| 2 | **出口端**（Alertmanager 容器崩溃 / prometheus-alert 容器崩溃 / bj-01 整机宕机） | 单点（全部在 bj-01） | Prometheus / watchdog 都正常评估出 firing 告警，但 Alertmanager 收不到或收到了发不出去——**告警在出口节点黑洞** | ❌ watchdog 也发到 bj-01 同一份 Alertmanager | 严重 |
| 3 | **链路**（gz-01/gz-02 ↔ bj-01 之间 Tailscale 网络分区） | 单链路 | Prometheus / watchdog 评估正常，但发不到 Alertmanager；bj-01 自身节点级告警（如自己的 node-exporter 抓不到）也走不出去 | ❌ watchdog 发同一条链路 | 严重 |
| 4 | **外部送达**（飞书机器人风控 / 飞书侧故障 / 公网出口故障） | 单送达 | Alertmanager 已发出 webhook，但飞书侧不投递——告警被静默丢弃 | ❌ watchdog 同一个飞书机器人 | 中等 |

### 2.1 核心结论

watchdog 与 Dead Man's Switch（DMS）是**互补**关系，不是替代关系：

- **watchdog**：解决失效域 #1（采集端冗余）——消除"采集系统自己挂了没人告警"的情形
- **DMS**：通过反向心跳一次性覆盖失效域 #2 #3 #4（出口端 / 链路 / 送达冗余）——消除"评估都对了但说不出去"的情形

完整的告警可用性 = watchdog（覆盖 #1）+ DMS（覆盖 #2 #3 #4），缺一不可。

### 2.2 v1.7 当前承担的 4 个失效域全是单点

把上表抽出来横看一眼，**当前 v1.7 架构里 4 个失效域全是单点**——只有"飞书侧故障"算"中等"，其余三个都是"严重"。这不是 v1.7 设计的失误，是 V1.5 引入告警链路时的有意识范围控制（聚焦"从无到有"）；但走到 v1.7 全量复盘 + v1.13 watchdog 设计这个节点上，再把"告警出口冗余"留白就不再合理——因为下一步 v1.13 会把入口 + 主监控节点迁到 gz-02，整个集群运维的可观测能力将完全寄托在 bj-01 的 Alertmanager 单点上。

## 3. 根因分析

### 3.1 设计 watchdog 时把"监控系统"当作单一组件来想冗余

watchdog 段落的原始措辞是"防范监控主节点（gz-02）宕机后告警链路中断，解决'谁来监控监控系统'的问题"——**关键词是"监控系统"作为一个整体**。这个表述把 Prometheus（采集 / 评估）、Alertmanager（出口）、prometheus-alert（出口） 当作一个不可拆分的概念来想，自然只想到"如果这个整体所在的节点挂了怎么办"。

但实际部署上 v1.7 的告警链路在物理拓扑上是**两节点拆开**的：Prometheus 在 gz-01（v1.13 后是 gz-02），Alertmanager + prometheus-alert 在 bj-01。**节点物理上已经拆开，failure domain 也应当跟着拆开看冗余**——可惜设计 watchdog 时这条没跟上，直觉上"给监控加冗余"=" 给主监控节点加备份采集"，于是只解决了一端，把另一端漏在视野之外。

### 3.2 高可用设计的通用陷阱：按组件名整体处理 vs 按 failure domain 分别审视

把这一条抽象出来，是高可用设计的一个通用陷阱：**HA 必须按 failure domain 分别审视，而不是按组件名整体处理**。

举几个同族陷阱（脱离本项目语境也成立）：

- "数据库 HA" 经常只想到主从复制（数据冗余），漏掉连接代理 / 中间件这一层（参见叙事点 #9 ProxySQL SPOF 也是同族识别——`Docs/narratives/v1.7.md` § 9）
- "应用 HA" 经常只想到多副本部署，漏掉负载均衡器自身的冗余
- "监控 HA" 经常只想到监控节点冗余，漏掉告警出口的冗余（**本审计**）

共性都是**把一个由多组件组成的系统当作单一节点来想 HA**——只在容易直觉到的那一段做冗余，其他段照样单点。识别这类盲点的方式不是回头复查"我冗余做得够不够"，而是**按 failure domain 重新枚举一遍**——对告警链路就是 "采集 / 评估 / 出口 / 送达"，对数据库链路就是 "应用 / 连接池 / 代理 / 引擎 / 复制"，等等。

| Review | 关注层面 | 一句话定位 |

## 4. 修复方案与排期

### 4.1 4 种候选方案对比

| 方案 | 核心机制 | 覆盖失效域 | 决策评价 |
|---|---|---|---|
| **A. Dead Man's Switch（DMS）** ★ 推荐 | Prometheus 配 `expr: vector(1)` 永真心跳规则；Alertmanager 路由到外部 SaaS（如 Healthchecks.io / Better Uptime / Dead Man's Snitch 等）；SaaS 按固定间隔期望收到心跳，超阈值反向告警到独立邮箱 | #2 #3 #4 一次覆盖 | **采用**：不扩节点；外部 SaaS 与自有节点物理隔离（消除 bj-01 单点）；落地工作量极小（一条 Prometheus rule + 一条 Alertmanager route + 一个 SaaS 账号 + 一项 vault webhook）；初级岗叙事友好，"为什么用 DMS" 比 "为什么不用 Alertmanager 集群" 更容易讲清 |
| B. Alertmanager 集群（双实例 HA） | gz-01 + bj-01 各部署一份 Alertmanager，Gossip 集群去重；Prometheus 同时发给两个实例 | 仅 #2（且仅"出口节点"层，不覆盖 prometheus-alert / 飞书侧） | **不采用**：① gz-01 当前 2C2G 资源紧张（架构快照 v1.7 § 节点总览），多扩一个 role 风险面增大；② 增加 Ansible role + Gossip 协议运维复杂度；③ **只解决 4 个失效域中的 1 个**——出口节点冗余有了，但链路（#3）和飞书侧（#4）依然单点；④ ROI 与初级岗叙事不匹配 |
| C. 第三方独立黑盒探测（UptimeRobot / Pingdom 等） | SaaS 从公网探测业务入口与监控入口 URL，独立告警 | 仅业务入口可达性，看不到内部状态 | **不采用为主方案**：黑盒探测看不到 MySQL 复制延迟、Redis 复制断开、ProxySQL 后端连通性等内部状态；适合作为补强（参见 `v1.7-ingress-probe-functional-depth-gap.md` 的入口探测改造），不适合替代 DMS |
| D. Prometheus + Alertmanager 全链路 HA | 双 Prometheus + 双 Alertmanager，Gossip + 自适应去重 + 应用同时上报多端 | #1 #2 全覆盖 | **不采用**：大厂方案；对当前 4 节点 + 初级岗叙事属于明显过度配置，违反 `.cursor/rules/00-project-context.mdc` 决策红线 "不堆砌当前用不到的组件" + "不引入无中小厂实际使用场景的技术" |

### 4.2 最终决策

1. **DMS 独立成新版本插入到 v1.8**——原 v1.8 起所有版本号顺延一格（v1.8 → v1.9，v1.9 → v1.10，... v1.16 → v1.17；v2.0 不动）
2. **watchdog 保留在顺延后的 v1.14**——但在原文 "目的：防范监控主节点（gz-02）宕机后告警链路中断" 一行后追加 **"覆盖边界声明"**，显式说明 watchdog 仅覆盖采集 / 评估端单点，不覆盖出口 / 链路 / 送达三层
3. **不引入 Alertmanager 集群（方案 B）也不做全链路 HA（方案 D）**——理由如上表

DMS 与 watchdog 共同构成**完整告警可用性**：watchdog 覆盖 #1，DMS 覆盖 #2 #3 #4。两者互补，缺一不可。

## 5. 面试叙事点

本审计催生两类叙事，配套写入 `Docs/narratives/v1.7.md`（详见 narratives 文件，本节只抽核心要点）：

### 5.1 叙事 1：如何识别盲点（从单点问题倒推到自己设计中的盲点）

> "我在讨论 'bj-01 整机挂了我还能不能收到告警' 这个问题时，顺手回头看自己 v1.13 watchdog 的设计，发现 watchdog 自己也是把告警发给 bj-01 上同一个 Alertmanager——我之前只防住了'谁来监控监控系统'，没防住'谁来送告警'。这是经典 'who watches the watcher' 在自己的设计里再嵌套了一层 'who watches the watcher's mouth'——告警系统按 failure domain 是 4 段（采集 / 评估 / 出口 / 送达），watchdog 只覆盖了第 1 段，2-4 段全单点。"

可迁移要点：**识别这类盲点的入口不是抽象 review，而是从一个具体单点问题倒推回自己已经做完的设计**——前者会陷入"我冗余够不够"的循环自证，后者能跳出来。

### 5.2 叙事 2：HA 设计要按独立失效域分别审视，不能按组件名整体处理

> "高可用设计的通用陷阱是把一个由多组件组成的系统当作单一节点来想 HA——只在容易直觉到的那一段做冗余，其他段照样单点。监控 HA 经常只想到监控节点冗余漏掉告警出口冗余、数据库 HA 经常只想到主从复制漏掉连接代理 / 中间件、应用 HA 经常只想到多副本漏掉负载均衡器自身——共性都是按组件名整体处理而非按 failure domain 分别审视。识别这类盲点的方式是按 failure domain 重新枚举一遍，对告警链路就是采集 / 评估 / 出口 / 送达，每段单独问'有没有冗余'。"

### 5.3 为什么 DMS 优先于 Alertmanager 集群

> "DMS 一条 rule + 一条 route 一次覆盖 4 个失效域里的 3 个（出口 / 链路 / 送达），不扩节点；Alertmanager Gossip 集群只覆盖 1 个失效域（出口节点），还要多扩一个 role 增加运维复杂度，且不解决链路和飞书侧。同样工作量做 DMS 比做集群覆盖面更广 + 落地更轻 + 与自有节点物理隔离——这是个明显的 ROI 选择。**HA 化的'量级'不是越大越好，是该覆盖的失效域都覆盖到、不该过度的别过度。**"
