# V1.7 CI 链路缺少 IaC 自身质量门禁

> **触发时间**：2026-05-19
> **触发版本**：V1.7
> **触发点**：每次请 AI 帮做代码 review 时反复看到它自报"现在我来检查敏感内容" / "现在我来进行敏感关键词检查"，追踪后定位到 `.cursor/rules/60-git-and-commits.mdc` 里那段提交前手工 `grep` 自检条款，与 AI 进一步交互后意识到这是一条"规则文件里写好的君子约定"，而不是"做不到就被拦下来"的强制门禁；再把视角拉回 V1.7 Jenkins Jobs 总览对照三条 Pipeline 覆盖范围，发现 IaC 代码侧整段 lint/scan 链路是空白的
> **状态**：已识别 → 待修复
> **关联文档**：
> - `Docs/architecture/v1.7.md` § "Jenkins Jobs 总览"：当前三条 Pipeline 的覆盖范围
> - `Docs/reviews/v1.7-iac-completeness-audit.md`：同期 IaC 完整性审计（产物层盲点）；本审计聚焦"流程层"盲点，互补
> - `Jenkinsfile`、`Jenkinsfile.registry-gc`：现有 Pipeline 定义
> - `.cursor/rules/60-git-and-commits.mdc`：当前仅靠手工 `grep` 的提交前敏感信息自检条款

---

## 1. 问题识别的起点

### 1.1 起点：AI review 反复念叨的那句"敏感关键词检查"

V1.7 期间频繁请 AI 协助做代码 review 与提交，多次看到 AI 在动手前自报："现在我来检查敏感内容" / "现在我来进行敏感关键词检查"。一开始没在意——出现频率高了，开始好奇这条行为是被哪条规则触发的。

追踪发现源头是 `.cursor/rules/60-git-and-commits.mdc` 里这一段：

```bash
git diff --cached | grep -iE "password|token|secret|webhook|api[_-]?key" | grep -v "vault\|credentials\|placeholder\|example"
```

继续与 AI 交互去理解这条规则的执行边界，关键问题是："这条 `grep` 实际由谁来执行？"答案是——**没有自动化执行方**：它写在规则文件里，AI 在 review 时会"自觉"按这条提示走一遍，但提交动作真正发生时，并没有任何 hook / CI stage 强制运行这条命令。换句话说，这条防线的全部强度建立在"提交者（人或 AI）记得执行"上。

意识到这一点后问题就清楚了：**规则文件描述的是"应该这么做"，CI 门禁才是"做不到就被拦下来"——两者性质不同，混在一起就成了一道伪门禁**。看起来"已经有规范"，实际是"没有强制执行的规范"，这种状态比"承认没有"更危险，因为提交前一旦疏忽，事后没有任何兜底。

### 1.2 顺着这条线索拉回 CI 链路对照

把视角从规则文件拉回 V1.7 Jenkins Jobs 总览，把三条 Pipeline 的覆盖范围列在一起对照：

| Job | 触发 | 覆盖范围 |
|---|---|---|
| `ruoyi-jenkins-docker-auto-deploy` | 应用仓库 push | 应用 Docker 镜像构建 + push Registry + 飞书"新版本就绪"通知 |
| `ansible-deploy` | docker-infra 仓库 push / 手动 | Ansible Check（`--check` dry-run）+ Deploy + Smoke Test + 飞书通知 |
| `registry-gc` | cron 每周日 02:00 | 保留最新 10 个 tag，旧 tag 软删除 + Registry GC |

对比"应用代码侧"和"IaC 代码侧"的覆盖差异：

- 应用代码：build → push → deploy → smoke 完整链路
- IaC 代码：**只有** deploy + smoke，**没有 build 之前的 lint 与 scan 环节**

`ansible-deploy` 里那条 "Ansible Check" stage 不能算质量门禁——`--check` 是"在目标节点上模拟执行 playbook 看会改什么"，验证的是**部署结果**，不是**代码本身的质量**。这是两件事，长期混在一起就掩盖了缺口。

至此整个盲点闭环清晰：规则文件那条手工 `grep` 是空头支票，CI 链路里又没有真实的 lint/scan stage 兜底——IaC 代码自从 V1.4 引入到现在，从来没有在合入主干前被自动校验过。

## 2. 系统性盘点

把"应当被 CI 自动检查、但目前只能靠人肉记得做"的事列出来：

| 编号 | 缺失检查项 | 当前依赖方式 | 严重等级 |
|------|------------|--------------|----------|
| #1 | Ansible / YAML 语法 lint | 无 | 中等：低级错误要等部署时才暴露，浪费 Pipeline 时间 |
| #2 | Ansible role 最佳实践检查（`ansible-lint`） | 无 | 中等：role 越多积累得越快 |
| #3 | Secret 提交扫描（密码 / token / 飞书 webhook URL / 阿里云 AccessKey 等明文凭据） | 规则文件里的手工 `grep` | **严重**：依赖人执行，一旦遗漏即泄漏到 Git 历史，不可逆 |
| #4 | vault 文件加密状态校验（首行应为 `$ANSIBLE_VAULT;1.1;AES256`） | 人肉 `head -1` | **严重**：误提交明文 vault 不可逆 |
| #5 | Jenkinsfile 自身的 declarative lint | 无 | 轻量：Jenkinsfile 改动频率低 |

#3 与 #4 不是"代码质量问题"，是"信息泄漏问题"——一旦发生不可恢复，必须自动化拦截，不能留在"君子约定"层面。

---

## 3. 根因分析

### 3.1 IaC 引入时优先解决"配置下发"，质量门禁留白

V1.4 引入 Ansible 时，重点放在"把手工配置下发自动化"，没有同步把"IaC 代码本身的质量门禁"建立起来。这与 `v1.4-retrospective.md` 的范围控制一致——优先解决"流程从无到有"的问题，"流程的质量保障"留给后续版本。

但 v1.5（告警）、v1.6（CI/CD 应用侧）、v1.7（备份恢复）三个版本主线都是新功能，CI 质量门禁这一项就一直没被排进路线，直到这次复盘对照才暴露。

### 3.2 规则文件的"伪门禁"现象

`60-git-and-commits.mdc` 里那段手工 `grep` 让维护者潜意识里认为"已经有规范在管"，反而降低了把它纳入 CI 强制门禁的紧迫感。**规则文件本身不是门禁**，规则文件描述的是"应该怎么做"；门禁是"做不到就被拦下来"。两者性质不同，混淆后就成了 #3 盲点的成因之一。

### 3.3 与同期 IaC 完整性审计的边界

`v1.7-iac-completeness-audit.md` 关注的是"IaC 是否完整描述了集群期望状态"（**产物层**问题——新机器能否复现）；本审计关注的是"IaC 代码本身的提交质量"（**流程层**问题——已提交的代码是否经过自动校验）。根因相同（V1.4 引入 IaC 时只解决了核心命题），但盲点性质不同，故拆为两份独立 review。

## 4. 修复方案与排期

### 4.1 修复方案

在 Jenkins 中新增一条 `iac-prelint` Job（或在现有 `ansible-deploy` Job 起始处新增 stage），覆盖：

| 工具 | 覆盖盲点 | 备注 |
|------|----------|------|
| `yamllint` | #1 | 单文件级 YAML 语法 |
| `ansible-lint` | #1 #2 | role 最佳实践 |
| `gitleaks` | #3 | Go 单二进制无依赖；自定义规则覆盖飞书 webhook (`open.feishu.cn/open-apis/bot/v2/hook/`) 与阿里云 AccessKey (`LTAI`) |
| `head -1 vault/secrets.yml` 校验 | #4 | 单行实现：`grep -q '^\$ANSIBLE_VAULT' vault/secrets.yml \|\| exit 1` |
| `jenkins-cli declarative-linter`（可选） | #5 | 与 lint stage 合并 |

**触发约束**：

- 触发时机：docker-infra 仓库 push（与现有 `ansible-deploy` 同源）
- 失败语义：任一项失败 → Pipeline `FAILURE` → 飞书 `jenkins-notify` 通知作者 → **不进入** `ansible-deploy` 后续 stage
- 白名单：vault 文件本身、`*.example`、明确标注 placeholder 的位置走白名单；白名单本身也入 Git，由 PR review 把关

### 4.2 排期

| 项 | 时间 | 备注 |
|---|------|------|
| 实施 | 1-2 个工作日 | 独立小任务，不必等任何主线版本（v1.8 起为 DMS / 基础可复现性等主线） |
| 文档归属 | **不开 proposal / runbook**（变更范围小）；完成后在本 review §6 后续追踪段直接回填实施细节 | 与 `.cursor/rules/10-docs-workflow.mdc` 小变更不强求 9 步流程一致 |
| 是否打 tag | **不打** `arch-v1.7.1`；视作 v1.7 工作期内的 CI 增强 | tag 留给架构语义变化使用 |

### 4.3 暂不做的

- **不引入 SonarQube / 重型代码质量平台**：当前 IaC 仓库规模不够，ROI 不足
- **不在 IaC 仓库引入 pre-commit 框架**：单 Jenkins stage 已覆盖；引入 pre-commit 会让本地工作流额外多一层依赖
- **不引入 Trivy 等镜像扫描**：本审计聚焦 IaC 代码侧；镜像扫描属于"应用 + K3s admission controller"主题，待 v1.15 K3s 落地后再起一份独立 review

## 5. 面试叙事点

> "我们 docker-infra 仓库的 CI 一开始覆盖的是'应用镜像构建 → 部署 → Smoke Test'这条链路，但 IaC 代码自身的提交质量是空白——靠规则文件里写的一段手工 `grep` 命令防泄漏，本质上等于没防。我做 V1.7 复盘时把三条 Pipeline 的覆盖范围列在一起对照，才发现这个缺口，在 Jenkins 上加了一条 `iac-prelint` Job：`yamllint` + `ansible-lint` + `gitleaks` + vault 加密状态校验，任一失败就阻断 Pipeline、飞书通知作者。这件事的工程意义不在工具多复杂，而在'IaC 也是代码、代码就要有质量门禁'——这一条很多中小厂的运维实践里其实是缺的。"

可追问场景与备答：

| 追问 | 备答 |
|---|---|
| 为什么是 gitleaks 不是 detect-secrets | gitleaks 是 Go 单二进制无依赖；detect-secrets 是 Python 需要更多环境配置——CI 内运行 gitleaks 更轻量 |
| 为什么不用 pre-commit hook 在本地拦截 | 本地 hook 可被 `--no-verify` 跳过（规则虽然明令禁用此参数，但属于"君子约定"）；CI 阶段是不可绕过的强制门禁 |
| 白名单怎么管 | vault 文件本身、`*.example`、明确标注 placeholder 走白名单；白名单也入 Git，由 PR review 把关；新增白名单条目需要 commit 信息说清楚理由 |
| 为什么不直接用 Ansible 的 `ansible-playbook --syntax-check` | `--syntax-check` 只查 playbook 语法，不查 role 最佳实践（如未声明变量、handler 误用、`when` 表达式风险等）；`ansible-lint` 才是覆盖最佳实践的工具 |
