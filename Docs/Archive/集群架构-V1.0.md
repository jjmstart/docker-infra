## 架构拓扑

```
互联网用户
    │
    ▼
gz-01（入口网关）
├── Nginx（反向代理 + SSL 卸载 + 静态主站）
│     ├── jjmstart.com         → 本地静态文件 (0ms)
│     ├── ruoyi.jjmstart.com   → gz-03 + gz-02 负载均衡 (同城 ~5ms)
│     ├── diary.jjmstart.com   → bj-01 (跨城 ~35ms)
│     ├── jenkins.jjmstart.com → bj-01 (跨城 ~35ms)
│     └── monitor.jjmstart.com → bj-01 (跨城 ~35ms)
│
├────────── 同城 ~5ms ──────────┐
│                                │
gz-03（业务主节点）               gz-02（业务副节点）
├── ruoyi-admin-1                ├── ruoyi-admin-2
├── MySQL 主库                   ├── Redis Slave1
├── Redis Master                 └── Sentinel2
└── Sentinel1
                    ↕ Tailscale ~35ms
             bj-01（运维 + 灾备）
             ├── Jenkins CI/CD
             ├── Prometheus + Grafana
             ├── Diary (Memos)
             ├── Redis Slave2（异地灾备）
             └── Sentinel3
```

## Redis 哨兵集群

```
gz-03 Master ──复制──→ gz-02 Slave1（同城，priority=100）
     │
     └─────复制──→ bj-01 Slave2（异地，priority=200）

Sentinel 集群：3 节点（gz-03 / gz-02 / bj-01），quorum=2
故障切换顺序：gz-02(同城) > bj-01(异地)
```

## 网络互联

所有节点通过 Tailscale WireGuard 加密隧道互联，不依赖公网端口暴露。

| 链路 | 延迟 | 用途 |
|---|---|---|
| gz-01 ↔ gz-03 | ~5ms | Nginx → ruoyi-admin-1 |
| gz-01 ↔ gz-02 | ~5ms | Nginx → ruoyi-admin-2 |
| gz-01 ↔ bj-01 | ~35ms | Nginx → Jenkins / Grafana / Diary |
| gz-03 ↔ gz-02 | ~5ms | MySQL 跨节点访问、Redis 同城复制 |
| gz-03 ↔ bj-01 | ~35ms | Redis 异地复制 |

## 演进方向

### 第一阶段：备案通过后立即执行（优先级：高）

**1. 端口标准化**

将 Nginx 端口从 `8443:443` 改为 `443:443`，去掉所有 conf.d 中的 `:8443` 跳转。用户访问不再需要手动输入端口号，同时 `error_page 497` 也可以移除。

**2. 开启 HTTPS 全站强制 + HSTS**

在 Nginx 中添加 HSTS 响应头，告诉浏览器后续一律走 HTTPS，防止中间人降级攻击：

```nginx
add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
```

**3. 接入 Cloudflare CDN**

备案后域名可以正常走 443，适合接入 Cloudflare（免费版）：
- 前端 Vue 静态资源全球 CDN 加速，首屏加载提速 50%+
- API 请求回源到 gz-01 Nginx，动态流量不受影响
- 自动 SSL 证书管理，不再需要手动续签
- 免费 DDoS 防护 + WAF 基础规则

### 第二阶段：提升可靠性（优先级：中）

**4. Nginx 双节点高可用**

当前 gz-01 是单点入口，它宕机则全站不可用。在 gz-02 上部署第二个 Nginx，通过 DNS 轮询或 Cloudflare 负载均衡实现入口层高可用：

```
DNS 轮询
  ├── gz-01 Nginx（主）
  └── gz-02 Nginx（备）
```

**5. MySQL 主从复制**

当前 MySQL 是单实例，无灾备。建议在 gz-02 上搭建 MySQL 从库，实现同城主从：

```
gz-03 MySQL Master ──binlog 复制──→ gz-02 MySQL Slave（只读）
```

- 主库宕机时可手动提升从库为主库（RPO 接近 0）
- 从库可分担读查询（若依的报表、列表查询走从库）
- 后期可升级为 MySQL Group Replication 或 ProxySQL 读写分离

**6. 监控告警完善**

当前 Prometheus + Grafana 只做了数据采集和展示，缺少告警：
- 配置 Alertmanager，对接企业微信/钉钉/邮件通知
- 关键告警规则：节点宕机、容器 OOM、Redis 主从断连、MySQL 复制延迟、磁盘使用率 > 80%、SSL 证书即将过期
- 添加若依应用层监控（JVM 内存、GC 频率、接口响应时间）

### 第三阶段：自动化与效率（优先级：中低）

**7. CI/CD 流水线改造**

当前 Jenkins 构建的若依镜像只在 bj-01 本地，部署到 gz-03/gz-02 需要手动 `docker save + scp`。改进方案：
- 搭建私有 Docker Registry（Harbor），Jenkins 构建后推送到 Registry
- gz-03/gz-02 从 Registry 拉取镜像，实现一键部署
- 或采用轻量方案：Jenkins 构建后通过 SSH 直接在目标节点执行 `docker compose pull && docker compose up -d`

**8. 基础设施即代码（IaC）**

当前各节点配置散落在各自的 `/opt/docker` 目录，缺乏版本管理：
- 将所有 docker-compose.yml、Nginx 配置、Prometheus 配置纳入 Git 仓库
- 使用 Ansible 实现配置分发自动化（一条命令同步到所有节点）
- 敏感信息（密码、密钥）使用 Ansible Vault 或 SOPS 加密存储

**9. 日志集中管理**

各节点日志分散，排查问题需要逐台 SSH 查看。建议部署轻量日志栈：
- Loki + Promtail（比 ELK 轻量 10 倍，配合现有 Grafana 直接使用）
- Promtail 采集各容器日志 → 发送到 Loki → 在 Grafana 中统一检索
- bj-01 有 16G 内存，适合部署 Loki 存储

### 第四阶段：长期规划（优先级：低）

**10. Kubernetes 迁移评估**

当前 4 台服务器的 Docker Compose 方案足够应对现有规模。当满足以下条件时可考虑迁移到 K8s：
- 服务数量超过 15 个
- 需要自动扩缩容
- 需要滚动更新零停机部署
- 团队规模扩大，需要 RBAC 权限管理

轻量方案：K3s（单节点 512MB 内存即可运行），可以先在 bj-01 上试点。

**11. 数据库云托管评估**

当自建 MySQL 运维成本（备份、监控、版本升级、性能调优）显著增长时，可评估迁移到云厂商的 RDS 服务。优点是免运维、自动备份、读写分离开箱即用；缺点是成本较高。
