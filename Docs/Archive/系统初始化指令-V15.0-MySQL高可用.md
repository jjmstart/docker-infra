# 系统初始化指令 V15.0 - MySQL 高可用专项

> 配套架构文档：`集群架构-V1.2.md`。本文档描述从 **MySQL 单点主库（gz-03）** 向 **一主两从 + 半同步复制 + ProxySQL 读写分离 + bj-01 延迟复制** 的演进目标、任务清单与决策，供实施与面试复盘使用。

---

## 1. 身份与目标

- **角色**：资深 SRE Tech Lead（与 V14.2 Redis/节点迁移专项一致）。
- **简历叙事**：异构云多活环境下的 **MySQL HA 实战**（主从复制、读写分离、监控与演练）。
- **当前 MySQL 基线**：主库部署于 gz-03，Compose 见 `/opt/docker/backend/mysql/docker-compose.yml`（`mysql:8.0`，Tailscale `100.92.5.116:3306` 供 gz-02 访问）。**Phase 1 需显式开启 binlog / GTID**（当前 `command` 中尚未包含）。

---

## 2. 当前架构（Before）

```
ruoyi-admin-1 (gz-03) ──┐
                         ├──► MySQL 主库 (gz-03)   ← 单点
ruoyi-admin-2 (gz-02) ──┘     Tailscale 100.92.5.116:3306
```

**主要风险**

- 单点故障：gz-03 MySQL 不可用则全站数据库不可用。
- 读写均打主库，无读分流。
- 无从库则无可靠 Binlog 消费与 PITR 基础。
- 节点变更时缺少「先扩从再切」的退路。

---

## 3. 目标架构（After，V1.2）

```
ruoyi-admin-1 (gz-03) ──┐
                          ├──► ProxySQL (gz-03) :6033
ruoyi-admin-2 (gz-02) ──┘    写 → Master（gz-03）
                              读 → Slave1（gz-02，在线读）
                               │            │
                      ┌────────┘            └────────┐
                      ▼ 写                           ▼ 读
          ┌───────────────────────┐     ┌──────────────────────┐
          │ gz-03 · MySQL Master  │     │ gz-02 · MySQL Slave1 │
          │ 100.92.5.116:3306     │     │ 100.79.132.125:3306  │
          │ server-id=1 · GTID   │     │ server-id=2          │
          │ 半同步主端            │     │ 半同步从端·实时复制   │
          └──────────┬────────────┘     └──────────────────────┘
                     │ 半同步复制
          ┌──────────┴──────────────┐
          ▼ 同城·ACK ~5ms           ▼ 跨城·ACK ~35ms
    gz-02 Slave1（在线读）    bj-01 Slave2（灾备·延迟 3600s）
                               SOURCE_DELAY=3600s
                               不参与在线读流量
```

**读流量分离（已确定）**

| 选型 | 说明 |
|------|------|
| **ProxySQL（gz-03 集中部署）** | 应用连接 :6033，写路由 hostgroup 0（Master），读路由 hostgroup 1（Slave1）；bj-01 Slave2 因延迟复制不纳入读 hostgroup |

## 4. 节点与角色

| 节点 | 规格 | MySQL 角色 | 备注 |
|------|------|------------|------|
| gz-03 | 4C8G | Master + **ProxySQL** | 主写；`server-id=1`；GTID；半同步主端；ProxySQL `:6033` 集中部署 |
| gz-02 | 4C4G | Slave1 | 同城从；`server-id=2`；半同步从端；实时复制；在线读流量；故障接管优先 |
| bj-01 | 4C16G | Slave2（延迟） | 异地从；`server-id=3`；半同步从端；**`SOURCE_DELAY=3600s`**；灾备防误删；不参与在线读 |

## 5. 迁移任务清单（Backlog）

### Phase 1：主库准备（gz-03，低停机窗口）

- [ ] `command` 块新增：`log-bin=mysql-bin`、`server-id=1`、`gtid_mode=ON`、`enforce_gtid_consistency=ON`、`binlog_format=ROW`、`binlog_expire_logs_seconds=604800`。
- [ ] `command` 块新增半同步：`plugin-load-add=rpl_semi_sync_source=semisync_source.so`、`rpl_semi_sync_source_enabled=ON`、`rpl_semi_sync_source_wait_for_replica_count=1`、`rpl_semi_sync_source_timeout=5000`。
- [ ] 确认数据目录与卷：`./data:/var/lib/mysql`（见现有 Compose）。
- [ ] 创建复制账号 `repl_user`（`REPLICATION SLAVE` 最小权限）。
- [ ] 验证：`SHOW MASTER STATUS`、`SHOW VARIABLES LIKE 'rpl_semi_sync%'`、`SHOW VARIABLES LIKE 'gtid_mode'`。

### Phase 2：同城从库（gz-02）

- [ ] 部署 MySQL（`server-id=2`，`read_only=ON`，半同步从端：`plugin-load-add=rpl_semi_sync_replica=semisync_replica.so`、`rpl_semi_sync_replica_enabled=ON`）。
- [ ] 快照：`mysqldump --single-transaction --master-data=2 --set-gtid-purged=ON`。
- [ ] 经 **Tailscale**（100.79.132.125）传快照，避免公网明文。
- [ ] `CHANGE REPLICATION SOURCE TO`（GTID 模式）→ `START REPLICA`，验证 `Seconds_Behind_Source=0`。
- [ ] 回到 gz-03 验证：`SHOW STATUS LIKE 'rpl_semi_sync_source_status'` 应由 OFF 变 ON。

### Phase 3：异地从库（bj-01）

- [ ] 部署 MySQL（`server-id=3`，`read_only=ON`，同 Phase 2 半同步从端配置）。
- [ ] 同 Phase 2 流程传快照、导入、建立复制。
- [ ] **配置延迟复制**：`STOP REPLICA SQL_THREAD; CHANGE REPLICATION SOURCE TO SOURCE_DELAY=3600; START REPLICA SQL_THREAD;`
- [ ] 验证：`SHOW REPLICA STATUS\G` 确认 `SQL_Delay=3600`；记录跨城复制 IO_Thread 延迟基线。

### Phase 4：ProxySQL 读写分离（gz-03）

- [ ] 创建 `/opt/docker/backend/proxysql/docker-compose.yml`，部署 ProxySQL，绑定 `100.92.5.116:6033`（MySQL 协议）、`:6032`（Admin）。
- [ ] 编写 `proxysql.cnf`：hostgroup 0（写）→ gz-03 Master，hostgroup 1（读）→ gz-02 Slave1；查询规则 SELECT → hostgroup 1；bj-01 不纳入读 hostgroup。
- [ ] 修改 ruoyi-admin-1 / ruoyi-admin-2 数据源 URL → `100.92.5.116:6033`，重启服务。
- [ ] 验证 ProxySQL Admin `stats_mysql_query_digest`，确认 SELECT 命中 Slave1，写操作命中 Master。

### Phase 5：监控（gz-01 Prometheus）

- [ ] 各节点 Compose 新增 `mysqld_exporter`（`:9104`，绑定 Tailscale IP）。
- [ ] `prometheus.yml` 增加三个 mysqld_exporter targets（gz-03 / gz-02 / bj-01）。
- [ ] Grafana 导入 MySQL Overview 面板（如 Dashboard ID 7362）。
- [ ] 配置告警规则：复制延迟 > 10s、主从断连、连接数超阈值、半同步从库数为 0。

### Phase 6：故障演练

- [ ] 模拟 gz-03 MySQL 不可用 → gz-02 Slave1 提升为新主 → 更新 ProxySQL 写 hostgroup → bj-01 重新指向新主。
- [ ] 记录 **RTO**（目标 < 5 分钟）与 **RPO**（半同步下目标 ≈ 0；降级 async 场景 ≤ 5s）。
- [ ] 验证 bj-01 延迟复制的误删恢复流程（`STOP REPLICA SQL_THREAD` → 等待 relay log 回放至误删前时间点）。
- [ ] 恢复原拓扑，文档化切换 SOP。

## 6. 关键技术决策（已确定）

| 决策点 | 结论 | 说明 |
|--------|------|------|
| **复制模式** | **半同步（Semi-sync）** | `rpl_semi_sync_source_wait_for_replica_count=1`，至少 1 从库 ACK 后主库才返回；超时 5s 自动降级 async 保可用；同城 gz-02 ~5ms ACK，写延迟影响极小 |
| **GTID** | **开启** | `gtid_mode=ON` + `enforce_gtid_consistency=ON`；简化从库接入、故障恢复与 `mysqldump --set-gtid-purged=ON` |
| **binlog 格式** | **ROW** | MySQL 8.0 默认；复制一致性最佳 |
| **快照工具** | **mysqldump** | 当前数据量小；`--single-transaction`（InnoDB 不锁表）+ `--master-data=2` + `--set-gtid-purged=ON` |
| **bj-01 延迟复制** | **开启，SOURCE_DELAY=3600s** | binlog 实时接收（IO_Thread 实时），SQL_Thread 延迟 1 小时应用；防止误删/误改在 1 小时内向异地扩散 |
| **读写分离** | **ProxySQL（gz-03 集中部署）** | `:6033` 对应用透明；写 → hostgroup 0（Master）；读 → hostgroup 1（Slave1 gz-02）；bj-01 因延迟复制不参与在线读 |
| **自动切换** | **手动切换 + 告警** | 不引入 MHA/Orchestrator；依赖 Prometheus 告警触发人工操作；切换 SOP 在 Phase 6 演练后文档化 |

## 7. 交互与排障约定

- **数据一致性**：每 Phase 明确 GTID / binlog position / 从库状态检查命令。
- **排障**：`SHOW REPLICA STATUS`、错误号、`tcpdump`/`ss` 查 3306、`journalctl -u docker` 等。
- **回滚**：每步保留「主库可独立服务」的回退路径（停从、不切写等）。

## 10. 面试复盘提纲（迁移完成后填写）

- **方案设计**
  - 为何 1 主 2 从：业务节点数量匹配、同城从提供在线读分担、异地从提供灾备
  - 为何选半同步：RPO 接近 0（同城 ACK ~5ms），写延迟影响极小；超时降级机制保证可用性
  - 为何 bj-01 用延迟复制：防止 DBA 误删/误改在异地扩散，提供 1 小时人工干预窗口
  - 为何选 ProxySQL 而非应用层：与应用解耦，主从切换无需改代码，路由规则可热更新
  - 为何暂不用 Orchestrator：4 节点规模，手动切换 SOP + 监控告警足够；引入 Orchestrator 增加运维复杂度

- **风险控制**
  - 复制延迟：`mysqld_exporter` + `rpl_semi_sync_source_no_tx_count` 指标告警
  - 网络分区：半同步超时降级 async，保证写可用；分区恢复后自动恢复半同步
  - 误操作防线：bj-01 延迟复制 1 小时 + Prometheus 告警 + 定期 mysqldump 备份
  - ProxySQL 单点：V1.2 阶段接受该风险；V1.3 考虑 ProxySQL Cluster 或双节点热备

- **可量化结果**
  - RTO 目标：< 5 分钟（人工切换）
  - RPO 目标：半同步正常 ≈ 0；降级 async 后 ≤ 5s（超时窗口）
  - 监控覆盖：复制延迟 / 主从断连 / 连接数 / 半同步从库数 / ProxySQL 路由统计
  - 告警规则数：≥ 5 条（延迟 > 10s / 断连 / 连接数超阈 / 半同步从 0 / Slave2 延迟异常）
