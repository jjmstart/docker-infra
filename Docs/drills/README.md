# drills/ — 故障演练记录

本目录沉淀**主动验证已有能力**的故障演练。每次演练对应一份独立文档，记录"演练目标 → 前置基线 → 演练步骤 → 实测观察 → 心得 → 待改进项"完整闭环。

## 子体系定位

故障演练与 reviews 并列，是横向、跨版本、不绑定 Git tag 的**运维成熟度建设**：

- **reviews**：从代码与配置中识别"理论盲点"（Why & Gap）
- **drills**：用线上实操验证"实际表现"（Verification）
- **architecture / runbooks / retrospectives**：仍按版本演进节奏管理

演练验证三类问题：

1. **告警链路时延**：故障发生 → Prometheus 评估 → Alertmanager 分组 → 飞书到达，全程实测耗时
2. **故障摘除自动性**：Nginx upstream、ProxySQL hostgroup、Sentinel 选主等自动处理是否如设计
3. **恢复操作可重复性**：`Docs/runbooks/` 里写的恢复步骤在故障真实发生时是否可执行

## 与其他文档的边界

| 文件 | 回答的问题 |
|---|---|
| `Docs/runbooks/vX.Y-to-vX.Z.md` | 怎么把架构从 A 升级到 B（**Transition**） |
| `Docs/retrospectives/vX.Y-retrospective.md` | 单次架构升级中踩了什么坑（**Why**） |
| `Docs/reviews/v{触发版本}-{主题}.md` | 跨版本主题型盲点（**Why & Gap**） |
| **本目录** | 已有能力在受控故障下的实际表现（**Verification**） |
| `Docs/sli-slo.md` | 对外承诺什么、违约怎么办（**Contract**） |

## 文件命名

`v{演练时架构版本}-{英文 slug}.md`

- **演练时架构版本**：与当时最新 `arch-vX.Y` 对齐，便于和架构快照交叉对照
- **英文 slug**：英文短主题，便于 grep 与 CLI 操作

示例：

- `v1.7-gz02-app-failure.md`：v1.7 架构下，gz-02 业务实例故障演练
- `v1.7-mysql-master-down.md`：v1.7 架构下，MySQL Master 故障切换演练
- `v1.9-redis-sentinel-failover.md`：v1.9 架构下，Redis Sentinel 自动选主演练

同一类故障在不同架构版本下都需要演练时（例如入口从 gz-01 迁到 gz-02 后），新建独立文档而不是覆盖旧文档，保留演练史。

## 单篇文档建议结构

每篇 drill 至少包含以下七节，可按主题增减：

### 1. 文档元数据（文件开头，引用块格式）

```markdown
> **演练日期**：YYYY-MM-DD
> **演练时架构版本**：vX.Y
> **演练范围**：一句话说清楚要验证什么
> **演练等级**：受控演练（停业务）/ 受控演练（不停业务）/ 真实故障复盘
> **状态**：计划中 / 待执行 / 已完成
> **关联文档**：链接到 architecture / sli-slo / 相关 review 等
```

### 2. 演练目标

逐条列出本次演练要回答的问题。每条问题必须可被实测数据回答，避免"看一下"、"试试"这类模糊描述。

### 3. 前置基线

演练开始前必须验证的基线状态：所有相关 target up、近期无未恢复告警、相关服务监控指标处于稳态。前置不达标禁止开始演练。

### 4. 演练步骤

每个步骤六要素（与 runbooks 一致，但聚焦"诱发故障 + 观察 + 恢复"三段，不是架构变更）：

- ① 步骤目标
- ② 节点
- ③ 具体命令
- ④ 同时观察的窗口（Prometheus query / Alertmanager UI / 飞书群 / 容器日志）
- ⑤ 期望现象
- ⑥ 异常退出条件（演练越界时立即终止的判断标准）

### 5. 实测观察

逐项填写实测时间戳与观察结果。**禁止造数**：未实测的格子保留 _待回填_，不允许凭印象估值。建议表格格式，至少包含：

- 故障注入时刻
- Nginx upstream 摘除时刻 / ProxySQL hostgroup 切换时刻 / Sentinel 选主时刻
- Prometheus alert pending 时刻 → firing 时刻
- Alertmanager 收到时刻 → 飞书到达时刻
- 用户视角"页面 503 / 5xx" 持续时长
- 恢复指令开始时刻 → 服务可达时刻

### 6. 心得 / 反直觉发现

3–5 条 bullet。优先记录"和事前预期不一致"的观察，而非"和预期一致"的常态确认——后者属于基线，不是演练价值。

### 7. 待改进项

演练暴露出的问题分类登记：

- **告警阈值需要调整**：用具体数据论证当前阈值偏严或偏松
- **恢复步骤需要补 runbook**：演练中发现某步骤未文档化或与文档不一致
- **新发现的盲点**：是否需要新开 review，路线版本号建议
- **下次演练改进**：本次演练自身的步骤、观察窗口、节奏问题

## 写作约束

- **不写实施细节扩展到架构升级范围**：演练只验证"已有能力"，不顺势改架构；如演练触发架构变更需求，新开 proposal
- **不在演练中引入未上线的能力**：演练对象必须是当前已部署、已声明 SLO 的服务
- **不允许演练溢出**：演练等级"不停业务"则不得停业务；越界即终止并记录
- **演练文档不删改**：实测观察一旦写入即视为历史档案，新洞察追加到"待改进项"或开新演练，不修改既有数据

## 触发时机

| 场景 | 是否触发 |
|---|---|
| `Docs/sli-slo.md` 季度演练目标到期 | 是 |
| 新版本（如 v1.9 HA 配置状态修复）落地后验证新增能力 | 是 |
| 真实故障发生后即使已恢复，也按"真实故障复盘"模板补一份 | 是 |
| 仅为面试准备而做的"练手演练" | 否（无业务必要性，避免演练成本被求职焦虑驱动） |
| 周期性"为演练而演练" | 否（演练必须有具体待验证假设） |
