## 1. 演练目标

本次演练用具体实测数据回答以下问题：

1. **Nginx upstream 自动摘除**：当 gz-02 `ruoyi-admin-2:8080` 完全不可达时，Nginx 默认轮询负载是否能在用户无感知的情况下自动跳过该 upstream，把全部流量打到 gz-03 `ruoyi-admin-1`？
2. **blackbox 探测的视角差异**：blackbox-exporter 探测的是 `https://ruoyi.jjmstart.com`（Nginx 入口），不是直接探测 `100.79.132.125:8080`。在"两个 upstream 中的一个挂掉"这个场景下，`probe_success{service="ruoyi"}` 是否能保持为 1？
3. **NodeDown 告警时延**：杀掉 ruoyi-admin-2 容器**不会**触发 `NodeDown`（节点 node-exporter 仍在跑）；演练中是否有任何告警被触发？如果没有，这是合理的（业务实例级别故障目前无业务层告警），还是应当作为待改进项记入？
4. **告警链路端到端时延（如有触发）**：从故障注入 → Prometheus pending → Prometheus firing → Alertmanager 收到 → prometheus-alert 转换 → 飞书消息到达，每段耗时各是多少？是否符合 `alert_group_wait: 30s` + 各告警 `for` 时间的理论值？
5. **恢复路径可重复性**：恢复 gz-02 业务实例后，Nginx upstream 是否自动重新纳入轮询，blackbox 探测能否回到 1，无需任何人工 Nginx reload？

## 3. 演练步骤

### Phase 1 — 故障注入（gz-02）

| 要素 | 内容 |
|---|---|
| ① 步骤目标 | 让 gz-02 `ruoyi-admin-2` 容器停止运行，模拟单业务实例宕机 |
| ② 节点 | gz-02（`100.79.132.125`） |
| ③ 具体命令 | `ssh root@100.79.132.125 "docker stop ruoyi-admin-2 && date -Iseconds"` |
| ④ 观察窗口 | ① Prometheus UI: `probe_success{service="ruoyi"}` 与 `probe_duration_seconds{service="ruoyi"}` 实时图<br>② `tail -f` gz-01 Nginx access log 与 error log<br>③ Alertmanager UI 在 `Alerts` 标签实时刷新<br>④ 飞书 `docker-infra-alert` 群<br>⑤ 浏览器实时刷新 `https://ruoyi.jjmstart.com` 业务页面 |
| ⑤ 期望现象 | ① `probe_success{service="ruoyi"}` 维持 1（Nginx 把流量切到 gz-03）<br>② Nginx error log 出现少量 `connect() failed` 直至 upstream 被标记 unavailable<br>③ 浏览器刷新无 5xx，可能有偶发 1–2 秒慢响应<br>④ 不触发 `BlackboxProbeFailed`（这是验证假设之一） |
| ⑥ 异常退出条件 | `probe_success{service="ruoyi"}` 持续 ≥ 30 秒为 0；或浏览器持续 5xx 超过 10 秒 → 立即执行 Phase 3 恢复 |

### Phase 2 — 持续观察（10 分钟窗口）

| 要素 | 内容 |
|---|---|
| ① 步骤目标 | 观察 10 分钟，看是否有任何告警被触发；记录 Nginx upstream 行为是否稳定 |
| ② 节点 | bj-01（监控查询）+ 浏览器（业务验证） |
| ③ 具体命令 | 浏览器每 30 秒刷新一次 `https://ruoyi.jjmstart.com`；Prometheus UI 持续观察 `probe_success` 与 `probe_duration_seconds` |
| ④ 观察窗口 | 同 Phase 1 |
| ⑤ 期望现象 | 10 分钟内 `probe_success=1` 维持；不触发任何与 ruoyi 直接相关的告警；浏览器访问持续可用 |
| ⑥ 异常退出条件 | 任何不在 §1 演练目标范围内的告警被触发 → 立即终止演练，记录上下文 |

### Phase 3 — 恢复

| 要素 | 内容 |
|---|---|
| ① 步骤目标 | 恢复 gz-02 业务实例，验证 Nginx upstream 是否自动重新纳入 |
| ② 节点 | gz-02 |
| ③ 具体命令 | `ssh root@100.79.132.125 "docker start ruoyi-admin-2 && date -Iseconds"` |
| ④ 观察窗口 | 同 Phase 1，重点观察 Nginx error log 是否停止报 connect failed、`probe_duration_seconds` 是否恢复历史均值 |
| ⑤ 期望现象 | ruoyi-admin-2 容器 healthy；Nginx 自动恢复双 upstream 轮询；探测延迟回到基线 |
| ⑥ 异常退出条件 | 容器无法启动或启动后健康检查失败 → 走标准排障路径（查看容器日志、`.env`、ProxySQL 连通性），不再属于演练范围 |

### Phase 4 — 演练后验证

| 要素 | 内容 |
|---|---|
| ① 步骤目标 | 确认演练前后服务状态一致，无残留 |
| ② 节点 | bj-01 |
| ③ 具体命令 | 对照 §2 前置基线表，逐项重新检查 |
| ④ 期望现象 | 全部回到演练前状态；Alertmanager 无遗留 firing |
| ⑤ 异常退出条件 | 任一项不一致 → 不结束演练，先排障，记录到 §7 待改进项 |
