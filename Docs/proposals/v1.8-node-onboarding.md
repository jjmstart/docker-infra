# 架构草案 v1.8 — 六节点基础接入

> **状态**：讨论中
> **目标版本**：v1.8
> **前置版本**：v1.7（MySQL 逻辑备份 + OSS + 恢复演练）

---

## 1. 设计目标

当前 v1.7 集群由 4 个节点组成（gz-01 / gz-02 / gz-03 / bj-01），后续路线图规划了 K3s 承载无状态服务（阶段 3）和 gz-02 角色重整（阶段 2），两个阶段均需要额外节点承担新职责。

v1.8 目标是将新购两台广州节点安全、可重复地接入现有基础设施，**只做基础接入，不承载任何业务**：

1. **Tailscale 接入**：通过 Ansible role 管理 Tailscale 安装与 tailnet 加入流程，实现网络层 IaC；接入后所有后续操作走 Tailscale 内网，不依赖公网端口
2. **基础运行时**：在两台新节点上部署 `roles/docker-daemon/`，确保 Docker 环境与现有节点一致、幂等可重复
3. **监控接入**：部署 `roles/node-exporter/`，并在 gz-01 的 Prometheus 中新增 scrape target，使新节点纳入现有 Grafana 监控面板
4. **SSH 权限模型优化**：新节点采用 `admin-alex + ansible_become: true`，替代现有节点的 root 直接登录模式，还清技术债的同时形成面试可叙事的演进叙事

---

## 2. 方案范围

### 2.1 新增节点

| 节点 | 规格 | 云厂商 | OS | SSH 用户 | 阶段 1 角色 |
|---|---|---|---|---|---|
| gz-04 | 2C4G6M | 百度云·广州 | Ubuntu 24.04.1 LTS | admin-alex（sudo，需密码） | 空白节点（阶段 3 作为 K3s 控制面） |
| gz-05 | 2C4G6M | 腾讯云·广州 | Ubuntu 22.04.5 LTS | admin-alex（sudo，需密码） | 空白节点（阶段 2 接管 gz-02 职责） |

### 2.2 新增 Ansible Role

| Role | 位置 | 说明 |
|---|---|---|
| `roles/tailscale/` | 新建 | 安装 Tailscale（apt 方式）、加入 tailnet（`tailscale up --authkey`）、幂等检查（已加入则跳过） |

### 2.3 现有 Role 扩展至新节点

| Role | 变化 |
|---|---|
| `roles/docker-daemon/` | 扩展至 gz-04 / gz-05；需兼容 Ubuntu 24.04（PEP 668，`--break-system-packages`）和 Ubuntu 22.04 两种 pip 行为 |
| `roles/node-exporter/` | 扩展至 gz-04 / gz-05；端口绑定沿用现有模式（Tailscale IP:9100） |

### 2.4 新增 Playbook

| 文件 | 说明 |
|---|---|
| `playbooks/setup_tailscale.yml` | Tailscale bootstrap 专用 playbook，通过公网 IP 执行；接入完成后即切换到 Tailscale IP，后续操作不再使用此 playbook |

### 2.5 inventory 变更

**新增分组 `managed_nodes`**（承载 docker-daemon 等全节点基础 role）：

```yaml

  hosts:
    gz-01:
    gz-02:
    gz-03:
    gz-04:
    gz-05:
    bj-01:
```

**扩展 `exporter_nodes`**，新增 gz-04 / gz-05。

**新增主机变量**（gz-04 / gz-05）：

```yaml
gz-04:
  ansible_host: 182.61.39.35   # bootstrap 阶段用公网 IP；接入 tailnet 后替换为 Tailscale IP
  ansible_user: admin-alex
  ansible_become: true
  ansible_become_password: "{{ gz04_become_password }}"

gz-05:
  ansible_host: 43.138.186.227  # bootstrap 阶段用公网 IP；接入 tailnet 后替换为 Tailscale IP
  ansible_user: admin-alex
  ansible_become: true
  ansible_become_password: "{{ gz05_become_password }}"
```

### 2.6 新增 vault 密钥

在 `vault/secrets.yml` 中新增以下 3 项（ansible-vault 加密）：

| 变量名 | 用途 |
|---|---|
| `tailscale_authkey` | Tailscale 可复用 Auth Key（Reusable，非 Ephemeral），用于 `tailscale up --authkey` |
| `gz04_become_password` | gz-04 admin-alex 的 sudo 密码 |
| `gz05_become_password` | gz-05 admin-alex 的 sudo 密码 |

### 2.7 Prometheus 变更

在 gz-01 的 `roles/monitor-stack/` prometheus 配置中，为 `node_exporter` job 新增 gz-04 / gz-05 的 Tailscale IP:9100 target（Tailscale IP 接入后填入）。

## 3. 技术选型说明

### 3.1 Tailscale 接入：Ansible role，不选手动操作

| 对比项 | Ansible role 管理 | 手动登录操作 |
|---|---|---|
| 可重复性 | 幂等，节点重建后重跑 playbook 即可 | 每次重建需手动重复操作 |
| 可审计性 | Git 记录 role 内容，变更有历史 | 无记录，依赖人工记忆 |
| 面试叙事 | "网络层纳入 IaC 管理" | 无叙事价值 |
| 实施复杂度 | 略高（需写 role） | 低 |

**结论**：规模虽小，但 IaC 的核心价值在于"可重复、可审计"。Tailscale 接入作为节点生命周期的第一步，用 Ansible 管理是正确的工程实践，且 role 本身结构简单（安装 + `tailscale up` + 幂等检查），实施成本低。

### 3.2 Auth Key 选型：Reusable，不选 Ephemeral

| 类型 | Reusable Key | Ephemeral Key |
|---|---|---|
| 节点是否持久 | 是 | 否（断线后自动从 tailnet 删除） |
| 适用场景 | 长期托管的服务器节点 | 临时 CI/CD 容器、短生命周期实例 |
| 安全策略 | Key 本身可设置过期时间 + 随时吊销 | 节点退出即销毁，无需吊销 |

**结论**：gz-04 / gz-05 是长期托管节点，使用 Reusable Key；Key 存入 vault 加密，随时可吊销。

### 3.3 新节点 SSH 用户：admin-alex + become，不选 root

| 对比项 | admin-alex + become | root 直接登录 |
|---|---|---|
| 安全性 | 最小权限原则，符合生产规范 | 高风险，任何错误操作直接作用于 root |
| Ansible 兼容性 | 完全兼容，`become: true` 是标准用法 | 兼容但属于安全反模式 |
| 现有节点影响 | 不影响（旧节点保持 root，V1.8 不迁移） | — |
| 面试叙事 | "V1.8 引入新节点时有意还清 root SSH 技术债，旧节点迁移留作后续版本" | 无叙事价值 |

**结论**：现有 4 个节点延用 root 是 V1.0 时期为降低学习复杂度做出的有意识取舍；V1.8 引入新节点是修正这一模式的自然时机。不改旧节点，是因为在有生产服务的节点（gz-03 MySQL Master、bj-01 Jenkins）上变更 SSH 模型需要单独规划和验证，不应混入"基础接入"的版本主题。

### 3.4 ansible_host 过渡：注释标注，不选独立 bootstrap inventory

Tailscale bootstrap 阶段 `ansible_host` 临时使用公网 IP；接入后更新为 Tailscale IP。使用 YAML 注释在同一行说明状态（`# bootstrap 阶段用公网 IP；接入 tailnet 后替换为 Tailscale IP`），通过 runbook 明确"Phase X 执行后手动更新此行"，操作清晰可追溯，不引入额外 inventory 文件。

## 5. 执行阶段概览

| Phase | 名称 | 操作节点 | 一句话目标 |
|---|---|---|---|
| 1 | 准备工作 | bj-01（本地） | vault 新增 3 项密钥；inventory 加入 gz-04/gz-05 bootstrap 配置 |
| 2 | Tailscale bootstrap | gz-04 / gz-05（经公网 IP） | 安装 Tailscale，加入 tailnet，获取 Tailscale IP |
| 3 | inventory 切换 + 延迟实测 | bj-01（本地） | 更新 ansible_host 为 Tailscale IP；`tailscale ping` 实测七节点两两延迟 |
| 4 | Docker 环境部署 | gz-04 / gz-05（经 Tailscale） | 运行 `roles/docker-daemon/`，验证幂等 |
| 5 | node-exporter 部署 | gz-04 / gz-05（经 Tailscale） | 运行 `roles/node-exporter/`，验证幂等 |
| 6 | Prometheus 接入 | gz-01（经 Ansible） | 更新 prometheus.yml，重载配置，确认 targets up |
| 7 | 全量验收 | 全节点 | Ansible 幂等检查；Grafana 面板确认新节点指标可见 |

## 6. 验收标准

| 验收项 | 可验证的成功条件 |
|---|---|
| Tailscale 接入 | gz-04 / gz-05 出现在 Tailscale 控制台；`tailscale ping gz-04` / `gz-05` 从 bj-01 可达 |
| 内网延迟实测 | `tailscale ping` 实测 gz-04/gz-05 与现有 5 节点的两两延迟已记录（填入架构快照） |
| Docker 安装 | gz-04 / gz-05 上 `docker info` 正常；`docker network ls` 可见 `global_gateway` |
| node-exporter 运行 | Prometheus UI 中 `up{instance="<gz-04 tailscale_ip>:9100"} = 1`；gz-05 同 |
| Grafana 可见 | Grafana Node Exporter 面板中新节点主机指标（CPU / 内存 / 磁盘）可见 |
| Ansible 幂等 | 对 gz-04/gz-05 重复执行 docker-daemon + node-exporter role，`changed=0 failed=0` |
| 敏感信息 | `vault/secrets.yml` 保持 AES256 加密；inventory 中不含明文密码（引用 vault 变量） |
