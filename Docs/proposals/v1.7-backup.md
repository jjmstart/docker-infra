# 架构草案 v1.7 — 备份恢复最小闭环

> **状态**：讨论中
> **目标版本**：v1.7
> **前置版本**：v1.6（私有 Registry + CI/CD 全链路）

---

## 1. 设计目标

当前 v1.6 架构已具备完整的应用交付链路（CI/CD + Smoke Test），但数据层缺乏经过演练验证的备份恢复机制。MySQL 虽然有主从复制（gz-03 主库、gz-02 实时从库、bj-01 延迟从库 3600s），但主从复制**无法防止数据误删除**：任何 `DROP TABLE` 或无条件 `DELETE` 都会实时同步到所有从库。

v1.7 目标是建立一条最小但可运转的备份闭环：

1. **定期全量逻辑备份**：对 gz-03 MySQL Master 每日执行 `mysqldump`，备份文件带时间戳，本地保留 3 天
2. **上传对象存储**：备份完成后自动上传至阿里云 OSS，远端保留 30 天（由 OSS 生命周期策略管理）
3. **恢复演练**：从 OSS 下载备份文件，在 bj-01 起临时测试库（端口 3307），导入并校验数据完整性，记录 RTO
4. **RPO/RTO 说明**：完整记录备份策略的数据保护上限，形成可量化的运营结论

## 2. 方案范围

### 2.1 新增组件

| 组件 | 说明 |
|---|---|
| `roles/mysql-backup/` | 新建 Ansible role，管理备份脚本下发、ossutil 安装和 cron 配置 |
| `mysql_backup.sh`（模板） | 备份脚本（Jinja2 模板），执行 mysqldump → 压缩 → 本地清理旧文件 → ossutil 上传 |
| ossutil | 阿里云官方 OSS CLI 工具，固定版本，下发到 gz-03，通过 Ansible `get_url` 安装 |
| `playbooks/setup_backup.yml` | 独立 playbook，针对 gz-03 执行 mysql-backup role |

### 2.2 新增 vault 密钥

在 `vault/secrets.yml` 中新增以下 4 项（ansible-vault 加密存储）：

| 变量名 | 用途 |
|---|---|
| `oss_access_key_id` | 阿里云 RAM 用户 AccessKey ID |
| `oss_access_key_secret` | 阿里云 RAM 用户 AccessKey Secret |
| `oss_bucket_name` | OSS bucket 名称 |
| `oss_endpoint` | OSS endpoint（如 `oss-cn-guangzhou.aliyuncs.com`） |

### 2.3 新增 inventory 变量

在 `inventory/group_vars/all.yml` 中新增以下公共变量（无敏感信息）：

```yaml
mysql_backup_dir: /opt/docker/backup/mysql
mysql_backup_retention_days: 3
mysql_backup_cron_hour: "2"
mysql_backup_cron_minute: "0"
ossutil_version: "1.7.18"        # 固定版本，保证幂等
ossutil_install_dir: /usr/local/bin
```

### 2.4 涉及节点

| 节点 | 角色 | v1.7 新增操作 |
|---|---|---|
| gz-03 | MySQL Master | 下发备份脚本、安装 ossutil、配置 cron |
| bj-01 | Ansible 控制 + 恢复演练节点 | 下载备份、起临时库 mysql-restore-test（端口 3307）、导入、校验 |

## 3. 技术选型说明

### 3.1 mysqldump，不选 mydumper

| 对比项 | mysqldump | mydumper |
|---|---|---|
| 安装 | 随 MySQL 客户端自带，无需额外安装 | 需要额外安装二进制 |
| 并发 | 单线程 | 多线程并行 |
| 适用数据量 | 中小数据量（MB 级 ~ 数十 GB） | 大数据量（GB 级 ~ TB 级） |
| 输出格式 | 单个 `.sql` 文件，便于理解和恢复 | 目录 + 多文件，恢复需配套 `myloader` |
| 锁表 | `--single-transaction` 对 InnoDB 不锁表 | 同样支持无锁备份 |
| 文档复杂度 | 低，命令直观，步骤单一 | 略高，需要解释 myloader 的配合 |

**结论**：若依 MySQL 数据量估计在几十 MB 量级，mysqldump 完全胜任。引入 mydumper 在当前规模下是过度设计，且增加文档与操作复杂度而无收益。若未来数据库增长到需要分钟级备份窗口，再评估升级。

### 3.2 阿里云 OSS，不选腾讯云 COS

| 对比项 | 阿里云 OSS | 腾讯云 COS |
|---|---|---|
| 工具 | ossutil | coscli |
| 市场认知度（国内） | 更高，业界对象存储事实标准 | 次之 |
| 生命周期策略 | 支持，控制台一行配置 | 支持，功能相同 |
| 本项目云厂商分布 | gz-01 阿里云，接受度高 | gz-02 腾讯云 |

**结论**：两个云账号均已有，选 OSS 是因为国内市场认知度更高、ossutil 工具链成熟、社区文档齐全；gz-01 本身就是阿里云节点，账号体系已就位，减少一份额外凭据管理。

### 3.3 备份源选 gz-03 主库，不选从库

备份从库可以减少对主库的影响，但：
- 若依当前数据量小，mysqldump `--single-transaction` 对 InnoDB 主库读写几乎无影响
- bj-01 延迟从库数据延迟 3600s，不是最新数据，备份不合适作为全量基准
- gz-02 实时从库理论上可用，但从库主备切换时数据状态需要额外确认
- **备份主库**是中小公司最常见且最直接的做法，路径短、可追溯性高

### 3.4 恢复演练在 bj-01 的临时测试库

- bj-01 资源最充裕（4C16G），承载多个运维工具，磁盘空间够用
- bj-01 已有延迟从库运行在 `127.0.0.1:3306`，临时测试库使用端口 **3307** 避免冲突
- 临时库容器名 `mysql-restore-test`，演练后立即删除，不污染生产
- 演练全程走 Tailscale 内网（ossutil 下载、docker run），不依赖公网端口

## 4. 影响范围

| 影响项 | 说明 |
|---|---|
| gz-03 磁盘空间 | 新增 `/opt/docker/backup/mysql/` 目录，保留 3 天备份；若依数据库估计 < 100MB，压缩后每份 < 10MB，3 天占用 < 30MB，影响可忽略 |
| gz-03 cron | 新增每天 02:00 的 cron 任务，mysqldump 期间 CPU 有短暂峰值（秒级） |
| bj-01 磁盘空间 | 恢复演练时临时下载备份文件（< 10MB），演练完毕删除 |
| bj-01 端口 3307 | 临时测试库存在期间占用，演练完毕 `docker rm -f` 释放 |
| vault/secrets.yml | 新增 4 个 OSS 凭据变量，对现有加密文件做增量更新 |
| 现有主从复制 | 无影响。mysqldump 使用 `--single-transaction`，恢复演练在独立临时库，主从链路完全隔离 |
| CI/CD Pipeline | 无影响。本次不涉及 Jenkinsfile 修改 |

## 5. 验收标准

| 验收项 | 可验证的成功条件 |
|---|---|
| Ansible 幂等 | `ansible-playbook playbooks/setup_backup.yml --check` 全量重复执行 `changed=0 failed=0` |
| 脚本下发 | gz-03 `/opt/docker/backup/mysql/` 目录存在，`mysql_backup.sh` 权限为 `0750` |
| cron 配置 | `crontab -l`（gz-03 root）可见 `0 2 * * * /opt/docker/backup/mysql/mysql_backup.sh` |
| 备份文件生成 | 手动执行脚本后，gz-03 本地产生 `ruoyi_backup_YYYYMMDD_HHMMSS.sql.gz` 文件，大小 > 0 |
| OSS 上传 | 在 OSS 控制台或 `ossutil ls` 可见对应备份文件 |
| 本地保留清理 | 执行多次备份后，本地只保留最近 3 天文件，旧文件被自动清理 |
| 恢复演练 | 从 OSS 下载备份文件，导入 bj-01 临时测试库（端口 3307），执行校验 SQL 返回预期结果 |
| 恢复耗时记录 | runbook 中记录完整恢复流程的实测耗时（含下载 + 导入 + 校验），作为 RTO 基准 |
| 敏感信息 | `vault/secrets.yml` 保持 AES256 加密状态；`all.yml`、脚本模板中不含明文凭据 |

## 6. RPO / RTO 估算

| 指标 | 说明 | 估算值 |
|---|---|---|
| **RPO**（最多丢失多少数据） | 每日一次全量备份，最坏情况下丢失备份窗口内 24 小时的数据变更 | **最大 24 小时** |
| **RPO（辅助）** | 结合 bj-01 延迟从库（3600s 延迟窗口），针对误操作场景可将实际 RPO 缩短到 1 小时内 | **辅助 1 小时** |
| **RTO**（恢复需要多长时间） | 含确认问题（~5min）+ 从 OSS 下载备份（~1min）+ 导入（~3-5min）+ 校验（~2min） | **估算 15~30 分钟** |

**提升 RPO 的路径（v1.7 不做，作为后续演进点）**：开启 Binlog 定期归档，实现 PITR（Point-in-Time Recovery），可将 RPO 从 24h 降至分钟级。这需要持续追踪和归档 Binlog 文件，复杂度显著上升，留给后续版本评估。

## 7. 待确认问题

> 以下问题已在草案讨论阶段确认，记录结论供参考。

| 问题 | 结论 |
|---|---|
| OSS 还是 COS？ | **OSS**：两个账号均已有，选阿里云 OSS 因国内市场认知度更高 |
| mysqldump 还是 mydumper？ | **mysqldump**：数据量小，单线程足够，无需额外安装，命令与恢复流程直观 |
| 备份源选主库还是从库？ | **主库（gz-03）**：数据最新，无延迟，`--single-transaction` 对 InnoDB 不锁表 |
| 恢复演练的临时库端口？ | **3307**：避免与 bj-01 现有延迟从库（3306）冲突 |
| 是否做 Binlog 备份（PITR）？ | **不做**：v1.7 聚焦逻辑备份最小闭环，PITR 留给后续版本 |
| 是否做 Redis 备份？ | **不做**：Redis 主要作为缓存，数据丢失可接受；v1.7 范围克制 |
| 是否做备份监控告警？ | **不做**：告警可后续补充，v1.7 聚焦备份-恢复链路本身 |
