下述三个 PoC 是 v1.9 实施第一天就要做的事——任何一个失败都会让整个版本方案需要重新设计，所以前置 1-2 天用最小实验验证清楚，比 role 改一半发现策略错的成本低一个数量级。

### 5.1 PoC 1：Druid 是否能透传 `jdbc:mysql:loadbalance://` URL

**问题本质**

应用层连数据库通常经过三层：

```
应用代码 → Druid 连接池 → MySQL Connector/J（JDBC 驱动）→ ProxySQL → MySQL
```

MySQL Connector/J 这个驱动支持多种连接协议前缀：

| URL 前缀 | 行为 |
|---|---|
| `jdbc:mysql://host/db` | 标准单端点（当前若依用的） |
| `jdbc:mysql:loadbalance://h1,h2/db` | 多端点请求级负载均衡（**v1.9 要用的**） |
| `jdbc:mysql:replication://master,slave/db` | 显式主从模式，`readOnly=true` 时路由到 slave |

但**驱动支持 ≠ 连接池支持**。Druid 夹在应用代码和 JDBC 驱动中间，会"解析"应用传入的 JDBC URL：

- 早期 Druid 版本对非标 URL 协议有严格校验，遇到 `mysql:loadbalance` 前缀可能直接抛 `IllegalArgumentException`
- 即使能创建连接，Druid 的探活机制（`validationQuery=SELECT 1`、`testOnBorrow`、`testWhileIdle`）在多端点协议下行为不确定——一条连接背后可能是 host1 也可能是 host2，Druid 不知道"这次探活打到哪个后端"，故障后能不能正确驱逐失活连接是问号
- Druid 的连接池统计（活跃数 / 等待数）和 Connector/J 的多端点状态怎么对账，文档不够明确

**验证步骤**

1. 在 bj-01 起一个**最小化 Spring Boot demo**（参考若依 `ruoyi-admin/pom.xml` 用相同版本 Druid + `mysql-connector-j`；不引入任何业务依赖；只有一个数据源 + 一个 `/health` 接口跑 `SELECT 1`）。这个 demo 是"测试白车"——空载、变量少，结果易于解读
2. demo 的 `application.yml` 配置：

   ```yaml
   spring:
     datasource:
       url: jdbc:mysql:loadbalance://100.92.5.116:6033,100.79.132.125:6033/ry-vue?useUnicode=true&characterEncoding=utf8
       username: <test_user>
       password: <test_password>
       druid:
         validation-query: SELECT 1
         test-on-borrow: true
         test-while-idle: true
   ```

3. 启动 demo，**观察启动日志是否报错**——重点看是否出现 `IllegalArgumentException` / `unsupported url`
4. 调用 `/health` 看返回 200；连续打 100 次，观察 Druid 监控页（`/druid/sql.html`）的连接来源分布是否符合预期（应该在两个 ProxySQL 之间均衡）
5. **模拟故障**：`docker stop proxysql`（gz-03 上）；同时持续 `for i in {1..60}; do curl /health; sleep 1; done` 观察
6. **期望结果**：30 秒内 `/health` 有短暂失败（< 10 次），随后恢复 200——这说明 Druid + Connector/J 多端点 failover 行为正常

**失败时的应对方案**

| 失败模式 | 应对方案 |
|---|---|
| 启动直接报 URL 解析错 | **方案 B**：应用层做双数据源 + Spring `AbstractRoutingDataSource`——在 `application.yml` 配两个独立 DataSource（指向 gz-03 / gz-02 ProxySQL），自定义路由策略让主源失败时切到备用源。工作量翻倍但完全自主，且不依赖驱动版本兼容性 |
| 启动正常但 failover 不工作（持续打到死的那一端） | **方案 C**：换连接池——用 HikariCP（Spring Boot 2.x 默认）替代 Druid。HikariCP 对 JDBC URL 透传更宽松，但失去 Druid 的监控页。需要评估是否值得 |
| failover 工作但探活 SQL 出错 | **方案 D**：调整 Druid 探活参数（`validation-query-timeout`、`time-between-eviction-runs-millis`）让它能更快驱逐失活连接；如果调不通也回退到方案 B |

### 5.2 PoC 2：ProxySQL Cluster 在已有 datadir 上首次启用是否需要走 Admin SQL 兜底

**问题本质**

ProxySQL 启动时有个"datadir 行为差异"——

```
启动时检查 /var/lib/proxysql/proxysql.db 是否存在：
├── 不存在 → 从 proxysql.cnf 读全部配置，初始化 SQLite runtime DB
└── 存在   → 只读 cnf 的 admin_variables 段，其他配置全部从 runtime DB 读
```

gz-03 现在已经在跑，datadir 里已经有 `proxysql.db`。我们在 cnf 里加：

- `admin_variables.cluster_username/cluster_password` → 在 `admin_variables` 段内，**应该会被读**
- `proxysql_servers=(...)` 段（声明 peer 节点列表）→ **不在 `admin_variables` 段内**，按上面规则可能被忽略

如果 `proxysql_servers` 段被忽略，gz-02 ProxySQL 上线后两节点之间无法识别 peer，**Cluster 根本没建起来**——看起来没报错但实际上和"两个互不知道对方的独立 ProxySQL"没区别。这是个**静默失效**陷阱，和叙事点 #7（Jenkins Excluded Regions 静默失效）是同一族陷阱。

**验证步骤**

1. 在 bj-01 起一个**测试用 ProxySQL 容器**（独立 datadir，不影响线上），先用"无 Cluster 配置"的 cnf 启动一次，让它初始化 datadir：

   ```bash
   docker run -d --name proxysql-test \
     -v /tmp/proxysql-test-data:/var/lib/proxysql \
     -v /tmp/proxysql-test.cnf:/etc/proxysql.cnf \
     proxysql/proxysql:latest
   ```

2. 确认初始化成功后，**修改 cnf 加入 Cluster 配置**：

   ```ini
   admin_variables=
   {
       ...
       cluster_username="cluster_user"
       cluster_password="..."
       cluster_check_interval_ms=1000
   }
   
   proxysql_servers=
   (
       { hostname="100.79.132.125", port=6032, weight=0, comment="gz-02 peer" }
   )
   ```

3. `docker restart proxysql-test`
4. 进入 Admin Interface 查询：

   ```bash
   mysql -h127.0.0.1 -P6032 -u<admin> -p<password> \
     -e "SELECT * FROM runtime_proxysql_servers;"
   ```

5. **观察结果**：
   - 表里出现 peer 配置 → cnf 改动生效，**幸运情况**
   - 表为空 → cnf `proxysql_servers` 段被忽略，需要走 Admin SQL 兜底

**失败时的应对方案**

如果 cnf 段不生效，v1.9 实施流程改为"配置文件 + Admin SQL 兜底"：

```bash
# 1. 渲染 cnf（包括 admin_variables.cluster_*，但不依赖 proxysql_servers 段）
# 2. docker compose up
# 3. ★ 通过 Admin SQL 注入 Cluster 配置（roles/proxysql/tasks/main.yml 新增）
mysql -h<host> -P6032 -u<admin> -p<password> << 'EOF'
  INSERT INTO proxysql_servers(hostname, port, weight, comment) 
    VALUES ('100.79.132.125', 6032, 0, 'gz-02 proxysql peer');
  LOAD PROXYSQL SERVERS TO RUNTIME;
  SAVE PROXYSQL SERVERS TO DISK;
EOF
# 4. 验证：SELECT * FROM stats_proxysql_servers_status;
```

**注意**：首次部署时 gz-02 ProxySQL 还没起来，gz-03 单边写入 `proxysql_servers` 是 OK 的（peer 处于 OFFLINE 状态），等 gz-02 起来后会自动切到 ONLINE。

这条兜底逻辑要写进 role 主流程（不是"出错才补"），保证幂等：任何时候重跑 role，`INSERT ... ON DUPLICATE KEY` 风格的语义都能保持期望状态。

### 5.3 PoC 3：两节点 cnf 渲染差异是否会引发 Cluster 同步震荡

**问题本质**

ProxySQL Cluster 是 **push-based 自动同步**协议，触发条件是"我的 runtime config checksum 和 peer 不一样"：

```
节点 A 启动 → 算自己 runtime config checksum → 和 peer B 对比
              ↓
              checksum 不同？
              ├── A 的 epoch 更新 → push 给 B（B 全量覆盖）
              └── B 的 epoch 更新 → pull from B（A 全量覆盖）
```

我们的 role 用同一个 Jinja 模板渲染两节点 cnf。模板里如果有"按节点变化的字段"，两节点 cnf 渲染产物就会有 diff → 首次启动后 runtime checksum 不同 → 自动同步触发。

可能造成差异的位置：

- `comment="gz-03 MySQL Master"` 这种带 hostname 的字符串
- 用 `inventory_hostname` 或 `ansible_host` 渲染出的 server 列表顺序差异
- ProxySQL 自己的 hostname / IP 引用（虽然现在 cnf 没用，但未来扩展可能引入）

**最坏情况**：每次 Ansible 重跑都覆盖一次 cnf restart → 节点 A 拉回到自己版本 → 立刻被 peer 同步覆盖 → 下次重跑又拉回 → **配置反复震荡**。从外部看是"配置好像一直在变"，排查极其费时。

**验证步骤**

1. 在 bj-01 用 `--check` 模式分别模拟两节点视角渲染 cnf：

   ```bash
   ansible-playbook playbooks/setup_database.yml --tags proxysql \
     --limit gz-03 --check --diff > /tmp/proxysql-gz03.cnf 2>&1
   ansible-playbook playbooks/setup_database.yml --tags proxysql \
     --limit gz-02 --check --diff > /tmp/proxysql-gz02.cnf 2>&1
   diff /tmp/proxysql-gz03.cnf /tmp/proxysql-gz02.cnf
   ```

2. **看 diff 是否有差异**——理论上两节点 cnf 应当完全一致；如有差异说明模板有按节点变化的字段
3. 真实启用 Cluster 后，连续重跑 Ansible 3 次，每次跑完查：

   ```sql
   SELECT * FROM stats_proxysql_servers_status;   -- 看 checksum 和 epoch
   SELECT * FROM stats_proxysql_servers_metrics;  -- 看同步频次
   ```

4. **期望结果**：epoch 在 3 次重跑之间**保持稳定**，checksum **不变**——说明无震荡
5. **异常信号**：epoch 持续递增、checksum 在重跑之间反复变化、`stats_proxysql_servers_metrics` 的同步事件计数器快速上涨——震荡发生

**失败时的应对方案**

如果出现震荡：**纯粹双路径策略**——把 cnf 的角色彻底收窄到"只承担初始 bootstrap"，业务配置全部走 Admin SQL：

```
cnf 只保留：
  - datadir 指定
  - admin_variables（含 cluster_*）
  - mysql_variables（threads / max_connections 等静态运行参数）

cnf 不再渲染：
  - mysql_servers     ← 改走 Admin SQL INSERT + LOAD ... TO RUNTIME + SAVE ... TO DISK
  - mysql_users       ← 同上
  - mysql_query_rules ← 同上
  - proxysql_servers  ← 同上（与 PoC 2 兜底方案合并）
```

这样两节点 cnf 完全一致（只剩静态运行参数），Cluster 同步只在"业务配置真正变化"时触发，符合官方推荐的"以 runtime DB 为真相"原则。

这套策略实际上是 ProxySQL Cluster 的**官方推荐落地姿势**，但首次实施要付出额外工作量（role 要写更多 Admin SQL 任务），所以先 PoC 验证"简易方案能不能用"，不行再升级到纯双路径。
