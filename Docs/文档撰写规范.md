# 文档撰写规范 — 如何新增一个架构版本

> **本文件用途**：将本文件提供给 AI，它即可独立完成新版本文档的撰写，并知道需要同步更新哪些已有文件。无需额外说明。

> **V1.4 之后的重要约定**：主文档位置随控制仓库迁移到 `bj-01:/opt/docker-infra/Docs/`。各节点 `/opt/docker/Docs/` 若存在，仅视为历史副本或运行目录中的旧文件，不再作为主文档来源。

---

## 0. 文档体系概览

`/opt/docker-infra/Docs/` 下有三个子文件夹，每次架构演进需要在三个文件夹中各新增一份文件，并更新四处索引。

```
/opt/docker-infra/Docs/
├── README.md                          ← 根目录索引，每次都要更新
├── 文档撰写规范.md                     ← 本文件，无需修改
├── architecture/
│   ├── README.md                      ← 每次都要更新
│   ├── v1.0.md                        ← 历史归档，只读
│   ├── v1.1.md                        ← 历史归档，只读
│   └── v1.2.md                        ← 当前最新（新版本写完后改为历史归档）
├── runbooks/
│   ├── README.md                      ← 每次都要更新
│   ├── v1.0-to-v1.1.md
│   └── v1.1-to-v1.2.md
└── retrospectives/
    ├── README.md                      ← 每次都要更新
    ├── v1.1-retrospective.md
    └── v1.2-retrospective.md
```

---

## 1. 每次新版本需要完成的操作清单

假设当前最新版本为 `vX.Y`，新版本为 `vX.Z`，逐项完成以下操作：

| 序号 | 操作 | 文件路径 |
|------|------|----------|
| 1 | **新建**架构快照 | `architecture/vX.Z.md` |
| 2 | **新建**操作手册 | `runbooks/vX.Y-to-vX.Z.md` |
| 3 | **新建**演进总结 | `retrospectives/vX.Z-retrospective.md` |
| 4 | **修改**旧版架构快照：文档说明中"当前最新"→"历史归档" | `architecture/vX.Y.md` |
| 5 | **更新** architecture/README.md：旧版状态改为"历史归档"，新增新版行，标注新版为"当前最新" | `architecture/README.md` |
| 6 | **更新** runbooks/README.md：新增手册行 | `runbooks/README.md` |
| 7 | **更新** retrospectives/README.md：新增总结行 | `retrospectives/README.md` |
| 8 | **更新** Docs/README.md：各索引表格新增新版行，旧版标注归档 | `Docs/README.md` |

---

## 2. architecture/vX.Z.md — 架构快照撰写规范

**用途**：AI 快速把握集群整体状态的参考快照，扔给 AI 就能完整理解四台服务器的角色、服务、关系与网络拓扑。不包含操作步骤，不包含演进过程说明。

**文件命名**：`v{版本号}.md`，如 `v1.3.md`、`v2.0.md`

### 必须包含的章节（按顺序）

#### 章节 1：文档说明（文件第一行之后，H2 标题）

一到两句话说明本版本的核心变更，并标注是否为当前最新版本。

```markdown
## 文档说明

V1.3 相对 V1.2 的核心变更：[一句话描述变更内容]。**本版本为当前最新版本。**
```

历史版本改为：
```markdown
**本版本为历史归档，非当前最新版本，请参阅 [v1.3.md](v1.3.md)。**
```

---

#### 章节 2：AI 上下文引导（Context Bootstrap）

专供 AI 快速建立上下文，人工阅读可跳过。固定包含以下四个子节：

**2.1 仓库根目录与管理方式**（固定内容，每版相同）

```markdown
**仓库根目录与管理方式**

- 仓库根目录：`/opt/docker`
- 所有服务均以 Docker Compose 管理
- 网络：`global_gateway`（`docker network create global_gateway` 已在各节点存在）
- 节点 hostname 约定：`gz-01` / `gz-02` / `gz-03` / `bj-01`
```

**2.2 节点互联方式**（固定内容，每版相同）

```markdown
**节点互联方式**

所有节点通过 **Tailscale WireGuard** 加密隧道互联，不依赖公网端口暴露。
```

**2.3 关键文件路径索引**（每版按实际更新）

用代码块或表格列出本版本涉及的关键文件路径，格式参考：

```
/opt/docker/
├── backend/
│   ├── mysql/docker-compose.yml     # gz-03 MySQL 主库
│   ├── proxysql/docker-compose.yml  # gz-03 ProxySQL
│   └── ...
├── monitor/
│   └── prometheus/prometheus.yml    # gz-01 Prometheus 采集配置
└── Docs/
    └── ...（文档目录本身）
```

**2.4 本版本核心技术决策**（每版按实际更新）

表格格式，三列：决策点 / 选型 / 理由。只列本版本新增或变更的决策，已有决策不重复列出。

```markdown
| 决策点 | 选型 | 理由 |
|--------|------|------|
| [决策点名称] | [具体选型] | [选择该方案的核心原因，1-2句] |
```

---

#### 章节 3：节点总览

固定六列表格，节点角色一列要反映本版本最新角色（如新增 MySQL Master、ProxySQL 等）：

```markdown
| 节点 | 配置 | 云厂商 | Tailscale IP | 公网 IP | 角色 |
|------|------|--------|--------------|---------|------|
| gz-01 | 2C2G | 阿里云·广州 | 100.117.7.75 | 8.163.9.112 | [本版本角色] |
| gz-02 | 4C4G | 腾讯云·广州 | 100.79.132.125 | 123.207.59.177 | [本版本角色] |
| gz-03 | 4C8G | 火山引擎·广州 | 100.92.5.116 | 118.145.70.66 | [本版本角色] |
| bj-01 | 4C16G | 京东云·北京 | 100.118.69.78 | 117.72.174.148 | [本版本角色] |
```

节点配置、云厂商、IP 为固定值，不得随意修改。

---

#### 章节 4：各节点服务详情

每个节点一个 H3 子章节，格式：

```markdown
### gz-XX（节点角色名）

| 服务 | 容器名 | 端口 | 说明 |
|------|--------|------|------|
| [服务名] | [容器名] | [端口绑定] | [说明，本版新增的服务标注"**新增**"，有变化的标注"**V1.X**：..."，无变化的标注"无变化"] |
```

端口绑定格式：`{Tailscale IP}:{端口}` 或 `127.0.0.1:{端口}` 或 `容器内 {端口}`，与实际 compose 文件一致。

---

#### 章节 5：架构拓扑图

**保留原文的 ASCII 图或 mermaid 图，不改写已有图形结构。** 在此基础上按本版本变更增减节点/连线/标注。

- ASCII 图：用于快速文本预览
- Mermaid 图：用 `flowchart TB` 格式，每个节点用 `subgraph` 包裹，节点标签包含 Tailscale IP

如本版本有新增组件，在 ASCII 图和 mermaid 图中都要体现。

---

#### 章节 6：网络互联

固定五条链路（如无新增跨节点通信则保持不变），表格三列：链路 / 延迟 / 用途。

```markdown
| 链路 | 延迟 | 用途 |
|------|------|------|
| gz-01 ↔ gz-03 | ~5ms | [本版本该链路承载的服务] |
| gz-01 ↔ gz-02 | ~5ms | [本版本该链路承载的服务] |
| gz-01 ↔ bj-01 | ~35ms | [本版本该链路承载的服务] |
| gz-03 ↔ gz-02 | ~5ms | [本版本该链路承载的服务] |
| gz-03 ↔ bj-01 | ~35ms | [本版本该链路承载的服务] |
```

延迟值固定（~5ms / ~35ms），不得修改。

---

#### 章节 7：与上一版本的差异（v1.1 起必须包含）

1-3 条 bullet，简述本版本相对上一版本的变更点，**不展开操作细节**，操作细节属于 runbooks。

```markdown
## 与上一版本的差异（相对 vX.Y）

- **[变更点 1 标题]**：一句话说明做了什么，结果是什么。
- **[变更点 2 标题]**：同上。
- **[变更点 3 标题]**（可选）：同上。
```

---

### 写作禁止项

- 不得包含 Phase 操作步骤（属于 runbooks）
- 不得包含踩坑记录（属于 retrospectives）
- 不得包含"演进方向"（面向未来的规划，不是当前状态快照）
- 不得捏造节点 IP、配置规格等事实信息

## 3. runbooks/vX.Y-to-vX.Z.md — 操作手册撰写规范

**用途**：扔给 AI 后能立即进入操作协助模式，知道每一步做什么、怎么验证、出错如何回滚。

**文件命名**：`v{起始版本}-to-v{目标版本}.md`，如 `v1.2-to-v1.3.md`

### 必须包含的章节（按顺序）

#### 章节 1：演进概述

一句话，说明本次做了什么、为什么。

#### 章节 2：前置条件

表格，两列：项目 / 要求。至少包含：当前架构版本、涉及节点的基线状态（相关容器是否运行、磁盘空间要求）、节点间网络连通性。

#### 章节 3：节点与文件速查

表格，四列：节点 / Tailscale IP / 角色 / 涉及关键文件。只列本次演进涉及的节点和文件。

#### 章节 4：交互约定（固定内容，每版相同）

```markdown
## 交互约定

- **分步交互**：每次只给当前步骤，等反馈后再继续。
- **每步格式**：① 目标说明 ② 节点 ③ 具体命令 ④ 预期输出 ⑤ 验证方法 ⑥ 回滚方式。
- **遇到报错**：提供完整输出，再给出排障方向。
- **跨节点操作**：每次明确告知在哪个节点执行。
```

#### 章节 5：Phase 清单（总览表格）

表格，四列：Phase / 名称 / 所在节点 / 一句话目标。让读者在操作前有全局认知。

#### 章节 6：各 Phase 详细步骤

每个 Phase 一个 H3 子章节，固定包含以下六个要素：

```
**① 目标说明**：做什么，为什么。

**② 节点**：节点名（Tailscale IP）

**③ 具体命令**：可直接执行的 shell 命令或 SQL，用代码块包裹。

**④ 预期输出**：命令执行成功时的终端输出，用代码块包裹。

**⑤ 验证方法**：独立的验证命令，确认该步骤效果已生效。

**⑥ 回滚方式**：该步骤失败或需撤销时的回退操作。
```

**命令规范**：
- `docker exec` + heredoc 输入时，必须用 `-i` 不加 `-t`（`-t` 与非交互模式冲突）
- SQL heredoc 结束符使用 `'EOF'`（带引号，防止变量展开）
- 密码类变量引用格式：`"${MYSQL_ROOT_PASSWORD}"`（从 `.env` 文件读取）
- 跨节点文件传输走 Tailscale IP，不走公网

#### 章节 7：验收标准

表格，两列：指标 / 目标值。用可量化的指标（如 `Seconds_Behind_Source=0`、`targets 3/3 up`）而非模糊描述。

## 4. retrospectives/vX.Z-retrospective.md — 演进总结撰写规范

**用途**：演进完成后的复盘文档，帮助理解技术决策背后的原因，总结经验教训。面向有一定技术基础但不一定精通该方向的读者。

**文件命名**：`v{版本号}-retrospective.md`，如 `v1.3-retrospective.md`

### 必须包含的章节（按顺序）

#### 章节 1：本次演进一句话摘要

一句话，概括本次做了什么，最终效果是什么（可量化则量化）。

#### 章节 2：最终达成的架构状态

表格，三列：Phase / 完成内容 / 关键验证指标。每行对应一个 Phase，指标要用可观测的数值。

#### 章节 3：重要技术决策解释

Q/A 形式，每个决策点一对 Q/A。Q 的格式：`**Q：为什么选择 X 而不是 Y？**`

写作要求：
- 以初级工程师/新人能看懂的方式解释（但不必过度简化）
- 先说"如果不这么做会怎样"，再说"这么做的好处"，形成对比
- 引用本项目的具体数据（如 ~5ms ACK、3600s 延迟窗口），而非泛泛而谈
- 只解释本版本新增的技术决策，已在旧版本解释过的不重复

#### 章节 4：踩坑记录

表格，四列：坑点描述 / 根因 / 解决方法 / 下次规避手段。

- 只记录**实际发生**的问题，不虚构
- 如有已知必写踩坑条目（如 `docker exec -it` 与 heredoc 冲突），必须写入
- 根因要说清楚技术原理，不能只写"配置错误"

#### 章节 5：操作心得

3-5 条 bullet，提炼本次实操中值得记住的经验。格式：**加粗标题**：具体说明。

要求：
- 具体、可操作，避免"要细心"这类无效总结
- 优先记录"反直觉"或"容易被忽略"的经验
- 可包含工具使用技巧、命令习惯、验证方式等

#### 章节 6：遗留问题 / 下一步建议

用 bullet 列出：本次未解决的风险点、下一版本应优先处理的事项、需要补充的自动化/监控项。

## 5. README 索引文件更新规范

每次新增版本后，以下四个 README 文件必须同步更新。

### architecture/README.md

版本索引表格新增一行（新版本），并将原"当前最新"行的状态改为"历史归档"：

```markdown
| [vX.Z.md](vX.Z.md) | VX.Z | [一句话说明] | **当前最新** ✓ |
```

### runbooks/README.md

手册索引表格新增一行：

```markdown
| [vX.Y-to-vX.Z.md](vX.Y-to-vX.Z.md) | [演进内容摘要] | [涉及节点列表] |
```

### retrospectives/README.md

总结索引表格新增一行（完成日期填写实际完成的年月）：

```markdown
| [vX.Z-retrospective.md](vX.Z-retrospective.md) | VX.Z | [完成年月] | [一句话摘要] |
```

### Docs/README.md

三个索引表格（architecture、runbooks、retrospectives）各新增对应行，architecture 表格同时将旧版标注修改。

## 6. 全局写作规范

| 规范项 | 要求 |
|--------|------|
| 语言 | 全文使用中文；代码块、命令、文件路径、容器名保持英文原样 |
| 表格 | 统一使用 Markdown 表格（`\|` 分隔），表头下方加 `\|---|` 分隔行 |
| 代码块 | 使用三反引号 + 语言标识（` ```bash ` / ` ```sql ` / ` ```yaml ` / ` ```markdown `），无语言时用空代码块 |
| IP 地址 | 节点 IP 固定值：gz-01=100.117.7.75、gz-02=100.79.132.125、gz-03=100.92.5.116、bj-01=100.118.69.78，不得修改 |
| 延迟值 | 固定值：同城（gz-01/gz-02/gz-03 三节点之间）~5ms；跨城（任意节点 ↔ bj-01）~35ms，不得修改 |
| 端口格式 | 绑定端口写法：`{IP}:{宿主端口}:{容器端口}` 或简写 `{IP}:{端口}` |
| 节点规格 | 固定值：gz-01=2C2G、gz-02=4C4G、gz-03=4C8G、bj-01=4C16G，不得修改 |
| 忠实于事实 | 不捏造架构事实；retrospectives 的技术决策解释可适当扩写，但必须与源文件技术选型一致 |
| 避免的写法 | 不在架构快照中写操作步骤；不在操作手册中写技术决策原因；不在演进总结中列原始命令清单 |

## 7. 快速参照：各文件夹定位一句话

| 文件夹 | 定位 | 触发时机 |
|--------|------|----------|
| `architecture/` | 集群**现在是什么样的** | 每次架构状态变化时新增一版 |
| `runbooks/` | 集群**怎么从 A 变成 B** | 每次执行架构演进时撰写 |
| `retrospectives/` | 集群**演进过程中学到了什么** | 演进完成后复盘时撰写 |
