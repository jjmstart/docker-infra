# 操作手册：v1.4 → v1.5

## 演进概述

在 V1.4 的 Ansible + Jenkins 自动化配置管理基础上，新增 Prometheus alert rules、blackbox-exporter、Alertmanager 和飞书 webhook adapter，建立“指标采集 → 规则判断 → 告警聚合 → 手机通知”的告警闭环，解决服务异常时只能人工查看 Grafana、无法主动收到通知的问题。

---

### Phase 1：部署前基线检查

**① 目标说明**

确认当前集群处于 V1.4 稳定状态，避免在已有服务异常时引入告警系统，导致后续无法区分新问题与历史问题。

**② 节点**：bj-01（100.118.69.78）

**③ 具体命令**

```bash
cd /opt/docker-infra

git status --short

ansible all -i inventory/hosts.yml -m ping

ansible monitor_nodes -i inventory/hosts.yml -m shell -a '
cd /opt/docker/monitor &&
docker compose ps
'

ansible mysqld_exporter_nodes -i inventory/hosts.yml -m shell -a '
docker ps --format "{{.Names}}\t{{.Status}}" | grep -E "node-exporter|mysqld-exporter|mysql"
'
```

**④ 预期输出**

```text
git status --short 无输出，或仅有本次明确计划修改的文档文件。

所有节点 ping 为 SUCCESS。
```

**⑥ 回滚方式**

本 Phase 只读检查，不产生变更。若发现异常，先停止 V1.5 演进，修复 V1.4 基线服务后再继续。

### Phase 2：初始化告警变量与密钥

**① 目标说明**

新增 V1.5 告警系统所需的普通变量和飞书 webhook token。普通变量写入 `inventory/group_vars/all.yml`，敏感信息写入 `vault/secrets.yml` 并保持 ansible-vault 加密。

**② 节点**：bj-01（100.118.69.78）

**③ 具体命令**

在 `inventory/group_vars/all.yml` 中追加或合并以下非敏感变量：

```yaml
# Alerting
alertmanager_port: 9093
alertmanager_host: "{{ bj_01_ip }}"
alertmanager_external_url: "http://{{ bj_01_ip }}:9093"

alertmanager_webhook_adapter_port: 8090
alertmanager_webhook_adapter_image: "bougou/alertmanager-webhook-adapter:v1.1.10"
alertmanager_webhook_adapter_signature: "docker-infra"

blackbox_exporter_port: 9115
blackbox_exporter_image: "prom/blackbox-exporter:v0.25.0"

alert_repeat_interval: 4h
alert_group_wait: 30s
alert_group_interval: 5m

alert_disk_free_percent_critical: 15
alert_inode_free_percent_warning: 15
alert_memory_available_percent_warning: 10
alert_cpu_usage_percent_warning: 85
alert_mysql_realtime_replica_lag_seconds_warning: 60
alert_mysql_delay_replica_lag_seconds_warning: 3900
```

确认 `inventory/hosts.yml` 中新增告警节点分组：

```yaml
    alertmanager_nodes:
      hosts:
        bj-01:
```

使用 ansible-vault 写入飞书 webhook token：

```bash
cd /opt/docker-infra

ansible-vault edit vault/secrets.yml
```

在 vault 中新增以下变量，真实值按飞书机器人页面填写：

```yaml
alerting_feishu_webhook_token: "REPLACE_WITH_FEISHU_BOT_TOKEN"
```

> `alerting_feishu_webhook_token` 建议只保存 webhook URL 最后一段 token，不保存完整 URL。Alertmanager 模板中拼接为 `http://alertmanager-webhook-adapter:8090/webhook/send?channel_type=feishu&token={{ alerting_feishu_webhook_token }}`。当前选用的 `bougou/alertmanager-webhook-adapter` 的 `--signature` 是消息前缀，不是飞书机器人签名校验；因此 V1.5 建议在飞书机器人安全设置中使用“自定义关键词”，关键词包含 `docker-infra`。

**④ 预期输出**

```text
普通变量文件只包含端口、镜像、阈值等非敏感配置。
vault/secrets.yml 仍保持 ansible-vault 加密状态。
```

**⑤ 验证方法**

```bash
cd /opt/docker-infra

ansible-vault view vault/secrets.yml | grep 'alerting_feishu_webhook_token'

grep -R "REPLACE_WITH_FEISHU\|open.feishu.cn" \
  inventory roles playbooks Docs \
  --exclude-dir=.git || true
```

预期：

```text
ansible-vault view 能看到飞书 webhook token 变量。
普通目录中不能出现真实飞书 webhook URL 或 token。
```

**⑥ 回滚方式**

```bash
cd /opt/docker-infra

# 如果尚未提交，手工删除 group_vars 中新增 alerting 变量和 hosts.yml 中 alertmanager_nodes 分组。
ansible-vault edit vault/secrets.yml
# 删除 alerting_feishu_webhook_token 后保存。
```

### Phase 3：新增 Alertmanager role

**① 目标说明**

创建 `roles/alertmanager/`，在 bj-01 部署单实例 Alertmanager 和飞书 webhook adapter。Alertmanager 负责告警分组、去重、恢复通知、静默、抑制和路由；adapter 负责将 Alertmanager webhook 转换为飞书机器人消息。

**② 节点**：bj-01（100.118.69.78）

**③ 具体命令**

创建目录：

```bash
cd /opt/docker-infra

mkdir -p roles/alertmanager/{tasks,templates,handlers}
```

创建 `roles/alertmanager/tasks/main.yml`：

```yaml
---
- name: Ensure alertmanager directories exist
  ansible.builtin.file:
    path: "{{ item }}"
    state: directory
    mode: "0755"
  loop:
    - "{{ docker_root }}/monitor/alertmanager"
    - "{{ docker_root }}/monitor/alertmanager/data"

- name: Render alertmanager compose file
  ansible.builtin.template:
    src: docker-compose.yml.j2
    dest: "{{ docker_root }}/monitor/alertmanager/docker-compose.yml"
    mode: "0644"
  notify: restart alertmanager stack

- name: Render alertmanager config
  ansible.builtin.template:
    src: alertmanager.yml.j2
    dest: "{{ docker_root }}/monitor/alertmanager/alertmanager.yml"
    mode: "0644"
  notify: restart alertmanager stack
```

创建 `roles/alertmanager/handlers/main.yml`：

```yaml
---
- name: restart alertmanager stack
  community.docker.docker_compose_v2:
    project_src: "{{ docker_root }}/monitor/alertmanager"
    state: present
    recreate: auto
```

创建 `roles/alertmanager/templates/docker-compose.yml.j2`：

```yaml
services:
  alertmanager:
    image: prom/alertmanager:v0.27.0
    container_name: alertmanager
    restart: unless-stopped

    ports:
      - "{{ ansible_host }}:{{ alertmanager_port }}:9093"

    volumes:
      - ./alertmanager.yml:/etc/alertmanager/alertmanager.yml:ro
      - ./data:/alertmanager

    command:
      - "--config.file=/etc/alertmanager/alertmanager.yml"
      - "--storage.path=/alertmanager"
      - "--web.external-url={{ alertmanager_external_url }}"
      - "--web.listen-address=0.0.0.0:9093"

    networks:
      - default
      - {{ docker_network_name }}

    logging:
      driver: "json-file"
      options:
        max-size: "10m"
        max-file: "3"

  alertmanager-webhook-adapter:
    image: {{ alertmanager_webhook_adapter_image }}
    container_name: alertmanager-webhook-adapter
    restart: unless-stopped

    command:
      - "--listen-address=:{{ alertmanager_webhook_adapter_port }}"
      - "--signature={{ alertmanager_webhook_adapter_signature }}"
      - "--tmpl-lang=zh"

    networks:
      - default

    logging:
      driver: "json-file"
      options:
        max-size: "10m"
        max-file: "3"

networks:
  {{ docker_network_name }}:
    external: true
```

创建 `roles/alertmanager/templates/alertmanager.yml.j2`：

```yaml
global:
  resolve_timeout: 5m

route:
  receiver: feishu
  group_by: ['alertname', 'cluster', 'job', 'instance', 'severity']
  group_wait: {{ alert_group_wait }}
  group_interval: {{ alert_group_interval }}
  repeat_interval: {{ alert_repeat_interval }}

receivers:
  - name: feishu
    webhook_configs:
      - url: "http://alertmanager-webhook-adapter:{{ alertmanager_webhook_adapter_port }}/webhook/send?channel_type=feishu&token={{ alerting_feishu_webhook_token }}"
        send_resolved: true

inhibit_rules:
  - source_matchers:
      - alertname = "NodeDown"
    target_matchers:
      - alertname =~ "NodeExporterDown|NodeDiskSpaceLow|NodeInodeLow|NodeMemoryLow|NodeCPUHigh"
    equal: ['instance']
```

**④ 预期输出**

```text
roles/alertmanager/ 目录创建完成。
docker-compose.yml.j2 不包含飞书真实 token。
alertmanager.yml.j2 通过 vault 变量引用飞书 token。
```

**⑤ 验证方法**

```bash
cd /opt/docker-infra

ansible-playbook -i inventory/hosts.yml playbooks/setup_monitor.yml \
  --tags alertmanager \
  --limit bj-01 \
  --check \
  --diff \
  --ask-vault-pass
```

预期：

```text
Ansible check mode 能正常渲染 alertmanager role，无 undefined variable。
```

**⑥ 回滚方式**

```bash
cd /opt/docker-infra

rm -rf roles/alertmanager

# 如果已部署到 bj-01，可停止并删除告警发送侧 compose 栈：
ssh root@100.118.69.78 'cd /opt/docker/monitor/alertmanager && docker compose down'
```

创建 `roles/monitor-stack/templates/rules/node.yml.j2`：

```yaml
groups:
  - name: node
    rules:
      - alert: NodeDown
        expr: up{job="node-exporter"} == 0
        for: 2m
        labels:
          severity: critical
        annotations:
          summary: "节点不可达"
          description: "{{ "{{" }} $labels.instance {{ "}}" }} 的 node-exporter 连续 2 分钟不可采集。"

      - alert: NodeDiskSpaceLow
        expr: (node_filesystem_avail_bytes{fstype!~"tmpfs|overlay",mountpoint="/"} / node_filesystem_size_bytes{fstype!~"tmpfs|overlay",mountpoint="/"}) * 100 < {{ alert_disk_free_percent_critical }}
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "根分区可用空间不足"
          description: "{{ "{{" }} $labels.instance {{ "}}" }} 根分区可用空间低于 {{ alert_disk_free_percent_critical }}%。"

      - alert: NodeInodeLow
        expr: (node_filesystem_files_free{fstype!~"tmpfs|overlay",mountpoint="/"} / node_filesystem_files{fstype!~"tmpfs|overlay",mountpoint="/"}) * 100 < {{ alert_inode_free_percent_warning }}
        for: 10m
        labels:
          severity: warning
        annotations:
          summary: "根分区 inode 可用率偏低"
          description: "{{ "{{" }} $labels.instance {{ "}}" }} 根分区 inode 可用率低于 {{ alert_inode_free_percent_warning }}%。"

      - alert: NodeMemoryLow
        expr: (node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes) * 100 < {{ alert_memory_available_percent_warning }}
        for: 10m
        labels:
          severity: warning
        annotations:
          summary: "节点可用内存偏低"
          description: "{{ "{{" }} $labels.instance {{ "}}" }} 可用内存低于 {{ alert_memory_available_percent_warning }}%。"

      - alert: NodeCPUHigh
        expr: (1 - avg by (instance) (rate(node_cpu_seconds_total{mode="idle"}[5m]))) * 100 > {{ alert_cpu_usage_percent_warning }}
        for: 10m
        labels:
          severity: warning
        annotations:
          summary: "节点 CPU 使用率持续偏高"
          description: "{{ "{{" }} $labels.instance {{ "}}" }} CPU 使用率连续 10 分钟超过 {{ alert_cpu_usage_percent_warning }}%。"
```
