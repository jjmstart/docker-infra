# 操作手册：v1.3 → v1.4

## 演进概述

废弃 V1.3 未完成的 "各节点 clone + git pull" 方案，引入 Ansible 统一配置管理：将全集群所有节点的配置文件收录进仓库并重构为 Ansible 标准目录结构，以 bj-01 作为 Ansible 控制节点，配置 ansible-vault 加密敏感信息，最终通过 Jenkins Pipeline 实现 git push → 自动部署的全自动 CI/CD 工作流。

### Phase 2：仓库重构为 Ansible 目录结构

**① 目标说明**

在 gz-03 上将现有仓库重构为高内聚低耦合的 Ansible 目录结构，遵循以下设计原则：
- **按功能角色划分 Inventory 分组**（不按物理机），一台机器可出现在多个功能组
- **Playbook 以功能领域命名**（`setup_database.yml` 而非 `gz-03.yml`），解耦部署逻辑与物理主机
- **用 `{{ ansible_host }}` 替代自定义 `tailscale_ip` 变量**，直接复用 Inventory 已有信息
- **主机特有变量（如 `mysql_server_id`）内联写在 Inventory** 中，无需独立 `host_vars/` 文件
- **所有 role 调用加 `tags`**，支持 `--tags mysql` 等精准部署

**② 节点**：gz-03（100.92.5.116）

**③ 具体命令**

**步骤 2.1：创建 Ansible 目录骨架（无 host_vars/）**

```bash
cd /opt/docker

# 创建目录结构（注意：不需要 host_vars/，主机变量直接写在 inventory 里）
mkdir -p inventory group_vars vault playbooks
mkdir -p roles/mysql-master/{tasks,templates,handlers}
mkdir -p roles/mysql-replica/{tasks,templates,handlers}
mkdir -p roles/redis-master/{tasks,templates,handlers}
mkdir -p roles/redis-replica/{tasks,templates,handlers}
mkdir -p roles/proxysql/{tasks,templates,handlers}
mkdir -p roles/ruoyi/{tasks,templates,handlers}
mkdir -p roles/nginx/{tasks,templates,handlers}
mkdir -p roles/monitor-stack/{tasks,templates,handlers}
mkdir -p roles/node-exporter/{tasks,templates,handlers}
```

**步骤 2.2：创建 inventory/hosts.yml（按功能组划分，内联主机变量）**

设计要点：
- 每台物理机在 `all.hosts` 下定义一次，包含所有主机特有变量
- `children` 下的功能组只引用主机名，不重复定义变量
- 一台机器可同时出现在多个功能组（如 gz-03 同时属于 `db_master_nodes`、`db_proxy_nodes`、`app_nodes`）

```bash
cat > inventory/hosts.yml << 'EOF'
all:
  hosts:
    gz-01:
      ansible_host: 100.117.7.75
      ansible_user: root

    gz-02:
      ansible_host: 100.79.132.125
      ansible_user: root
      mysql_server_id: 2          # mysql-replica role 使用
      ruoyi_instance: "ruoyi-admin-2"  # ruoyi role 使用，区分实例名

    gz-03:
      ansible_host: 100.92.5.116
      ansible_user: root
      mysql_server_id: 1          # mysql-master role 使用
      ruoyi_instance: "ruoyi-admin-1"  # ruoyi role 使用，区分实例名

    bj-01:
      ansible_host: 100.118.69.78
      ansible_user: root
      mysql_server_id: 3          # mysql-replica role 使用
      mysql_source_delay: 3600    # mysql-replica role 使用，延迟复制

  children:
    # ── 入口网关 ──────────────────────────────────
    gateway_nodes:
      hosts:
        gz-01:

    # ── 监控主节点（Prometheus + Grafana）─────────
    monitor_nodes:
      hosts:
        gz-01:

    # ── 若依应用节点（两实例，跨两台机器）──────────
    app_nodes:
      hosts:
        gz-03:
        gz-02:

    # ── MySQL 主库 ────────────────────────────────
    db_master_nodes:
      hosts:
        gz-03:

    # ── MySQL 从库（实时 + 延迟，共用同一 role）──
    db_replica_nodes:
      hosts:
        gz-02:           # server-id=2，无延迟，在线读
        bj-01:           # server-id=3，延迟 3600s，纯灾备

    # ── ProxySQL 读写分离代理 ─────────────────────
    db_proxy_nodes:
      hosts:
        gz-03:

    # ── Redis 主节点 ──────────────────────────────
    redis_master_nodes:
      hosts:
        gz-03:

    # ── Redis 从节点 ──────────────────────────────
    redis_replica_nodes:
      hosts:
        gz-02:
        bj-01:

    # ── Node Exporter（全节点）───────────────────
    exporter_nodes:
      hosts:
        gz-01:
        gz-02:
        gz-03:
        bj-01:

    # ── mysqld_exporter（有 MySQL 的节点）────────
    mysqld_exporter_nodes:
      hosts:
        gz-02:
        gz-03:
        bj-01:

    # ── 运维工具节点（Jenkins、Diary）────────────
    ops_nodes:
      hosts:
        bj-01:
EOF
```

**步骤 2.5：将 docker-compose.yml 迁移为 Jinja2 模板**

**关键原则：用 `{{ ansible_host }}` 替代所有硬编码 IP，不引入任何自定义 IP 变量。**

以 `mysql-master` 为例——完整展示模板写法：

```bash
# 保留 git history，用 git mv 而非 cp
git mv backend/mysql/docker-compose.yml roles/mysql-master/templates/docker-compose.yml.j2
```

编辑 `roles/mysql-master/templates/docker-compose.yml.j2`，关键变更点：

```yaml
# roles/mysql-master/templates/docker-compose.yml.j2
services:
  mysql:
    image: {{ mysql_image }}          # ← 来自 group_vars/all.yml
    container_name: mysql
    restart: always
    environment:
      - MYSQL_ROOT_PASSWORD={{ mysql_root_password }}  # ← 来自 vault/secrets.yml
      - TZ=Asia/Shanghai
    ports:
      - "127.0.0.1:3306:3306"
      - "{{ ansible_host }}:3306:3306"   # ← ansible_host = inventory 中 ansible_host 的值
                                          #   gz-03 上渲染为 100.92.5.116:3306:3306
    volumes:
      - "{{ service_root }}/backend/mysql/data:/var/lib/mysql"
    command:
      - --server-id={{ mysql_server_id }}   # ← 来自 inventory 主机变量（gz-03=1）
      - --gtid_mode=ON
      # ...其他参数保持原样...
    networks:
      - {{ docker_network }}

networks:
  {{ docker_network }}:
    external: true
```

> **为什么用 `{{ ansible_host }}` 而不定义 `tailscale_ip` 变量？**
> inventory 中的 `ansible_host` 已经就是 Tailscale IP，直接引用即可。多定义一个同值变量只会增加维护负担，且可能造成两者不一致的隐患。

对 `mysql-replica` 模板，处理延迟复制的可选参数：

```yaml
# roles/mysql-replica/templates/docker-compose.yml.j2
    command:
      - --server-id={{ mysql_server_id }}
      - --gtid_mode=ON
      # bj-01 有 mysql_source_delay=3600，gz-02 无此变量，用 default(0) 安全处理
      {% if mysql_source_delay | default(0) > 0 %}
      - --replica_delay={{ mysql_source_delay }}
      {% endif %}
```

对 `ruoyi` 模板，用 `ruoyi_instance` 变量区分两个实例的容器名：

```yaml
# roles/ruoyi/templates/docker-compose.yml.j2
services:
  {{ ruoyi_instance }}:           # gz-03 渲染为 ruoyi-admin-1，gz-02 渲染为 ruoyi-admin-2
    image: ruoyi-jenkins-docker:latest
    container_name: {{ ruoyi_instance }}
    ports:
      - "{{ ansible_host }}:8080:8080"
```

其余 role 的模板（redis-master、redis-replica、proxysql、nginx、monitor-stack、node-exporter）按同样方式处理：将所有硬编码 IP 替换为 `{{ ansible_host }}`，密码替换为对应 vault 变量。

**步骤 2.6：创建各 role 的 tasks/main.yml 和 handlers/main.yml**

以 `mysql-master` 为完整示例（展示幂等性写法和 handler 优雅重启）：

```bash
cat > roles/mysql-master/tasks/main.yml << 'EOF'
- name: 确保 MySQL 数据目录存在
  file:
    path: "{{ service_root }}/backend/mysql/data"
    state: directory
    mode: '0755'

- name: 部署 docker-compose.yml
  template:
    src: docker-compose.yml.j2
    dest: "{{ service_root }}/backend/mysql/docker-compose.yml"
    mode: '0644'
  notify: restart mysql   # 文件有变更才触发 handler，无变更则幂等跳过

- name: 部署 .env 文件
  template:
    src: env.j2
    dest: "{{ service_root }}/backend/mysql/.env"
    mode: '0600'
  notify: restart mysql
EOF

cat > roles/mysql-master/handlers/main.yml << 'EOF'
# MySQL 是有状态服务，handler 只在配置真正变更时触发
# 正常运营中 compose 文件极少变化，不会频繁重启
- name: restart mysql
  community.docker.docker_compose_v2:
    project_src: "{{ service_root }}/backend/mysql"
    state: present
    recreate: auto    # 只有 compose 文件或镜像有变化才重建容器
EOF
```

`nginx` role 的 handler 示例（用 reload 代替 restart，零停机）：

```bash
cat > roles/nginx/handlers/main.yml << 'EOF'
- name: reload nginx
  community.docker.docker_container_exec:
    container: nginx
    command: nginx -s reload
  # 配置文件变更时只 reload，不重启容器，零停机
EOF
```

`monitor-stack` 的 Prometheus handler（用 HTTP reload，零停机）：

```bash
cat > roles/monitor-stack/handlers/main.yml << 'EOF'
- name: reload prometheus
  uri:
    url: "http://localhost:9090/-/reload"
    method: POST
    status_code: 200
  # prometheus.yml 变更时热重载，不重启容器

- name: restart grafana
  community.docker.docker_compose_v2:
    project_src: "{{ service_root }}/monitor"
    state: present
    recreate: auto
EOF
```

`ruoyi` role 的 handler 示例（无状态服务，双实例滚动重启）：

```bash
cat > roles/ruoyi/handlers/main.yml << 'EOF'
- name: restart ruoyi
  community.docker.docker_compose_v2:
    project_src: "{{ service_root }}/backend/ruoyi"
    state: present
    recreate: always   # 无状态服务，直接重建（两实例不会同时重启，Nginx 保证高可用）
EOF
```

**步骤 2.7：创建以功能领域命名的 Playbook（核心改造点）**

Playbook 按"做什么"命名，而非"在哪台机器上"命名。`hosts:` 绑定功能组名。

```bash
# ── 数据库（MySQL 主从 + ProxySQL）─────────────────
cat > playbooks/setup_database.yml << 'EOF'
---
- hosts: db_master_nodes
  vars_files:
    - ../vault/secrets.yml
  roles:
    - role: mysql-master
      tags: ['mysql', 'mysql-master', 'db']

- hosts: db_replica_nodes
  vars_files:
    - ../vault/secrets.yml
  roles:
    - role: mysql-replica
      tags: ['mysql', 'mysql-replica', 'db']

- hosts: db_proxy_nodes
  vars_files:
    - ../vault/secrets.yml
  roles:
    - role: proxysql
      tags: ['proxysql', 'db']
EOF

# ── Redis（主从哨兵）──────────────────────────────
cat > playbooks/setup_redis.yml << 'EOF'
---
- hosts: redis_master_nodes
  vars_files:
    - ../vault/secrets.yml
  roles:
    - role: redis-master
      tags: ['redis', 'redis-master']

- hosts: redis_replica_nodes
  vars_files:
    - ../vault/secrets.yml
  roles:
    - role: redis-replica
      tags: ['redis', 'redis-replica']
EOF

# ── 应用层（若依后端）────────────────────────────
cat > playbooks/setup_app.yml << 'EOF'
---
- hosts: app_nodes
  vars_files:
    - ../vault/secrets.yml
  roles:
    - role: ruoyi
      tags: ['ruoyi', 'app']
EOF

# ── 入口网关（Nginx）─────────────────────────────
cat > playbooks/setup_gateway.yml << 'EOF'
---
- hosts: gateway_nodes
  vars_files:
    - ../vault/secrets.yml
  roles:
    - role: nginx

EOF

# ── 监控（Prometheus + Grafana）──────────────────
cat > playbooks/setup_monitor.yml << 'EOF'
---
- hosts: monitor_nodes
  vars_files:
    - ../vault/secrets.yml
  roles:
    - role: monitor-stack

# ── 指标采集（全节点 node-exporter + mysqld_exporter）
```

步骤 4.4（推荐）：通过 gz-01 Nginx 反向代理接收 GitHub Webhook

如果 bj-01 Jenkins 不对外直接暴露（或不想把 Jenkins 8080 端口开到公网），可利用 gz-01（有公网 IP、已运行 Nginx）做 HTTP 层反向代理，将 GitHub 的 Webhook 请求转发给内网 Jenkins，只暴露 `/github-webhook/` 这一个路径。

```nginx
# gz-01 上：/etc/nginx/conf.d/jenkins-webhook.conf
server {
    listen 80;
    server_name <gz-01公网IP或域名>;    # 与 GitHub Webhook Payload URL 保持一致

    location /github-webhook/ {

        proxy_set_header   Host              $host;
        proxy_set_header   X-Real-IP         $remote_addr;
        proxy_set_header   X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_read_timeout 10s;
    }

    # 拒绝其他所有路径（防止 Jenkins 管理界面被意外访问）
    location / {
        return 403;
    }
}
```

**① 目标说明**

在 bj-01 Jenkins 中创建 Pipeline Job，将 vault 密码存入 Jenkins Credentials，编写 Jenkinsfile，实现 GitHub push → Jenkins 自动执行 `ansible-playbook site.yml` 的完整 CI/CD 链路。

**② 节点**：bj-01 Jenkins（100.118.69.78:8080）

**③ 具体命令**

步骤 8.1：确认 Jenkins 容器内有 Ansible

```bash
# 在 bj-01 宿主机执行
docker exec jenkins ansible --version 2>/dev/null
```

如果 Ansible 不在 Jenkins 容器内（返回报错），需要重建 Jenkins 容器使其包含 Ansible：

```bash
# 创建自定义 Dockerfile（在 bj-01 宿主机）
mkdir -p /opt/docker/jenkins-build
cat > /opt/docker/jenkins-build/Dockerfile << 'EOF'
FROM jenkins/jenkins:lts
USER root
RUN apt-get update && apt-get install -y ansible python3-pip && rm -rf /var/lib/apt/lists/*
# 切换到 jenkins 用户安装 collection，确保权限正确
USER jenkins
RUN ansible-galaxy collection install community.docker
EOF

# 构建镜像
docker build -t jenkins-ansible:latest /opt/docker/jenkins-build/

# 更新 Jenkins 容器使用新镜像（先确认现有 compose 文件位置）
# 修改 Jenkins docker-compose.yml 中的 image 为 jenkins-ansible:latest
# 然后：
cd /path/to/jenkins/compose
docker compose up -d --force-recreate
```

步骤 8.1.5：确认 SSH Agent 插件已安装（Jenkinsfile 中 `sshagent` 块的前置依赖）

```
Jenkins → Manage Jenkins → Plugins → Installed plugins
  搜索：SSH Agent
  如未安装：切换到 Available plugins 标签 → 搜索 "SSH Agent" → Install without restart
  等待安装完成后重启 Jenkins（或在插件页面勾选 "Restart Jenkins when installation is complete"）
```

步骤 8.2：将 vault 密码存入 Jenkins Credentials

```
Jenkins → Manage Jenkins → Credentials → System → Global credentials → Add Credentials
  Kind: Secret text
  Scope: Global
  Secret: （输入你的 vault 密码）
  ID: ansible-vault-password
  Description: Ansible Vault Password
→ Create
```

步骤 8.3：将 bj-01 SSH 私钥存入 Jenkins（供 Ansible 连接其他节点使用）

```
Jenkins → Manage Jenkins → Credentials → Add Credentials
  Kind: SSH Username with private key
  Username: root
  Private Key: Enter directly → 粘贴 ~/.ssh/id_ed25519 内容
  ID: ansible-ssh-key
  Description: Ansible SSH Private Key
→ Create
```

步骤 8.4：在仓库根目录创建 Jenkinsfile（在 bj-01 的 /opt/docker 下）

```bash
cat > /opt/docker/Jenkinsfile << 'EOF'
pipeline {
    agent any

    environment {
        // 解决首次连接卡死问题：禁用 SSH 主机指纹交互确认
        ANSIBLE_HOST_KEY_CHECKING = "False"
        VAULT_PASS = credentials('ansible-vault-password')
    }

    stages {
        stage('Checkout') {
            steps {
                checkout scm
            }
        }

        stage('Validate') {
            steps {
                // sshagent 注入私钥，供 Ansible SSH 到各节点（依赖 SSH Agent Plugin，见步骤 8.1.5）
                sshagent(credentials: ['ansible-ssh-key']) {
                    sh '''
                        echo "${VAULT_PASS}" > /tmp/.vault_pass
                        ansible-playbook playbooks/site.yml \
                            -i inventory/hosts.yml \
                            --vault-password-file /tmp/.vault_pass \
                            --syntax-check
                        rm -f /tmp/.vault_pass
                    '''
                }
            }
        }

        stage('Deploy') {
            steps {
                sshagent(credentials: ['ansible-ssh-key']) {
                    sh '''
                        echo "${VAULT_PASS}" > /tmp/.vault_pass
                        ansible-playbook playbooks/site.yml \
                            -i inventory/hosts.yml \
                            --vault-password-file /tmp/.vault_pass \
                            --diff
                        rm -f /tmp/.vault_pass
                    '''
                }
            }
        }
    }

    post {
        always {
            sh 'rm -f /tmp/.vault_pass'
        }
        success {
            echo 'Ansible 部署成功，全节点配置已同步'
        }
        failure {
            echo '部署失败，请检查 Ansible 输出日志'
        }
    }
}
EOF

git add Jenkinsfile
git commit -m "feat: add Jenkinsfile for CI/CD pipeline

- Validate stage: ansible-playbook --syntax-check
- Deploy stage: ansible-playbook site.yml --diff
- Vault password injected from Jenkins credentials
- sshagent injects SSH private key for Ansible SSH connections
- ANSIBLE_HOST_KEY_CHECKING=False prevents interactive hang on first connect"
git push
```

步骤 8.5：在 Jenkins 创建 Pipeline Job

```
Jenkins → New Item
  名称：ansible-deploy
  类型：Pipeline
→ OK

配置页面：
  Build Triggers（二选一，强烈推荐 Webhook）：
    ☑ GitHub hook trigger for GITScm polling（Webhook 方式，实时触发，推荐）
      → 前提：GitHub 能访问 Jenkins 入口（直接公网 或 Phase 4 步骤 4.4 的 gz-01 Nginx 代理）
    或 ☑ Poll SCM，Schedule: H/2 * * * *（轮询降级方案，有约 2 分钟延迟，消耗 GitHub API 额度）

  Pipeline：
    Definition: Pipeline script from SCM
    SCM: Git
    Repository URL: git@github.com:your-username/docker-infra.git
    Credentials: 选择 ansible-ssh-key
    Branch: */main
    Script Path: Jenkinsfile

→ Save
```

**④ 预期输出**

手动点击 "Build Now" 触发第一次构建，Jenkins Console Output 应显示：

```
[Pipeline] stage (Validate)
...
playbook: playbooks/site.yml  ← syntax-check 通过

[Pipeline] stage (Deploy)
PLAY RECAP **************
gz-01 : ok=X  changed=0  unreachable=0  failed=0

```

**⑤ 验证方法**

```
Jenkins → ansible-deploy → 查看最近构建 → Console Output
确认最后一行：Finished: SUCCESS
```

**⑥ 回滚方式**

如 Pipeline 构建失败导致服务异常，可在各受影响节点手动恢复：

```bash
ssh root@<node_ip> "cd /opt/docker/backend/<service> && docker compose up -d"
```

### Phase 9：端到端验证

**① 目标说明**

对一处非关键配置做小幅修改（如 Grafana 端口注释），git push，验证整个自动化链路：push → Webhook/Polling 触发 Jenkins → Ansible 执行 → 配置到达目标节点。

**② 节点**：bj-01（git push）→ 全自动流程

**③ 具体命令**

```bash
cd /opt/docker

# 对非关键配置做一处无害修改（如在 group_vars/all.yml 添加注释）
echo "# e2e test $(date)" >> group_vars/all.yml

git add group_vars/all.yml
git commit -m "test: e2e pipeline verification"
git push
```

等待 Jenkins 自动触发（Webhook 约5-30秒，Polling 约2分钟），然后观察：

```
Jenkins → ansible-deploy → 查看最新构建 → Console Output
```

**④ 预期输出**

Jenkins 自动触发新构建，Console Output 末尾：

```
PLAY RECAP ****
gz-01 : ok=X  changed=0  unreachable=0  failed=0
...
Finished: SUCCESS
```

**⑤ 验证方法**

```bash
# 在 bj-01 查看 Jenkins 构建记录
# 确认 git push 后 Jenkins 自动启动了新构建（无需手动点击 Build Now）
# 确认构建成功且 changed=0（注释变更不影响远端文件）
```

打 v1.4 里程碑 tag：

## 验收标准

| 指标 | 目标值 |
|------|--------|
| Ansible 连通性 | `ansible all -m ping` 四节点全部返回 `pong` |
| 仓库结构 | inventory/ roles/ playbooks/ vault/ 目录均存在；playbooks/ 下无 `gz-*.yml` 文件；无 `host_vars/` 目录 |
| vault/secrets.yml | GitHub 上为密文（`$ANSIBLE_VAULT;1.1;AES256` 开头），无任何明文密码 |
| 全节点部署 | `ansible-playbook playbooks/site.yml` 运行结果 `failed=0` |
| 服务状态 | 四节点所有 Docker 容器均处于 `Up` 状态，业务接口可访问 |
| Jenkins Pipeline | ansible-deploy Job 手动触发构建 `Finished: SUCCESS` |
| 端到端触发 | git push 后 Jenkins 在 2 分钟内自动触发新构建并成功 |
| arch-v1.4 tag | `git tag -l` 包含 `arch-v1.4`，已推送至 GitHub |

## 日常使用工作流（演进完成后）

```bash
# 场景 A：修改 ProxySQL 路由（只影响 db_proxy_nodes）
cd /opt/docker
vi roles/proxysql/templates/proxysql.cnf.j2
git add roles/proxysql/templates/proxysql.cnf.j2
git commit -m "fix: adjust proxysql read hostgroup weight"
git push
# → Jenkins 触发 → ansible-playbook setup_database.yml --tags proxysql
#   → 精准替换 gz-03 的 proxysql.cnf → reload proxysql 容器

# 场景 B：只更新 Nginx 配置（零停机）
vi roles/nginx/templates/nginx.conf.j2
git commit -am "fix: update upstream ruoyi backends"
git push
# → ansible-playbook setup_gateway.yml --tags nginx
#   → nginx -s reload（不重启容器）

# 场景 C：修改密码（vault 操作）
ansible-vault edit vault/secrets.yml --vault-password-file ~/.vault_pass
git add vault/secrets.yml
git commit -m "chore: rotate mysql root password"
git push
# → 全节点 .env 自动重渲染，相关容器按 handler 优雅重启

# 场景 D：精准调试（不走 Jenkins，手动在 bj-01 执行）
ansible-playbook playbooks/setup_database.yml \
  -i inventory/hosts.yml \
  --vault-password-file ~/.vault_pass \
  --tags mysql-replica \
  --limit gz-02 \
  --check --diff
# 只在 gz-02 上空跑 mysql-replica 相关任务，确认变更内容后去掉 --check 执行

# ── V1.5 进阶备忘：通过 commit message 控制 --tags（当前不实现，记录思路）───────────────
# 目前 Jenkins 每次全量执行 site.yml，因 Ansible 幂等性保证无关服务不会被重复变更，完全可接受。
# 未来 V1.5 可在 Jenkinsfile 中解析 git commit message：
#   若 message 包含 [tags: nginx]，则自动在 ansible-playbook 命令后追加 --tags nginx
#   实现"提交即精准触发"，进一步减少每次流水线的执行时间。
# 示例 commit message：fix: update nginx upstream backends [tags: nginx]
```
