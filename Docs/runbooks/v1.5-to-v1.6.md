# 操作手册：v1.5 → v1.6

## 演进概述

在 V1.5 的 Ansible + Jenkins 自动化配置管理与告警体系基础上，新增私有 Docker Registry、CI 镜像构建推送 Pipeline、参数化 CD 部署、Smoke Test 和飞书部署通知，将现有「基础设施配置下发流水线」升级为「代码提交到服务上线的完整应用交付流水线」，解决镜像无中央存储、无版本追溯、CI/CD 完全割裂、部署后无服务验证的四大断裂点。

---

### Phase 2：初始化 Registry 变量与密钥

**① 目标说明**

在 `inventory/group_vars/all.yml` 中新增私有 Registry 相关的非敏感变量（地址、端口、镜像名、默认 tag）；在 `inventory/hosts.yml` 中新增 `registry_nodes` 分组，并为 gz-02/gz-03 添加 Docker daemon 主机变量；使用 ansible-vault 写入 Registry Basic Auth 用户名和密码。

**② 节点**：bj-01（100.118.69.78）

**③ 具体命令**

在 `inventory/group_vars/all.yml` 末尾追加：

```yaml
# ── 私有镜像仓库 ──────────────────────────────────────────────────────────────
registry_host: "100.118.69.78:5000"
registry_port: 5000
registry_image_tag: "registry:2"

# ── 应用镜像版本 ──────────────────────────────────────────────────────────────
ruoyi_image_name: "ruoyi"
ruoyi_image_tag: "latest"          # 默认值，Jenkins CD Job 传参时覆盖
```

在 `inventory/hosts.yml` 的 `children` 下新增 `registry_nodes` 分组（与 `alertmanager_nodes` 并列）：

```yaml
    registry_nodes:
      hosts:
        bj-01:
```

同时在 `inventory/hosts.yml` 的 `all.hosts` 下为 gz-02 和 gz-03 添加主机专属变量（保留 Phase 1 确认的现有配置）。主机变量必须写在 `all.hosts` 对应主机下，`children` 分组只引用主机名，不重复定义 `ansible_host`：

```yaml
    gz-02:
      ansible_host: 100.79.132.125
      ansible_user: root
      docker_registry_mirrors:
        - "https://mirror.ccs.tencentyun.com"

    gz-03:
      ansible_host: 100.92.5.116
      ansible_user: root
      docker_registry_mirrors:
        - "https://mirror.baidubce.com"
        - "https://docker.m.daocloud.io"
      docker_dns:
        - "223.5.5.5"
        - "114.114.114.114"
        - "8.8.8.8"
```

> 如果 `inventory/hosts.yml` 中 gz-02/gz-03 已有 `ansible_host`、`ansible_user`、`ruoyi_instance` 等字段，只需在对应主机下追加 `docker_registry_mirrors` 和 `docker_dns` 字段，不要移动主机到新的业务分组中。

使用 ansible-vault 写入 Registry 认证凭据：


在 vault 中新增以下变量（真实值自行填写，禁止在文档中明文记录）：

```yaml

registry_auth_password: "<填写 Registry 密码>"
```


### Phase 3：新增 roles/registry/ 并部署 Registry

**① 目标说明**

创建 `roles/registry/` Ansible role，在 bj-01 部署 Docker Registry v2 容器，配置 Basic Auth 认证（htpasswd）、数据目录挂载和 GC 所需的配置文件；新增 `playbooks/setup_registry.yml`；首次执行部署并验证 Registry 认证正常。

**② 节点**：bj-01（100.118.69.78）

**③ 具体命令**

创建目录：

```bash
cd /opt/docker-infra

mkdir -p roles/registry/{tasks,templates,handlers}
```

创建 `roles/registry/tasks/main.yml`：

```yaml
---
- name: Ensure registry directories exist
  ansible.builtin.file:
    path: "{{ item }}"
    state: directory
    mode: "0755"
  loop:
    - /opt/docker/registry
    - /opt/docker/registry/data
    - /opt/docker/registry/auth

- name: Generate registry htpasswd file
  community.general.htpasswd:
    path: /opt/docker/registry/auth/htpasswd
    name: "{{ registry_auth_username }}"
    password: "{{ registry_auth_password }}"
    crypt_scheme: bcrypt
  notify: restart registry

- name: Render registry config file
  ansible.builtin.template:
    src: config.yml.j2
    dest: /opt/docker/registry/config.yml
    mode: "0644"
  notify: restart registry

- name: Render registry compose file
  ansible.builtin.template:
    src: docker-compose.yml.j2
    dest: /opt/docker/registry/docker-compose.yml
    mode: "0644"
  notify: restart registry

- name: Apply pending registry changes
  ansible.builtin.meta: flush_handlers

- name: Ensure registry container is running
  community.docker.docker_compose_v2:
    project_src: /opt/docker/registry
    state: present
```

创建 `roles/registry/handlers/main.yml`：

```yaml
---
- name: restart registry
  community.docker.docker_compose_v2:
    project_src: /opt/docker/registry
    state: present
    recreate: always
```

创建 `roles/registry/templates/docker-compose.yml.j2`：

```yaml
services:
  registry:
    image: {{ registry_image_tag }}
    container_name: registry
    restart: unless-stopped

    ports:
      - "{{ ansible_host }}:{{ registry_port }}:5000"

    volumes:
      - /opt/docker/registry/data:/var/lib/registry
      - /opt/docker/registry/auth:/auth:ro
      - /opt/docker/registry/config.yml:/etc/docker/registry/config.yml:ro

    environment:
      REGISTRY_AUTH: htpasswd
      REGISTRY_AUTH_HTPASSWD_REALM: Registry Realm
      REGISTRY_AUTH_HTPASSWD_PATH: /auth/htpasswd

    networks:
      - {{ docker_network_name }}

    logging:
      driver: "json-file"
      options:
        max-size: "10m"
        max-file: "3"

networks:
  {{ docker_network_name }}:
    external: true
```

创建 `roles/registry/templates/config.yml.j2`（Registry 内部配置，GC 时使用）：

```yaml
version: 0.1
log:
  accesslog:
    disabled: false
  level: info
storage:
  filesystem:
    rootdirectory: /var/lib/registry
  delete:
    enabled: true
http:
  addr: :5000
auth:
  htpasswd:
    realm: Registry Realm
    path: /auth/htpasswd
```

创建 `playbooks/setup_registry.yml`：

```yaml
---
- name: Deploy Docker Registry
  hosts: registry_nodes
  become: true
  vars_files:
    - ../vault/secrets.yml
  roles:
    - role: registry
      tags: [registry]
```

同时将 Registry 纳入全量部署入口，在 `playbooks/site.yml` 中把 `setup_registry.yml` 放在 `setup_infra.yml` 之后：

```yaml
---
- import_playbook: setup_infra.yml
- import_playbook: setup_registry.yml
- import_playbook: setup_database.yml
- import_playbook: setup_redis.yml
- import_playbook: setup_app.yml
- import_playbook: setup_gateway.yml
- import_playbook: setup_monitor.yml
- import_playbook: setup_ops.yml
```

执行依赖检查：

```bash
cd /opt/docker-infra

ansible-galaxy collection list community.docker
ansible-galaxy collection list community.general
```

如缺少集合，先安装：

```bash
ansible-galaxy collection install community.docker community.general
```

> `community.general.htpasswd` 依赖目标 Python 环境的密码哈希能力；如执行时报 `passlib` 相关错误，需先在 bj-01 安装对应 Python 依赖后再继续。

执行语法检查和 check mode：

```bash

  --syntax-check \
  --ask-vault-pass

```

确认 diff 符合预期后执行正式部署：



### Phase 4：更新 gz-02/gz-03 Docker daemon 配置

**① 目标说明**

在 gz-02 和 gz-03 的 `/etc/docker/daemon.json` 中追加 `insecure-registries: ["100.118.69.78:5000"]`，保留各节点现有的 `registry-mirrors` 和 `dns` 配置，通过现有 `roles/docker-daemon` 渲染合并后的完整文件，并重启 Docker daemon 使配置生效。

**② 节点**：bj-01（100.118.69.78，下发到 gz-02/gz-03）

**③ 具体命令**

当前仓库已经存在专门管理 Docker daemon 的 `roles/docker-daemon/`，不要在 `roles/ruoyi/` 中重复管理 `/etc/docker/daemon.json`。更新 `roles/docker-daemon/templates/daemon.json.j2` 为支持主机变量的模板：

```jinja2
{
  "registry-mirrors": {{ docker_registry_mirrors | default([
    "https://docker.m.daocloud.io",
    "https://dockerhub.icu",
    "https://docker.1panel.live"
  ]) | to_json }},
  "live-restore": true,
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "10m",
    "max-file": "3"
  }{% if inventory_hostname in groups.get('app_nodes', []) %},
  "insecure-registries": ["{{ registry_host }}"]{% endif %}{% if docker_dns is defined %},
  "dns": {{ docker_dns | to_json }}{% endif %}
}
```

> `roles/docker-daemon/tasks/main.yml` 和 `roles/docker-daemon/handlers/main.yml` 已存在，无需新增 task/handler；handler 当前带有 `when: not ansible_check_mode`，适合 check mode 场景。

### Phase 8：新建 Registry GC Pipeline Job

**① 目标说明**

新建 `Jenkinsfile.registry-gc`，实现定期清理 Registry 中超出数量限制（保留最新 10 个 tag）的旧版本镜像，真正释放磁盘空间；在 Jenkins UI 中配置对应的 Pipeline Job，设置每周日凌晨 2:00 自动触发。

**② 节点**：bj-01（100.118.69.78）

**③ 具体命令**

在 docker-infra 仓库根目录创建 `Jenkinsfile.registry-gc`：

```groovy
pipeline {
  agent any
  options {
    timestamps()
  }
  triggers {
    // 每周日凌晨 2:00 自动触发
    cron('0 2 * * 0')
  }
  environment {
    REGISTRY_HOST = '100.118.69.78:5000'
    IMAGE_NAME    = 'ruoyi'
    KEEP_TAGS     = '10'
  }
  stages {
    stage('Query Tags') {
      steps {
        withCredentials([
          usernamePassword(
            credentialsId: 'registry-credentials',
            usernameVariable: 'REGISTRY_USER',
            passwordVariable: 'REGISTRY_PASS'
          )
        ]) {
          sh '''
            set -eu
            echo "查询 Registry 中全部 tag..."
            curl -sf -u "${REGISTRY_USER}:${REGISTRY_PASS}" \
              http://${REGISTRY_HOST}/v2/${IMAGE_NAME}/tags/list \
              | python3 -c "
import json, sys
data = json.load(sys.stdin)
tags = data.get('tags') or []
# 过滤出 BUILD_NUMBER-SHA 格式的版本 tag，按 BUILD_NUMBER 排序
versioned = sorted(
    [t for t in tags if '-' in t and t.split('-')[0].isdigit()],
    key=lambda t: int(t.split('-')[0])
)
all_tags = versioned + [t for t in tags if t not in versioned]
print('所有 tag（共 %d 个）：' % len(all_tags), all_tags)
to_delete = versioned[:-int('${KEEP_TAGS}')]
print('待删除 tag（共 %d 个）：' % len(to_delete), to_delete)
with open('tags_to_delete.txt', 'w') as f:
    f.write('\\n'.join(to_delete))
"
            echo "tags_to_delete.txt 内容："
            cat tags_to_delete.txt || echo "(无待删除 tag)"
          '''
        }
      }
    }
    stage('Delete Old Manifests') {
      steps {
        withCredentials([
          usernamePassword(
            credentialsId: 'registry-credentials',
            usernameVariable: 'REGISTRY_USER',
            passwordVariable: 'REGISTRY_PASS'
          )
        ]) {
          sh '''
            set -eu
            if [ ! -s tags_to_delete.txt ]; then
              echo "没有需要删除的 tag，跳过。"
              exit 0
            fi
            while IFS= read -r tag; do
              [ -z "$tag" ] && continue
              echo "正在软删除 tag: ${tag}"
              digest=$(curl -sf -u "${REGISTRY_USER}:${REGISTRY_PASS}" \
                -H "Accept: application/vnd.docker.distribution.manifest.v2+json" \
                -I http://${REGISTRY_HOST}/v2/${IMAGE_NAME}/manifests/${tag} \
                | grep -i "Docker-Content-Digest:" \
                | tr -d "\\r" \
                | awk "{print \\$2}")
              if [ -z "$digest" ]; then
                echo "  获取 digest 失败，跳过 tag: ${tag}"
                continue
              fi
              echo "  digest: ${digest}"
              curl -sf -u "${REGISTRY_USER}:${REGISTRY_PASS}" \
                -X DELETE http://${REGISTRY_HOST}/v2/${IMAGE_NAME}/manifests/${digest}
              echo "  已软删除 tag: ${tag}"
            done < tags_to_delete.txt
          '''
        }
      }
    }
    stage('Garbage Collect') {
      steps {
        sh '''
          set -eu
          echo "执行 Registry garbage-collect，释放无引用 blob..."
          docker exec registry \
            registry garbage-collect /etc/docker/registry/config.yml \
            --delete-untagged
          echo "GC 完成。"
          echo "当前 Registry 数据目录磁盘使用："
          du -sh /opt/docker/registry/data/
        '''
      }
    }
  }
  post {
    success {
      withCredentials([string(credentialsId: 'feishu-webhook-url-jenkins-notify', variable: 'FEISHU_URL')]) {
        sh '''
          set -eu
          deleted_count=$(wc -l < tags_to_delete.txt 2>/dev/null || echo 0)
          disk_usage=$(du -sh /opt/docker/registry/data/ 2>/dev/null | awk "{print \\$1}" || echo "unknown")
          curl -sf -X POST "$FEISHU_URL" \
            -H "Content-Type: application/json" \
            -d "{
              \\"msg_type\\": \\"text\\",
              \\"content\\": {
                \\"text\\": \\"✅ [docker-infra] Registry GC 完成：清理 ${deleted_count} 个旧 tag，当前数据目录占用 ${disk_usage}。\\"
              }
            }"
        '''
      }
    }
    failure {
      withCredentials([string(credentialsId: 'feishu-webhook-url-jenkins-notify', variable: 'FEISHU_URL')]) {
        sh '''
          set -eu
          curl -sf -X POST "$FEISHU_URL" \
            -H "Content-Type: application/json" \
            -d "{
              \\"msg_type\\": \\"text\\",
              \\"content\\": {
                \\"text\\": \\"❌ [docker-infra] Registry GC 失败：Job #${BUILD_NUMBER}，请查看 Jenkins。\\"
              }
            }"
        '''
      }
    }
  }
}
```

**在 Jenkins UI 中创建 GC Job**

1. Jenkins → New Item → 填写名称 `registry-gc` → 选择 Pipeline
2. Pipeline → Definition: Pipeline script from SCM
3. SCM: Git，Repository URL: docker-infra GitHub 仓库地址
4. Script Path: `Jenkinsfile.registry-gc`
5. Build Triggers：勾选「Build periodically」，Schedule: `0 2 * * 0`（Jenkinsfile 中的 cron trigger 也会生效，双保险）
6. 保存

```text
手工触发一次 Job，Query Tags 阶段输出当前 tag 列表；如 tag 数量 ≤ 10，Delete 阶段输出「没有需要删除的 tag」，GC 阶段正常执行。
```

## 验收标准

| 指标 | 目标值 |
|------|--------|
| Registry 部署 | `ansible-playbook setup_registry.yml` 重复执行 `changed=0 failed=0` |
| Registry 认证 | `curl -u user:pass http://100.118.69.78:5000/v2/` 返回 `{}` HTTP 200 |
| Registry 无认证 | 无 Bearer token 请求返回 HTTP 401 |
| Registry 数据持久化 | Registry 容器重启后 `/v2/ruoyi/tags/list` 仍能返回历史 tag |
| CI Pipeline | ruoyi main 分支 push 自动触发构建；Registry 出现 `{BUILD_NUMBER}-{GIT_SHA}` 格式新 tag |
| CI 飞书通知 | 构建成功时飞书收到「✅ 新版本就绪」通知，含正确 tag；失败时收到 ❌ 通知 |
| CI 分支策略 | feature 分支 push 不触发 Docker Build 和 Push |
| CD 参数化 | docker-infra Job 以 `IMAGE_TAG=xxx` 参数触发，Ansible Check 阶段 console 中确认 `ruoyi_image_tag=xxx` |
| CD 镜像更新 | 部署后 gz-02/gz-03 上 `docker inspect` 确认容器 Image 字段为 `100.118.69.78:5000/ruoyi:{tag}` |
| Smoke Test | 部署后等待 30s，Prometheus `min(probe_success{service="ruoyi"})` 返回 `1`，Jenkins stage 绿色 |
| CD 飞书通知 | 部署成功飞书收到「✅ 部署完成」通知；失败时收到 ❌ 通知 |
| 回滚能力 | docker-infra Job 填写旧版 IMAGE_TAG 后成功部署旧版本，Smoke Test 通过 |
| GC Job | 手工触发 registry-gc，Registry 保留最新 10 个版本 tag，多余 tag 被清理；飞书收到 GC 完成通知 |
| 全量幂等 | `playbooks/site.yml` 重复执行 `changed=0 failed=0`（registry 和 ruoyi role） |
| 敏感信息检查 | 普通文件中不出现真实 Registry 密码或飞书 webhook token |
