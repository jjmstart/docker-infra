# 操作手册：v1.6 → v1.7

## 演进概述

在 V1.6 已完成应用交付流水线的基础上，为 gz-03 MySQL Master 增加每日 `mysqldump` 逻辑备份、OSS 远端上传和 bj-01 临时库恢复演练，建立“定期备份 → 远端存储 → 恢复验证 → RPO/RTO 说明”的最小闭环。

---

## 前置条件

| 项目 | 要求 |
|------|------|
| 当前架构版本 | V1.6，私有 Registry、ruoyi CI/CD、Smoke Test 已部署并验收 |
| bj-01 | `/opt/docker-infra` Git 仓库存在；可执行 `ansible-playbook`；可 SSH 到 gz-03；Docker 正常运行；本次恢复演练时 `127.0.0.1:3307` 未被占用 |
| gz-03 | MySQL Master 容器 `mysql` 正常运行；`/opt/docker/backend/mysql/.env` 存在；根分区剩余空间 ≥ 2GB |
| OSS bucket | 已准备阿里云 OSS bucket；bucket 内备份前缀使用 `docker-infra/mysql/ruoyi`；bucket 建议配置 30 天生命周期清理 |
| OSS RAM 用户授权 | RAM 用户必须**已通过自定义授权策略**绑定该 bucket 的 `oss:PutObject` / `GetObject` / `DeleteObject` / `ListObjects` / `GetBucketLocation` 权限。仅新建 RAM 用户和 AccessKey 不会自动获得 bucket 权限，遗漏会导致 Phase 5 上传 `403 AccessDenied` |
| 备份范围 | 只备份业务库 `ry-vue`，不备份 Redis，不做 Binlog/PITR |
| 密钥管理 | OSS AccessKey、bucket、endpoint 必须写入 `vault/secrets.yml`；Git 普通文件中不得出现明文 AccessKey、Secret、数据库密码 |
| 网络 | gz-03 与 bj-01 均可访问 OSS endpoint；bj-01 可通过 Tailscale 管理 gz-03 |
| Ansible vault 密码 | 当前 `ansible.cfg` 设置了 `vault_password_file = vault/.vault-pass` 但该文件不存在；所有涉及 vault 的命令（`ansible-playbook` / `ansible -m shell` / `ansible-vault view` 等）必须显式追加 `--ask-vault-pass` 交互输入，否则会报 `Error getting vault password file ... not found` |

## 各 Phase 详细步骤

### Phase 1：基线检查

**① 目标说明**

确认当前 V1.6 基线稳定，gz-03 MySQL Master 可访问，bj-01 恢复演练端口未占用，避免在异常状态上继续叠加备份配置。

**② 节点**：bj-01（100.118.69.78）

**③ 具体命令**

> 全 runbook 的 ad-hoc 与 playbook 命令一律带 `--ask-vault-pass`：当前仓库 `ansible.cfg` 配置了 `vault_password_file = vault/.vault-pass`，但该密码文件没有提交到 Git；`ansible` 在初始化阶段会强制校验该文件存在，缺失则直接 `ERROR! The vault password file ... was not found`，必须用 `--ask-vault-pass` 显式覆盖（即使该命令本身并不解密 vault 变量）。

```bash
cd /opt/docker-infra

# 确认工作区状态，记录当前未提交文件，避免后续混淆
git status --short

# 确认 Ansible 可达
ansible db_master_nodes -i inventory/hosts.yml -m ping --ask-vault-pass
ansible ops_nodes -i inventory/hosts.yml -m ping --ask-vault-pass

# 确认 gz-03 MySQL 容器与 .env 存在
ansible db_master_nodes -i inventory/hosts.yml -m shell -a '
docker ps | awk "$NF == \"mysql\" {print}"
test -f /opt/docker/backend/mysql/.env
' --ask-vault-pass

# 确认业务库 ry-vue 存在，并粗略记录表数量
ansible db_master_nodes -i inventory/hosts.yml -m shell -a '
cd /opt/docker/backend/mysql
set -a && . ./.env && set +a
docker exec -i mysql mysql -uroot -p"${MYSQL_ROOT_PASSWORD}" -N -e "
SELECT schema_name FROM information_schema.schemata WHERE schema_name = '\''ry-vue'\'';
SELECT COUNT(*) FROM information_schema.tables WHERE table_schema = '\''ry-vue'\'';
"
' --become --ask-vault-pass

# 确认 gz-03 磁盘空间
ansible db_master_nodes -i inventory/hosts.yml -m shell -a 'df -h / /opt/docker || true' --ask-vault-pass

# bj-01 恢复演练端口 3307 不应被占用
ss -lntp | awk '$4 ~ /:3307$/ {print}'
```

**④ 预期输出**

```text
db_master_nodes 与 ops_nodes ping 均为 SUCCESS。
gz-03 docker ps 可见 mysql 容器 Up。
业务库查询输出包含 ry-vue，表数量 > 0。
gz-03 根分区或 /opt/docker 所在分区剩余空间 ≥ 2GB。
ss 检查 3307 无输出。
```

**⑤ 验证方法**

```bash
cd /opt/docker-infra

# MySQL 主库基础连通性
ansible db_master_nodes -i inventory/hosts.yml -m shell -a '
cd /opt/docker/backend/mysql
set -a && . ./.env && set +a
docker exec -i mysql mysqladmin ping -uroot -p"${MYSQL_ROOT_PASSWORD}" --silent
' --become --ask-vault-pass
```

预期：`mysqld is alive`

**⑥ 回滚方式**

本 Phase 只读检查，不产生变更。若 MySQL 容器异常、`ry-vue` 不存在或 3307 被占用，先修复基线问题再继续。

### Phase 2：初始化变量与密钥

**① 目标说明**

将 OSS 凭据写入 ansible-vault，将备份路径、数据库名、保留天数、cron 时间和 OSS prefix 写入普通变量文件。普通文件只存非敏感配置。

**② 节点**：bj-01（100.118.69.78）

**③ 具体命令**

在 `inventory/group_vars/all.yml` 末尾追加：

```yaml
# ── MySQL 逻辑备份（V1.7）───────────────────────────────────────────────────
mysql_backup_database: "ry-vue"
mysql_backup_dir: "{{ docker_root }}/backup/mysql"
mysql_backup_retention_days: 3
mysql_backup_cron_hour: "2"
mysql_backup_cron_minute: "0"
mysql_backup_oss_prefix: "docker-infra/mysql/ruoyi"

ossutil_version: "1.7.18"
ossutil_install_dir: /usr/local/bin
```

> **前置：阿里云 OSS 准备**
>
> 在执行 vault 编辑前，先在阿里云控制台完成以下两件事，否则 Phase 5 上传会报 `StatusCode=403, ErrorCode=AccessDenied`：
>
> 1. **创建 RAM 用户**并签发 AccessKey（不要直接使用主账号 AK）。
> 2. **授权该 RAM 用户对目标 bucket 的写权限**：可挂 `AliyunOSSFullAccess`（粒度粗），或自定义策略（推荐，最小权限）：
>
>    ```json
>    {
>      "Version": "1",
>      "Statement": [
>        {
>          "Effect": "Allow",
>          "Action": [
>            "oss:PutObject",
>            "oss:GetObject",
>            "oss:DeleteObject",
>            "oss:ListObjects",
>            "oss:GetBucketLocation"
>          ],
>          "Resource": [
>            "acs:oss:*:*:<bucket-name>",
>            "acs:oss:*:*:<bucket-name>/*"
>          ]
>        }
>      ]
>    }
>    ```
>
>    Resource 必须**两条都给**：bucket 本身（用于 `ListObjects`）+ `bucket/*`（用于对象级操作）。
>
> 3. 建议在 OSS 控制台同步配置 bucket **生命周期规则**：对前缀 `docker-infra/mysql/ruoyi/` 设置 30 天后自动删除（或转归档），与本地 3 天保留形成两段式留存。

使用 ansible-vault 写入 OSS 凭据：

```bash
cd /opt/docker-infra
ansible-vault edit vault/secrets.yml
```

在 vault 中新增以下变量，真实值只在 vault 编辑器中填写：

```yaml
oss_access_key_id: "<填写 RAM AccessKey ID>"
oss_access_key_secret: "<填写 RAM AccessKey Secret>"
oss_bucket_name: "<填写 OSS bucket 名称>"
oss_endpoint: "<填写 OSS endpoint，例如 oss-cn-guangzhou.aliyuncs.com>"
```

**④ 预期输出**

```text
inventory/group_vars/all.yml 新增 mysql_backup_* 和 ossutil_* 普通变量。
vault/secrets.yml 仍为 ansible-vault 加密文件。
```

**⑤ 验证方法**

```bash
cd /opt/docker-infra

# vault 文件必须仍是加密格式
sed -n '1p' vault/secrets.yml

# 能通过 vault 查看新变量名，但不要把真实值复制到普通文件
ansible-vault view vault/secrets.yml --ask-vault-pass | grep -E '^oss_(access_key_id|access_key_secret|bucket_name|endpoint):'

# 普通文件中不应出现 AccessKey 明文；如误匹配到变量名是正常的，真实值不能出现
# bj-01 默认未安装 ripgrep，使用 grep -rEi 替代
grep -rEi "LTAI|accessKeySecret" inventory roles playbooks Docs
```

**预期验证输出**

```text
vault/secrets.yml 第一行为 $ANSIBLE_VAULT;1.1;AES256。
ansible-vault view 可见 4 个 oss_* 变量名。
grep 检查普通文件中不出现真实 AccessKey（"LTAI" 前缀）或 Secret 明文。
```

**⑥ 回滚方式**

如变量写错，重新执行 `ansible-vault edit vault/secrets.yml` 修正；如普通文件误写入真实密钥，立即删除明文并到阿里云 RAM 控制台禁用并重建 AccessKey。

### Phase 3：新增 mysql-backup role 与 playbook

**① 目标说明**

创建 `roles/mysql-backup/` 和 `playbooks/setup_backup.yml`。该 role 在 gz-03 和 bj-01 安装 ossutil 并渲染 `/root/.ossutilconfig`；仅在 gz-03 下发备份脚本和 cron。

**② 节点**：bj-01（100.118.69.78）

> **强提醒（v1.6→v1.7 实操踩坑）**：在 IDE 中创建 `tasks/main.yml` 和 templates 文件后，**必须按 `Ctrl+S` 保存到磁盘**，再触发 Ansible 部署。曾遇到的故障：模板内容仅在 IDE buffer 中，磁盘上是 0 字节；Ansible 模板渲染时不会报错，gz-03 收到一个语法合法但内容为空的 `mysql_backup.sh`，task 仍显示 `changed=1`，但脚本执行 `rc=0` 无任何输出。Phase 4 部署后用 `wc -l` 或 `md5sum` 核对 gz-03 上的脚本字节数比看 `changed` 数更可靠。

**③ 具体命令**

```bash
cd /opt/docker-infra

mkdir -p roles/mysql-backup/tasks roles/mysql-backup/templates playbooks
```

创建 `roles/mysql-backup/tasks/main.yml`：

```yaml
---
- name: Ensure ossutil install directory exists
  ansible.builtin.file:
    path: "{{ ossutil_install_dir }}"
    state: directory
    mode: "0755"

- name: Install ossutil fixed version
  ansible.builtin.get_url:
    url: "https://gosspublic.alicdn.com/ossutil/{{ ossutil_version }}/ossutil64"
    dest: "{{ ossutil_install_dir }}/ossutil-{{ ossutil_version }}"
    mode: "0755"
  when: not ansible_check_mode

- name: Link ossutil binary
  ansible.builtin.file:
    src: "{{ ossutil_install_dir }}/ossutil-{{ ossutil_version }}"
    dest: "{{ ossutil_install_dir }}/ossutil"
    state: link
    force: true

- name: Render ossutil config
  ansible.builtin.template:
    src: ossutilconfig.j2
    dest: /root/.ossutilconfig
    owner: root
    group: root
    mode: "0600"
  no_log: true

- name: Ensure MySQL backup directory exists on master
  ansible.builtin.file:
    path: "{{ mysql_backup_dir }}"
    state: directory
    owner: root
    group: root
    mode: "0750"
  when: inventory_hostname in groups['db_master_nodes']

- name: Render MySQL backup script on master
  ansible.builtin.template:
    src: mysql_backup.sh.j2
    dest: "{{ mysql_backup_dir }}/mysql_backup.sh"
    owner: root
    group: root
    mode: "0750"
  when: inventory_hostname in groups['db_master_nodes']

- name: Configure MySQL backup cron on master
  ansible.builtin.cron:
    name: docker-infra mysql logical backup to OSS
    user: root
    minute: "{{ mysql_backup_cron_minute }}"
    hour: "{{ mysql_backup_cron_hour }}"
    job: "{{ mysql_backup_dir }}/mysql_backup.sh >> {{ mysql_backup_dir }}/mysql_backup.log 2>&1"
  when: inventory_hostname in groups['db_master_nodes']
```

创建 `roles/mysql-backup/templates/ossutilconfig.j2`：

```ini
[Credentials]
language=EN
endpoint={{ oss_endpoint }}
accessKeyID={{ oss_access_key_id }}
accessKeySecret={{ oss_access_key_secret }}
```

创建 `roles/mysql-backup/templates/mysql_backup.sh.j2`：

```bash
#!/usr/bin/env bash
set -Eeuo pipefail

MYSQL_ENV_FILE="{{ docker_root }}/backend/mysql/.env"
BACKUP_DIR="{{ mysql_backup_dir }}"
MYSQL_DATABASE="{{ mysql_backup_database }}"
RETENTION_DAYS="{{ mysql_backup_retention_days }}"
OSS_BUCKET="{{ oss_bucket_name }}"
OSS_PREFIX="{{ mysql_backup_oss_prefix }}"
OSSUTIL="{{ ossutil_install_dir }}/ossutil"

timestamp="$(date +%Y%m%d_%H%M%S)"
backup_file="${BACKUP_DIR}/ruoyi_backup_${timestamp}.sql.gz"
tmp_file="${backup_file}.tmp"

cd "{{ docker_root }}/backend/mysql"
set -a
# shellcheck disable=SC1090
. "${MYSQL_ENV_FILE}"
set +a

echo "[INFO] $(date '+%F %T') start mysql logical backup, database=${MYSQL_DATABASE}"

docker exec -e MYSQL_PWD="${MYSQL_ROOT_PASSWORD}" -i mysql \
  mysqldump \
  -uroot \
  --default-character-set=utf8mb4 \
  --single-transaction \
  --routines \
  --triggers \
  --events \
  --set-gtid-purged=OFF \
  --databases "${MYSQL_DATABASE}" \
  | gzip -c > "${tmp_file}"

test -s "${tmp_file}"
mv "${tmp_file}" "${backup_file}"

"${OSSUTIL}" cp -f "${backup_file}" "oss://${OSS_BUCKET}/${OSS_PREFIX}/$(basename "${backup_file}")" \
  --config-file /root/.ossutilconfig

find "${BACKUP_DIR}" -type f -name "ruoyi_backup_*.sql.gz" -mtime +"${RETENTION_DAYS}" -delete

echo "[INFO] $(date '+%F %T') backup uploaded: oss://${OSS_BUCKET}/${OSS_PREFIX}/$(basename "${backup_file}")"
```

创建 `playbooks/setup_backup.yml`：

```yaml
---
- name: Setup MySQL logical backup and restore drill tools
  hosts: "db_master_nodes:ops_nodes"
  become: true
  gather_facts: true
  vars_files:
    - ../vault/secrets.yml
  roles:
    - mysql-backup
```

**④ 预期输出**

```text
roles/mysql-backup/ 目录包含 tasks/main.yml 与两个 template。
playbooks/setup_backup.yml 存在。
普通文件中只出现 vault 变量名，不出现真实 OSS 密钥值。
```

**⑤ 验证方法**

```bash
cd /opt/docker-infra

# 注意：playbook syntax-check 也要带 --ask-vault-pass，因为 vars_files 引用了 vault
ansible-playbook -i inventory/hosts.yml playbooks/setup_backup.yml --syntax-check --ask-vault-pass

# 检查 4 个文件是否落盘成功（0 字节即说明 IDE 未保存）
wc -l roles/mysql-backup/templates/*.j2 \
      roles/mysql-backup/tasks/main.yml \
      playbooks/setup_backup.yml

# 检查普通文件是否误写真实密钥；变量名本身出现是正常的
grep -rEi "LTAI|accessKeySecret" roles/mysql-backup playbooks inventory
```

**预期验证输出**

```text
playbook: playbooks/setup_backup.yml
（无 ERROR / WARNING）

mysql_backup.sh.j2 行数 ≥ 40，ossutilconfig.j2 行数 ≥ 4，tasks/main.yml 行数 ≥ 50，setup_backup.yml 行数 ≥ 8。
任意文件行数为 0 / 1 即代表 IDE 未保存到磁盘，必须先回 IDE 按 Ctrl+S 保存再继续。

grep 检查普通文件中不出现 LTAI 或 accessKeySecret 真实值。
```

**⑥ 回滚方式**

```bash

rm -rf roles/mysql-backup
rm -f playbooks/setup_backup.yml
```

### Phase 4：部署备份配置并做幂等检查

**① 目标说明**

通过 Ansible 将 ossutil、OSS 配置、备份脚本和 cron 下发到目标节点，并确认重复执行稳定收敛。

**② 节点**：bj-01（100.118.69.78）发起；目标为 gz-03 与 bj-01。

**③ 具体命令**

```bash
cd /opt/docker-infra

# 语法检查
ansible-playbook -i inventory/hosts.yml playbooks/setup_backup.yml --syntax-check --ask-vault-pass

# 预览变更
ansible-playbook -i inventory/hosts.yml playbooks/setup_backup.yml --check --diff --ask-vault-pass

# 正式部署
ansible-playbook -i inventory/hosts.yml playbooks/setup_backup.yml --ask-vault-pass

# 重复执行，确认幂等
ansible-playbook -i inventory/hosts.yml playbooks/setup_backup.yml --check --diff --ask-vault-pass
```

**④ 预期输出**

```text
首次正式部署允许 changed > 0。
第二次 --check --diff 目标为 changed=0 failed=0。
如 get_url 因 check mode 被跳过，属于预期；正式部署后重复检查应稳定。
```

**⑤ 验证方法**

```bash
cd /opt/docker-infra

# gz-03：备份目录、脚本、cron 存在；脚本字节数 > 0
ansible db_master_nodes -i inventory/hosts.yml -m shell -a '
ls -ld /opt/docker/backup/mysql
ls -l /opt/docker/backup/mysql/mysql_backup.sh
wc -l /opt/docker/backup/mysql/mysql_backup.sh
crontab -l | grep -E "docker-infra mysql logical backup|mysql_backup.sh"
/usr/local/bin/ossutil --version
test -f /root/.ossutilconfig && stat -c "%a %n" /root/.ossutilconfig
' --become --ask-vault-pass

# bj-01：仅作为恢复节点，确认 ossutil 与 config 已准备好，不应有备份 cron
sudo /usr/local/bin/ossutil --version
sudo test -f /root/.ossutilconfig && sudo stat -c "%a %n" /root/.ossutilconfig
sudo crontab -l 2>/dev/null | grep "mysql_backup.sh" || echo "no backup cron on bj-01 (expected)"
```

> 关键点：**`mysql_backup.sh` 的 `wc -l` 必须 ≥ 40 行**。若为 0 行，说明 Phase 3 模板源文件 IDE 未保存，需要回到 IDE `Ctrl+S` 保存后重新执行本 Phase 第三步（正式部署）。重新部署后这个 task 应显示 `changed=1`，再次幂等检查回到 `changed=0`。

**⑥ 回滚方式**

```bash
cd /opt/docker-infra

# 删除 cron
ansible db_master_nodes -i inventory/hosts.yml -m cron -a \
  'name="docker-infra mysql logical backup to OSS" state=absent user=root' --become

# 删除已下发脚本和目录（确认无需要保留的备份文件后再执行）
ansible db_master_nodes -i inventory/hosts.yml -m file -a \
  'path=/opt/docker/backup/mysql state=absent' --become

# 删除 ossutil 配置（如后续不再用于恢复演练）
ansible "db_master_nodes:ops_nodes" -i inventory/hosts.yml -m file -a \
  'path=/root/.ossutilconfig state=absent' --become
```

### Phase 5：首次手动备份与 OSS 上传验证

**① 目标说明**

在 gz-03 手动执行一次备份脚本，确认可生成本地 `.sql.gz` 文件并上传到 OSS 指定前缀。

```text
[INFO] YYYY-MM-DD HH:MM:SS start mysql logical backup, database=ry-vue
```

**⑤ 验证方法**


**预期验证输出**


> **若上传报 `403 AccessDenied / bucket acl`**：根因是 RAM 用户没有该 bucket 的写权限（参见前置条件"OSS RAM 用户授权"）。`ossutil` 已通过 AccessKey 鉴权（否则会报 `InvalidAccessKeyId`/`SignatureDoesNotMatch`），但 RAM 用户策略未授予 bucket 操作。到阿里云 RAM 控制台 → 用户 → 权限管理 → 新增自定义授权策略，对该 bucket 授予 `oss:PutObject/GetObject/DeleteObject/ListObjects/GetBucketLocation`（Resource 必须同时包含 `acs:oss:*:*:<bucket>` 和 `acs:oss:*:*:<bucket>/*` 两条），保存后重跑本 Phase。


### Phase 6：bj-01 恢复演练与 RTO 记录

**① 目标说明**

从 OSS 下载最新备份到 bj-01，启动独立临时 MySQL 容器 `mysql-restore-test`（端口 3307），导入 `ry-vue` 并做数据校验。该演练不接入现有主从链路。

**② 节点**：bj-01（100.118.69.78），**必须以 root 身份执行**

> **为什么必须 root**：备份脚本是 root 用户的 cron job，`/root/.ossutilconfig` 权限为 `0600 root:root`，普通用户（如 `admin-alex`）执行 `ossutil` 会报 `Read config file error: open /root/.ossutilconfig: permission denied`，导致 `latest_object` 解析为空字符串，后续下载、解压、导入静默失败。在恢复演练前先 `sudo -i` 切到 root shell。

**③ 具体命令**

```bash
# Step 1：切到 root shell
sudo -i

# Step 2：读取 OSS bucket 名（也可以直接硬编码本次实际 bucket 名，避免每次输 vault 密码）
cd /opt/docker-infra
OSS_BUCKET_NAME="$(
  ansible-vault view vault/secrets.yml --ask-vault-pass \
    | awk -F': *' '$1 == "oss_bucket_name" {gsub(/"/, "", $2); print $2; exit}'
)"
echo "OSS_BUCKET_NAME=${OSS_BUCKET_NAME}"
test -n "${OSS_BUCKET_NAME}"

# Step 3：变量初始化
RESTORE_DIR=/tmp/mysql-restore-drill
OSS_PREFIX=docker-infra/mysql/ruoyi
RESTORE_CONTAINER=mysql-restore-test
RESTORE_ROOT_PASSWORD="$(openssl rand -base64 24)"
START_TS="$(date +%s)"

mkdir -p "${RESTORE_DIR}"
cd "${RESTORE_DIR}"

# Step 4：找到 OSS 上最新备份并下载
latest_object="$(
  ossutil ls "oss://${OSS_BUCKET_NAME}/${OSS_PREFIX}/" --config-file /root/.ossutilconfig \
    | awk '/ruoyi_backup_.*\.sql\.gz$/ {print $NF}' \
    | sort \
    | tail -n 1
)"
echo "latest_object=${latest_object}"
test -n "${latest_object}"

ossutil cp -f "${latest_object}" . --config-file /root/.ossutilconfig

backup_file="$(basename "${latest_object}")"
test -s "${backup_file}"
gzip -t "${backup_file}" && echo "gzip OK"

# Step 5：清理可能的同名残留容器
docker rm -f "${RESTORE_CONTAINER}" >/dev/null 2>&1 || true

# Step 6：启动独立临时库（127.0.0.1:3307 避开生产 3306 延迟从）
docker run -d \
  --name "${RESTORE_CONTAINER}" \
  --network global_gateway \
  -p 127.0.0.1:3307:3306 \
  -e MYSQL_ROOT_PASSWORD="${RESTORE_ROOT_PASSWORD}" \
  mysql:8.0 \
  --default-authentication-plugin=mysql_native_password \
  --performance_schema=off

# Step 7：等待 mysql init 完成
# mysql:8.0 镜像首次启动会执行 entrypoint init 流程（创建 root 账号等），中途会有一次 SIGTERM + 重启；
# 直接 mysqladmin ping 在过早阶段可能命中"init 中"假死状态，因此先 sleep 10 让 init 落地。
echo "waiting mysql-restore-test ready..."
sleep 10
for i in $(seq 1 60); do
  if docker exec -i "${RESTORE_CONTAINER}" \
       mysqladmin ping -h127.0.0.1 -uroot -p"${RESTORE_ROOT_PASSWORD}" --silent 2>/dev/null; then
    echo "mysql ready after ${i} attempts"
    break
  fi
  sleep 2
done

# Step 8：导入备份（docker exec 必须 -i 不加 -t；mysql 客户端用 -h127.0.0.1 强制 TCP 而非 socket，避开 socket 竞争）
gunzip -c "${backup_file}" | docker exec -i "${RESTORE_CONTAINER}" \
  mysql -h127.0.0.1 -uroot -p"${RESTORE_ROOT_PASSWORD}"

# Step 9：数据校验
docker exec -i "${RESTORE_CONTAINER}" \
  mysql -h127.0.0.1 -uroot -p"${RESTORE_ROOT_PASSWORD}" -e "
SHOW DATABASES LIKE 'ry-vue';
SELECT COUNT(*) AS table_count FROM information_schema.tables WHERE table_schema = 'ry-vue';
SELECT COUNT(*) AS sys_user_count FROM \`ry-vue\`.sys_user;
"

END_TS="$(date +%s)"
echo "============================================"
echo "restore_rto_seconds=$((END_TS - START_TS))"
echo "============================================"
```

**④ 预期输出**

```text
OSS_BUCKET_NAME=<bucket 名>
latest_object=oss://<bucket>/docker-infra/mysql/ruoyi/ruoyi_backup_YYYYMMDD_HHMMSS.sql.gz
ossutil 下载 Succeed: Total num: 1, size: <bytes>. OK num: 1(download 1 objects)
gzip OK
mysql ready after N attempts （N 通常 ≤ 5）
SHOW DATABASES LIKE 'ry-vue' 返回 ry-vue
table_count = <若依实际表数，本项目实测 21>
sys_user_count > 0（实测 4）
restore_rto_seconds=<秒数>（实测约 30~40s，含 OSS 下载 ~5s、容器初始化 ~14s、导入 + 校验 ~15s）
```

**⑤ 验证方法**

```bash
# 在 root shell 内继续验证（mysql-restore-test 容器和 RESTORE_ROOT_PASSWORD 仍在当前 shell 上下文）
docker ps --filter "name=mysql-restore-test" --format "{{.Names}}\t{{.Status}}\t{{.Ports}}"

# 通过 docker exec 二次校验（不依赖宿主机有 mysql 客户端）
docker exec -i mysql-restore-test \
  mysql -h127.0.0.1 -uroot -p"${RESTORE_ROOT_PASSWORD}" -e "
SHOW TABLES IN \`ry-vue\` LIMIT 5;
SELECT user_name FROM \`ry-vue\`.sys_user LIMIT 3;
"
```

**⑥ 回滚方式**

```bash
# 仍在 root shell 内
docker rm -f mysql-restore-test
rm -rf /tmp/mysql-restore-drill
```

该 Phase 不修改生产 MySQL，不影响 gz-03 → gz-02 / bj-01 主从复制链路。临时库的 `RESTORE_ROOT_PASSWORD` 由 `openssl rand` 当场生成，仅存于 shell 变量中，容器删除即失效，不会持久化到任何文件。
