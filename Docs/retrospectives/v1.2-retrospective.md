## 重要技术决策解释

**Q：为什么选择半同步复制而不是全同步或异步复制？**

A：这是一个在数据安全性与写性能之间的权衡。

- **异步复制**的风险是：主库提交事务返回成功后，从库可能还没有收到该事务的 binlog，此时主库宕机会导致该事务丢失（RPO > 0，数据丢失窗口取决于主从复制延迟）。
- **全同步复制**要求所有从库都写入并提交后主库才返回成功，安全性最高但性能代价最大——每次写操作都要等待跨城 bj-01（~35ms）的 ACK，对写密集型业务几乎不可接受。
- **半同步复制**的折中方案：主库提交后，只需等待**至少一个**从库将 relay log 写入磁盘（不需要从库提交）即可返回成功。同城 gz-02 的 ACK 延迟约 5ms，对写性能影响极小。当所有从库在 5 秒（timeout）内都未 ACK 时，自动降级为异步，保证主库写可用性不受极端网络分区影响。

在这个四节点集群中，gz-02 同城从库的 ~5ms ACK 让半同步成为近乎"无代价"的 RPO 改善，是显而易见的最优选择。

**Q：为什么 bj-01 要做延迟复制（SOURCE_DELAY=3600），而不是实时复制？**

A：延迟复制解决的是一类特殊灾难场景——**人为误操作**。

主从实时复制能防止硬件故障（主库盘坏、宕机），但防不了 `DROP TABLE` 或 `UPDATE` 忘加 `WHERE` 这类 SQL 操作错误：这些 DDL/DML 会立刻复制到所有从库，让所有从库同步"中招"，此时即使切换到从库也已经没有正确数据了。

延迟复制的原理是：IO_Thread 实时接收主库的 binlog（数据包实时到达），但 SQL_Thread 延迟 3600 秒才应用这些 binlog。这意味着如果发现误操作，在 1 小时内可以登录 bj-01，`STOP REPLICA SQL_THREAD`，然后等待 relay log 回放到误操作前的时间点，从 bj-01 恢复数据。

选择 3600 秒是基于经验估算：绝大多数误操作能在 1 小时内被发现（比如业务报错、用户反馈），超过 1 小时才发现的误操作通常需要依赖全量备份恢复，延迟复制窗口过长反而会增加 bj-01 本身的数据落后量。

**Q：为什么用 ProxySQL 做读写分离，而不是在应用层（若依代码里）配置多数据源？**

A：主要原因是**解耦**和**运维灵活性**。

在应用层配置多数据源（如 Spring AbstractRoutingDataSource）的方案，意味着主从拓扑信息会写入应用配置，每次主从切换（故障切换或计划维护）都需要修改配置、重启应用——对于多个 ruoyi 实例（ruoyi-admin-1 在 gz-03、ruoyi-admin-2 在 gz-02），需要协调多处变更，增加操作复杂度和出错风险。

ProxySQL 作为独立的代理层，应用只连接 `100.92.5.116:6033` 这一个地址，完全不感知后端是几个节点、各节点是什么角色。路由规则（SELECT → Slave1，其他 → Master）在 ProxySQL 内部维护，可热更新，无需重启任何应用服务。当需要主从切换时，只需通过 ProxySQL Admin 更新 `mysql_servers` 表，变更即时生效。这是在当前规模下最"运维友好"的实现方式。

**Q：为什么暂不引入 MHA 或 Orchestrator 做自动 Failover？**

A：4 台节点的规模不需要引入 Orchestrator 的复杂度。Orchestrator 本身也需要运维（部署、配置、升级、高可用），在集群规模较小时，引入它的运维成本高于它带来的收益。当前的手动切换 SOP（参见 Phase 6 故障演练）+ Prometheus 告警，已经能将 RTO 控制在 3-5 分钟内，对于这个业务规模是可以接受的。当集群扩展到 8 个以上 MySQL 节点，或者需要 7×24 无人值守的自动切换时，再引入 Orchestrator 更合适。

## 踩坑记录

| 坑点描述 | 根因 | 解决方法 | 下次规避手段 |
|----------|------|----------|-------------|
| `docker exec -it mysql ... <<'EOF'` 报错 `the input device is not a TTY` | `-t` 参数会分配伪终端，与 heredoc 的非交互模式冲突 | 改用 `docker exec -i`（去掉 `-t`） | heredoc 或管道输入时，`docker exec` 一律用 `-i` 不加 `-t` |
| ProxySQL Admin SQL 中更新 `hostname` 字段无效 | `mysql_servers` 表在 Admin SQL 中字段名是 `hostname`，但 `proxysql.cnf` 配置文件中字段名是 `address`，两个接口的字段名不一致 | Admin SQL 用 `hostname`，cnf 文件用 `address`，分别对照各自的文档 | 操作前先 `SELECT * FROM mysql_servers` 确认实际字段名，不凭记忆操作 |
| 从库复制账号密码不清楚，Phase 2/3 `CHANGE REPLICATION SOURCE TO` 报认证错误 | repl_user 密码存于各节点 `/opt/docker/backend/mysql/.env` 中的 `MYSQL_REPL_PASSWORD` 变量，未预先记录 | 从 `.env` 文件读取 `MYSQL_REPL_PASSWORD` 后再执行命令 | 实施前先 `cat /opt/docker/backend/mysql/.env` 确认所有密码变量 |
| Phase 6 故障演练首次实测 RTO 约 19 分钟 | 包含 ProxySQL Admin 字段名踩坑（`hostname` vs `address`）排查时间约 16 分钟，加上实际切换操作约 3 分钟 | 排清字段名问题后，后续切换流程顺畅 | 故障演练前先在非生产环境验证所有命令；SOP 中明确写出字段名 |

## 操作心得

- **`docker exec` + heredoc 的组合坑位值得专门记录**：`-i` 和 `-t` 的区别（交互式 TTY vs 管道输入）在日常操作中很少遇到，但一旦混用会在最需要快速操作的故障演练时卡住。记住口诀：有 heredoc/管道就用 `-i`，人工交互才加 `-t`。
- **ProxySQL 的两套操作接口需要分开记忆**：cnf 配置文件用 `address`，Admin SQL 用 `hostname`——这是历史遗留的不一致设计，官方文档在不同页面使用不同字段名，实际操作前一定要 `DESCRIBE mysql_servers` 确认当前接口的字段名。
- **密码和环境变量集中管理的重要性**：本次 repl_user 密码分散在各节点 `.env` 文件中，多节点操作时需要逐节点核对，增加了认知负担。建议后续将所有密码统一记录在安全位置（如团队密码管理工具），或考虑使用 Ansible Vault 统一管理。
- **延迟复制的实测效果超出预期**：Phase 6 中验证了 bj-01 延迟复制的误删恢复流程——`STOP REPLICA SQL_THREAD` 后，relay log 中保留了所有 1 小时内的操作记录，可以精确定位到误操作前的时间点，是一个低成本但高价值的灾备手段。
- **故障演练必须实际操练，不能只看文档**：此次演练暴露了命令层面的细节问题（字段名不一致），这些问题在文档审查中很难发现，只有实际操作才会踩到。建议每次大版本升级后都安排一次 GameDay。

## 遗留问题 / 下一步建议

- **ProxySQL 单点风险**：当前 ProxySQL 仅部署于 gz-03，ProxySQL 本身宕机则读写均不可达（比 MySQL 主库宕机影响面更大）。V1.3 应优先考虑 ProxySQL Cluster 或双节点热备。
- **手动切换 SOP 需要文档化为可执行 Runbook**：Phase 6 已验证流程，但命令分散在演练记录中，建议整理为单独的《MySQL 故障切换 SOP》，含故障判定标准、切换命令、验证命令、回滚命令，打印备查。
- **mysqldump 备份计划尚未自动化**：当前仅在部署时手动执行了 mysqldump，需要配置定时任务（cron）定期全量备份，并验证备份文件可恢复。
- **告警通道尚未接入**：Prometheus 告警规则已配置，但 Alertmanager 尚未部署，复制断连等关键事件无法自动通知到人。建议接入企业微信/钉钉 Webhook。
- **端口标准化（8443 → 443）**：备案通过后，Nginx 端口从 8443 改为 443 + 开启 HSTS + 接入 Cloudflare CDN，是下一个高优先级项。
