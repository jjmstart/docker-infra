# 演进总结：V1.1

## 本次演进一句话摘要

将监控栈（Prometheus + Grafana）从 bj-01 跨城迁移至 gz-01 同机部署，消除了"被监控节点宕机时监控链路同时不可用"的隐患，并借此机会补齐了全部四个节点的 Node Exporter 覆盖。

| Phase | 完成内容 | 关键验证指标 |
|-------|----------|-------------|
| 1 | bj-01 停止 Prometheus + Grafana 容器，配置注释保留 | `docker compose ps` 无 prometheus/grafana 运行中 |

## 重要技术决策解释

**Q：为什么要把 Prometheus 和 Grafana 从 bj-01 迁到 gz-01，而不是保持原位？**

A：监控系统的核心职责是在其他服务出现问题时发出警报。若监控与被监控的服务部署在同一节点（bj-01），一旦该节点宕机（无论是硬件故障、网络中断还是 OOM），监控栈本身也会同时失联，此时 Prometheus 无法采集数据、Grafana 无法访问、告警无法发出——正是最需要监控的时刻，监控反而不可用。迁至 gz-01 后，监控主节点与 Jenkins/Diary 等运维应用（bj-01）以及业务节点（gz-02/gz-03）解耦，三组节点互相独立，任意一组故障不影响监控链路。

**Q：为什么把监控放到 gz-01（入口网关）而不是单独新建一台节点？**

A：当前集群规模（4 台服务器）不足以支撑为监控单独申请一台节点的成本。gz-01 是 2C2G 的轻量入口网关，Nginx 本身资源消耗极低，Prometheus + Grafana + node-exporter 在采集 4 个节点的场景下内存占用约 300-500MB，与 Nginx 共存完全可行。此外，`monitor.jjmstart.com` 域名本已指向 gz-01，迁入后 Nginx 反代可走本地网络（0ms），访问体验也更好。

**Q：为什么 bj-01 的 Prometheus/Grafana 配置要注释保留而不是直接删除？**

A：注释保留的作用是留存"原始配置对照"——如果将来迁移过程中出现问题，可以快速对比新旧配置差异。此外，bj-01 的 `docker-compose.bj-01.yml` 命名约定也为后续各节点维护独立 compose（如 mysql、exporter）提供了参考范式。

## 踩坑记录

| 坑点描述 | 根因 | 解决方法 | 下次规避手段 |
|----------|------|----------|-------------|
| Prometheus 启动后 gz-02/gz-03 targets 显示 down | node-exporter 尚未在 gz-02/gz-03 部署，属预期现象，但容易误判为配置错误 | 按 Phase 顺序完成 gz-02/gz-03 node-exporter 部署后自动恢复 | 在 Prometheus 配置注释中说明"targets 按 Phase 逐步上线"，避免误判 |
| bj-01 同时运行旧版 docker-compose.yml 和新版 docker-compose.bj-01.yml 时容器名冲突 | 两个 compose 文件都定义了 `node-exporter` 容器，端口绑定 :9100 重复 | 先停止旧版 compose 中的 node-exporter，再启动新版 | 拆分 compose 文件时检查容器名和端口是否重复 |
| Nginx reload 后 monitor 域名短暂出现 502 | Grafana 容器启动后需几秒完成初始化才能接受连接，nginx reload 时正好命中启动窗口 | 等待 Grafana 就绪后自动恢复；无业务影响 | 可在 docker-compose 中为 grafana 添加 healthcheck，Nginx 启动依赖 healthcheck 通过 |

## 操作心得

- **监控迁移的关键顺序是"先建后拆"**：先在 gz-01 把 Prometheus + Grafana 跑通，确认采集正常，再停 bj-01 的旧服务，避免出现监控空窗期。
- **四节点 node-exporter 的 compose 拆分是值得的**：虽然每个节点单独维护一份 compose 增加了文件数量，但职责清晰（monitor compose 管 exporter，mysql compose 管数据库），后续排查和变更不会互相干扰。
- **Tailscale IP 绑定比 0.0.0.0 更安全**：node-exporter 绑定 Tailscale IP 而非公网 IP，确保指标端口仅在 VPN 内可访问，无需额外防火墙规则。
- **prometheus.yml 里的 targets 可以用注释说明节点身份**：`100.79.132.125:9100  # gz-02` 这类注释在排查采集问题时非常有帮助，比看裸 IP 直观很多。

## 遗留问题 / 下一步建议

- **MySQL 单点风险未解决**：当前 gz-03 MySQL 仍是单点，是下一个优先级最高的高可用改造项（即 V1.2）。
- **Prometheus 告警规则尚未配置**：当前只有采集和展示，无 Alertmanager 告警通道（企业微信/钉钉/邮件），节点宕机无法自动通知。
- **Grafana 持久化验证**：迁移后确认 Grafana 数据卷已正确挂载，避免容器重启后 Dashboard 丢失。
- **gz-01 资源水位**：监控栈迁入后 gz-01（2C2G）负载增加，建议在 Grafana 中为 gz-01 自身设置内存使用率告警阈值（如 > 80%）。
