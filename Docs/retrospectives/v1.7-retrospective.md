## 重要技术决策解释

**Q：为什么用 `mysqldump` 而不是 `mydumper`？**

A：`mydumper` 是多线程备份工具，适合 GB 级以上的大数据库——它的优势在于并行导出多张表、缩短备份窗口，但代价是引入额外二进制安装、需要配套 `myloader` 恢复、输出是多文件目录而非单一 SQL 文件。`ry-vue` 库当前数据量约 7MB（压缩前），`mysqldump --single-transaction` 耗时秒级，单线程完全够用。引入 `mydumper` 在这个规模下是过度设计，且增加了可叙事复杂度而无实质收益。`mysqldump` 的命令直观、输出可读、恢复步骤清晰，是面试中最能讲清楚的选择。

**Q：为什么选阿里云 OSS 而不是腾讯云 COS？**

A：两个账号都有，功能上没有决定性差异。选 OSS 的原因是：国内市场认知度更高，"备份上传到 OSS"在面试中的辨识度优于 COS；`gz-01` 本身是阿里云节点，账号体系已有；ossutil 工具链文档完整，社区资源丰富。这不是"OSS 技术上更好"，而是"同等条件下选认知度更高的"，这本身就是一个可叙事的决策逻辑。

**Q：为什么备份源选 gz-03 主库而不是从库？**

A：备份从库可以减少对主库的潜在影响，但：①`ry-vue` 数据量小，`--single-transaction` 对 InnoDB 主库几乎无影响；②bj-01 延迟从库数据落后 3600s，不是最新数据，以它作全量备份基准会导致 RPO 从 24h 悄悄变成 24h+1h；③gz-02 实时从库理论可用，但主备切换时从库状态需要额外确认。从主库备份是中小公司最常见且最直接的做法，面试叙事最简单清晰。

**Q：`mysqldump` 为什么要加 `--set-gtid-purged=OFF`？**

A：gz-03 开启了 binlog（`mysql_enable_binlog: true`），MySQL 在开启 GTID 的环境下，`mysqldump` 默认会在 dump 文件头部写入 `SET @@GLOBAL.GTID_PURGED=...`。这条语句的意图是告诉目标 MySQL"这份备份对应的 GTID 已被清理"，但在恢复到一个全新独立的临时库时，目标库的 GTID 集合为空，该语句会报错（`ERROR 3546: @@GLOBAL.GTID_PURGED can only be set when @@GLOBAL.GTID_EXECUTED is empty`）。加 `--set-gtid-purged=OFF` 后，dump 文件不包含任何 GTID 相关语句，可在任意 MySQL 8.0 实例上干净导入，不受源库 GTID 状态影响。

**Q：恢复演练为什么选择 bj-01 而不是 gz-03 本地？**

A：演练目的是验证备份文件的**可用性**，不是在生产库上测试恢复。在 gz-03 本地恢复会有两个风险：一是使用 `3306` 端口会与生产 MySQL Master 冲突，必须停止主库或换端口，操作复杂；二是任何失误都可能影响主库数据。bj-01 的 `3307` 端口完全独立，资源也最充裕（4C16G），演练结束后 `docker rm -f` 直接销毁，不留任何痕迹。这是"演练在隔离环境、生产不受影响"的基本原则。

**Q：为什么不做 Redis 备份？**

A：Redis 在本架构中主要用作应用缓存（Session、热点数据），数据丢失后应用可以自动重建缓存，不存在不可恢复的业务数据。v1.7 的目标是建立最小可运转的备份闭环，范围越小越容易收口、越能聚焦在"备份 → 恢复验证"的核心环节。堆砌 Redis 备份会分散注意力、增加复杂度，却不能带来等量的技术叙事价值。

**Q：为什么 RPO 目标是 24 小时，而不是更低？**

A：降低 RPO 的路径是增加备份频率或引入 Binlog 归档（PITR）。更高频率备份会占用更多磁盘和 OSS 空间，对当前数据量意义不大；PITR 需要持续追踪 binlog 文件、定期上传、恢复时重放——复杂度显著上升，超出 v1.7 "最小闭环"的范围定义。24h RPO 是合理的第一版基线：结合 bj-01 的 3600s 延迟从库，误操作场景实际可恢复窗口缩短到 1 小时以内，对学习项目来说这是一个兼顾实用性和可叙事性的起点。PITR 留给后续版本评估。

## 操作心得

- **"IDE 标签页显示了内容"不等于"磁盘上有内容"**：Cursor 等现代 IDE 的 buffer 和磁盘文件是两个状态。在"粘贴完内容但未 `Ctrl+S` 保存"的状态下，Ansible template task 会读磁盘空文件并下发，整个 task 显示 `changed=1` 且 `rc=0`，没有任何错误信号。排查这类问题最有效的命令是 `wc -l` 或 `stat -c "%s"`——任何 0 字节的模板文件都是信号，不要依赖 Ansible 的 `changed` 状态来判断"渲染了有效内容"。

- **ossutil 的 `/root/.ossutilconfig` 是 root 专属的配置，所有涉及 ossutil 的操作都要明确用户上下文**：备份脚本 cron 以 root 跑，没问题；恢复演练以普通用户跑，`permission denied`。这个问题在设计阶段就可以预见——只要注意到 `0600 root:root` 的权限设置，就应该在 runbook 里标注 `sudo -i`。以后凡是涉及 `/root/.*` 路径的操作，先确认当前用户。

- **RAM 用户授权应该在部署前验证，不要等到备份脚本跑起来才发现 403**：正确的顺序是：创建 RAM 用户 → 签发 AK → 配置授权策略 → **用 `ossutil ls oss://<bucket>/` 手动验证可达** → 再写入 vault、下发 Ansible。等到备份脚本执行失败才定位 403，需要在生产节点上搜证排查，成本高。提前验证是运维最小化返工的基本习惯。

- **403 AccessDenied vs InvalidAccessKeyId 是两个不同的错误层次**：`InvalidAccessKeyId`/`SignatureDoesNotMatch` 说明 AK 本身有问题（写错或失效）；`AccessDenied` 加 `bucket acl` 说明 AK 认证通过了，但 RAM 用户策略没有该操作权限。遇到 OSS 403 时先看 ErrorCode：`AccessDenied` → 去 RAM 控制台查授权策略；`InvalidAccessKeyId` → 重新检查 vault 中的 AK 值。

- **`ansible.cfg` 的 `vault_password_file` 配置是全局的**：即使某条 ad-hoc 命令不涉及任何 vault 变量，Ansible 在初始化时也会校验该文件是否存在。这是一个常见的"为什么我的 `ansible -m ping` 也报 vault 错误"的迷惑点。解决方式很简单——所有命令加 `--ask-vault-pass`；但更好的长期方案是在 bj-01 本地落地 `vault/.vault-pass` 文件（加入 `.gitignore`），彻底告别交互输入。

- **ossutil 上传进度条会污染 cron 日志**：ossutil 的进度输出是 `\r` 重画的单行文字（`Total num: 1, size: ... Progress: 29%... 64%...`），被重定向到日志文件后，不同进度行会覆盖同一物理行，导致日志文件只保留最后一帧进度，阅读困难。长期运行下去日志本身不会无限增长，但格式较差。可在 ossutil 命令后加 `2>/dev/null` 屏蔽进度输出，或用 `--loglevel error` 只输出错误，让日志更干净。

- **实测 RTO 34 秒远低于预估的 15-30 分钟**：预估的 15-30 分钟是含人工决策时间（确认故障 ~5min + 操作 ~10min）的端到端 RTO。纯技术操作（OSS 下载 4s + 容器 init 14s + 导入 + 校验 16s）只需 34 秒。这个数字的意义在于：它是面试中"你的 RTO 是多少"这个问题的实测答案，而不是估算。有实测数据支撑的 RTO 比"大约 XX 分钟"可信度高很多。

## 遗留问题 / 下一步建议

- **OSS bucket 生命周期策略需手动配置**：v1.7 方案建议在 OSS 控制台对 `docker-infra/mysql/ruoyi/` 前缀设置 30 天自动删除（或转低频存储），但这一步在 v1.7 实施过程中未纳入 Ansible 自动化（ossutil 可以配置生命周期，但增加了 role 复杂度）。当前状态：OSS 上的备份文件不会自动清理，需要人工定期执行或手动在控制台配置规则。

- **`mysql_backup.log` 未做日志轮转**：cron job 将输出重定向到 `mysql_backup.log`（追加写），ossutil 的进度输出包含 `\r` 重画字符，长期积累后日志文件可读性差。建议后续在备份脚本中改用 `ossutil --loglevel error` 抑制进度输出，或为 cron 日志配置 logrotate。

- **备份失败无告警**：当前备份脚本以 `set -Eeuo pipefail` 运行，失败时脚本退出码非 0，但 cron 只会将错误写入日志文件，没有主动通知。如果磁盘满、MySQL 宕机或 OSS 权限失效，备份静默失败直到人工查阅日志才能发现。建议 v1.9 或独立小任务补充：在脚本末尾检查备份文件年龄（`find ... -newer <threshold>`），若最新备份超过 25 小时未更新则通过飞书告警通知。

- **暂不做 PITR（Binlog 归档）**：将 RPO 从 24h 降到分钟级需要持续归档 binlog + 恢复时重放，复杂度显著上升，留给后续版本评估。当前 bj-01 的 3600s 延迟从库在误操作场景下已提供 1 小时的辅助恢复窗口。

- **恢复演练尚未实现 Jenkins Job 自动化**：当前恢复演练是手动 root shell 命令序列，需要人工触发。后续可考虑新增 Jenkins Job，定期（如每月一次）自动拉 OSS 最新备份、起临时库、校验、记录 RTO，将恢复能力验证纳入持续运维流程。
