## 重要技术决策解释

**Q：为什么废弃“各节点 git pull”，改成 bj-01 上的 Ansible 控制节点？**

A：如果继续让每台机器都 clone 同一个仓库，然后靠人工或脚本在各节点 `git pull`，最大的问题不是“能不能同步”，而是“同步边界不清楚”。每台机器的 `/opt/docker` 都是线上运行目录，里面混有 compose 文件、`.env`、数据目录、证书、Jenkins Home、Prometheus 数据等。一个误操作就可能把运行目录当成源码目录处理，轻则配置漂移，重则覆盖数据。

Ansible 的好处是把“控制面”和“运行面”拆开：bj-01 的 `/opt/docker-infra` 是控制仓库，各节点的 `/opt/docker` 只是部署目标。以后变更只从控制仓库发起，由 role 精准渲染目标文件；哪些文件会被改、哪些服务会重启，都能通过 `--check --diff` 预演。这比“到每台机器上拉代码”更适合运维配置管理。

**Q：为什么控制仓库放在 `/opt/docker-infra`，而不是继续放在 `/opt/docker`？**

A：`/opt/docker` 已经是线上运行目录，不只是“代码目录”。例如 bj-01 的 `/opt/docker/cicd/jenkins/data` 里有 Jenkins 配置和密钥，MySQL/Redis/Grafana 也有各自的数据目录。把 Git 仓库放在这里，会让人天然想在运行目录里做 `git pull`、`git reset`、`git clean` 这类源码仓库操作，风险很高。

`/opt/docker-infra` 的价值是明确告诉自己：这里是配置管理的控制仓库，不直接承载线上数据。Ansible 的 `docker_root: /opt/docker` 再明确告诉系统：渲染结果写到哪里。一个是“源”，一个是“目标”，边界清楚后，排障和审计都会简单很多。

**Q：为什么密码放进 ansible-vault，而 Jenkins 又用 Credentials 管理 SSH key 和 vault 密码？**

A：如果把 `.env` 或 vault 密码直接放进 Git，任何能读仓库的人都能拿到生产密码；如果只放在某台机器上，又会导致部署流程依赖人工记忆。ansible-vault 解决的是“敏感变量可以进入仓库但必须加密”的问题，Jenkins Credentials 解决的是“自动化执行时如何安全拿到解密材料”的问题。

这次最终形成了两层保护：`vault/secrets.yml` 是密文，GitHub 上看不到真实密码；Jenkins Pipeline 运行时通过 `ansible-vault-pass` Secret file 和 `ansible-ssh-key` SSH 私钥临时注入，日志中也被 Jenkins mask。这样既能自动化，又不把核心秘密写进 `Jenkinsfile` 或普通配置文件。

**Q：为什么 Pipeline 先跑 `--check --diff`，再跑正式 Deploy？**

A：Ansible 的危险点不是“它会不会执行”，而是“它会执行什么”。先跑 `--check --diff` 相当于在 Jenkins 日志里留下一份变更预览：哪些文件将改变、是否会触发 handler、是否有 failed。对于 MySQL、Redis、Jenkins 这种有状态或关键服务，这个预演很重要。

本次最终 Jenkins #3/#4 都显示 Deploy 阶段 `changed=0 failed=0`，说明当前仓库状态与四节点实际状态一致。以后如果某次提交导致 `changed` 非 0，也能在日志里直接看到原因，再判断是否符合预期。

**Q：为什么 Webhook 优先于 Poll SCM？**

A：Poll SCM 是 Jenkins 定期去问 GitHub“有没有变化”，简单但有延迟，还会产生无意义请求。Webhook 是 GitHub 在 push 后主动通知 Jenkins，事件驱动、延迟低，也更符合 CI/CD 的直觉。此次 GitHub ping 返回 `200`，测试提交 `7f943ac` 自动触发 Jenkins #4，证明公网入口、SSL、Jenkins Job 和仓库凭据都已串起来。

## 踩坑记录

| 坑点描述 | 根因 | 解决方法 | 下次规避手段 |
|----------|------|----------|-------------|
| `group_vars/all.yml` 放在仓库根目录时 Ansible 没加载 `docker_root` | 使用 `-i inventory/hosts.yml` 时，Ansible 会优先按 inventory 目录查找 `group_vars`；根目录 `group_vars` 不一定被加载 | 将文件移动到 `inventory/group_vars/all.yml` | Ansible 项目中让 `group_vars` 跟 inventory 放在一起，尤其是使用自定义 inventory 路径时 |
| MySQL `.env` 中的 `MYSQL_ROOT_PASSWORD` 与数据库内部 root 密码不一致 | Docker MySQL 的 `MYSQL_ROOT_PASSWORD` 只在首次初始化空 data 目录时生效；已有 data 目录后修改 `.env` 不会自动改数据库账号密码 | 迁移期间不强行重置 root，改用 ruoyi/exporter 用户验证业务路径 | 对有状态服务区分“容器环境变量”和“数据库内部状态”，不要把 `.env` 当成真实状态唯一来源 |
| `mysqld-exporter.cnf` 权限为 `0600` 导致 exporter 读不到配置 | 文件由 Ansible 写入宿主机，容器内进程用户不一定能读宿主机上 root-only 文件 | 将模板任务 mode 改为 `0644`，提交 `adda9c0` | 需要被容器内非 root 进程读取的挂载配置，权限要按容器用户验证 |
| Prometheus reload handler 访问 `http://127.0.0.1:9090/-/reload` 失败 | Prometheus 的 9090 只在容器内监听，没有映射到宿主机 127.0.0.1:9090 | 改成 `docker exec prometheus wget ... http://127.0.0.1:9090/-/reload`，提交 `5fe253f` | handler 的执行位置是宿主机，服务监听位置可能在容器内；写 handler 前先确认网络命名空间 |
| 在本地 `/opt/docker` 与 bj-01 `/opt/docker-infra` 之间产生认知混淆 | Cursor 工作区和真实控制仓库不是同一个路径；`/opt/docker` 在 bj-01 已不是 Git 仓库 | 后续所有 v1.4 控制仓库操作都以 bj-01 `/opt/docker-infra` 为准 | 每次 git 操作前先看 `pwd` 和 `git status --short`；运行目录和控制仓库不要混用 |
| Jenkins 容器内没有 `ansible` 命令 | 原镜像是 `jenkins/jenkins:lts-jdk21`，只挂载了 Docker CLI，没有安装 Ansible | 构建 `jenkins-ansible:latest`，安装 Ansible、Python、OpenSSH client，并让 Jenkins role 使用新镜像 | CI/CD 不是“宿主机能跑”就行，必须验证“Jenkins 容器内能跑” |
| Docker build 安装包时访问国外源长时间卡住 | 默认 Debian/PyPI 源在当前网络环境下不稳定 | Dockerfile 改用清华 Debian/PyPI 镜像源，构建成功并提交 `9cc4802` | 构建基础镜像时优先考虑当前地域可达的镜像源，并保留 Dockerfile 方便复现 |
| Jenkins 中 Ansible 2.20 报 `community.general.yaml callback plugin has been removed` | `ansible.cfg` 使用了旧写法 `stdout_callback = yaml`，新版 Ansible 已移除对应 callback 插件 | 改为 `stdout_callback = default` + `callback_result_format = yaml`，提交 `d752d52` | 升级工具链后要先跑最小验证，尤其是 Ansible callback、collection 这类兼容性点 |
| Jenkins Deploy 阶段被 `when { branch 'main' }` 跳过 | 普通 Pipeline Job 从 SCM checkout 时分支环境可能是 `origin/main`，不是 Multibranch Pipeline 的 `main` | 增加 `env.GIT_BRANCH == 'origin/main' || env.BRANCH_NAME == 'main'` 条件，提交 `9a83e19` | Jenkins Pipeline 的分支条件要结合 Job 类型验证，不同 Job 类型环境变量不同 |
| `git diff -- Jenkinsfile` 对新文件没有输出 | `Jenkinsfile` 是 untracked 文件，普通 `git diff` 默认不显示未跟踪文件内容 | 用 `sed -n` 查看内容，或先 `git add -N Jenkinsfile` 再 diff | 新文件确认内容时不要只依赖 `git diff`，先看 `git status --short` 判断是否 untracked |

## 操作心得

- **先试点、再扩展，比一次全量跑更安全**：gz-03 试点先暴露了 exporter 权限、变量加载、MySQL root 密码认知等问题，避免这些问题在全节点同时爆发。
- **`--check --diff` 是 Ansible 迁移的刹车系统**：每次正式执行前先看 diff，能把“我以为只改一行”变成“我确认只会改这些文件”。
- **健康验证要走真实业务路径**：直接用 MySQL root 失败并不能说明业务坏了；用 Ruoyi 的 `DB_HOST/DB_PORT/DB_USER/DB_PASS` 经 ProxySQL 查询 `SELECT 1`，才更贴近真实流量。
- **容器内外的 `127.0.0.1` 不是同一个东西**：Prometheus reload 的问题就是典型例子。排查容器服务时要先想清楚命令在哪个网络命名空间执行。
- **文档也会漂移，要随迁移一起修**：`/opt/docker` 与 `/opt/docker-infra`、`group_vars` 路径、Jenkinsfile 内容等都曾在文档中出现旧写法。迁移完成后立即修文档，能减少下一次会话的误操作。
- **控制仓库是 v1.4 后的唯一事实来源**：复盘、runbook、architecture 都应该放在 `/opt/docker-infra/Docs`，旧的 gz-03 `/opt/docker/Docs` 不再适合作为主文档位置。
