## 重要技术决策解释

**Q：Registry 为什么部署在 bj-01 而不是单独节点或 gz-01？**

A：本集群没有多余节点预算，Registry 需要跟随控制节点就近管理。bj-01 已承担 Ansible 控制节点、Jenkins、Alertmanager 等运维支撑角色，磁盘 16G 内存也最充裕；gz-01 是入口网关和监控主节点，资源更紧张。Registry 部署在 bj-01 的另一个好处是：Tailscale 内网中 gz-02/gz-03 拉取镜像时直接走加密隧道，无需开放额外公网端口。

**Q：为什么用 HTTP insecure-registries 而不是为 Registry 配置自签名 TLS？**

A：Registry 只在 Tailscale 内网（100.x.x.x）暴露，节点间流量已经过 WireGuard 加密，HTTP 不会明文泄露凭据。自签名证书需要在所有客户端节点分发 CA、更新 Docker daemon 的 `ca-certs`，维护成本远高于收益。在已有网络层加密的私网环境中，`insecure-registries` 是标准生产选择。

**Q：为什么 `docker_login` 和 `docker pull` 从 `community.docker` 模块改为 `shell` 模块？**

A：调试过程中发现 `community.docker.docker_login`（collection 3.7.0）在 check mode 下会真实发起认证请求，且在正式部署中与 docker Python SDK 7.x 存在兼容性问题（`http+docker` URL scheme 注册机制变化）。改为 `shell` 模块直接调用 `docker login`/`docker pull` CLI，行为完全可预期、不依赖 SDK 版本，并配合 `when: not ansible_check_mode` 在 check mode 下显式跳过，符合 Ansible 对"不能安全 dry-run 的任务"的处理规范。

**Q：Python Docker SDK 为什么通过 pip 而不是 apt 安装？**

A：Ubuntu apt 仓库中的 `python3-docker` 版本为 5.0.3，与系统 `requests` 库（2.28+）存在已知 bug——`requests` 新版本不再自动注册 `http+docker://` scheme，导致 SDK 连接 Docker daemon 时报 `Not supported URL scheme http+docker`。pip 安装 7.x 版本可以绕开此 bug，且通过在 `roles/docker-daemon` 中统一管理 `python3-pip` 安装、`/etc/pip.conf` 镜像源配置和旧 apt 包卸载，保持了幂等性和可追溯性。

**Q：为什么在 docker-daemon role 里管理 `/etc/pip.conf`，而不是在 ruoyi role 里？**

A：`/etc/pip.conf` 是主机级的 pip 基础设施配置，属于"运行 Docker 的节点需要具备的基础环境"，与 ruoyi 应用无关。ruoyi role 只负责应用本身的部署逻辑。将 pip 镜像源和 SDK 安装纳入 `docker-daemon` role，使得未来任何需要 Python Docker SDK 的 role 都能依赖这个已知良态，而不必在各自的 role 里重复处理基础设施前置条件。

**Q：CD Job 为什么用手工参数化触发 `IMAGE_TAG` 而不是自动接收 CI 产出的 tag？**

A：自动触发需要 CI 完成后调用 Jenkins API 触发 CD，涉及鉴权和跨 Job 参数传递，复杂度高。更重要的是，V1.6 将 CI（构建镜像）和 CD（部署上线）分开，中间保留一个人工决策点："新版本就绪"和"上线生产"不一定同步发生。参数化方式同时也天然支持回滚——填写旧版本 `IMAGE_TAG` 即可一键回滚，无需额外回滚机制。

**Q：Smoke Test 为什么通过 Prometheus `probe_success` 而不是直接 curl 若依接口？**

A：Prometheus 的 blackbox-exporter 已持续探测若依入口，`probe_success{service="ruoyi"} = 1` 这个指标完整覆盖了"Nginx → ruoyi 容器"的链路可达性。直接从 Jenkins（bj-01）curl 若依入口只能验证 bj-01 到 gz-01 的网络，无法验证业务链路整体健康；而 Prometheus 是从 gz-01 探测的，覆盖范围更完整。复用现有监控基础设施，不需要在 Jenkins 中额外维护测试用例。

**Q：飞书为什么要设计两个机器人，而不是用同一个？**

A：V1.5 的 `docker-infra-alert` 机器人出于安全要求设置了自定义关键词 `docker-infra`，所有 Prometheus 告警消息中都通过 `prometheus-fs` 模板的"项目：docker-infra"兜底。V1.6 新增了 ruoyi CI Pipeline 飞书通知，消息文本含 `[ruoyi]`，无法通过 `docker-infra` 关键词校验，飞书返回 `code:19024 Key Words Not Found`，但 `curl` 退出码仍为 0，Pipeline 整体显示 SUCCESS——这是静默失败，不容易察觉。根本矛盾在于：**告警机器人需要关键词过滤（防止噪音消息或 webhook 滥用），而 CI/CD 通知内容多样，不适合强制关键词约束**。正确做法是职责分离：`docker-infra-alert` 专用于 Prometheus 告警（保留关键词），Jenkins 通知使用独立的 `jenkins-notify`（不设关键词）。两个机器人加入同一飞书群，在群内天然区分来源，互不影响。

## 操作心得

- **`insecure-registries` 要从"谁需要访问 Registry"而不是"谁跑业务容器"的视角设计**：本次初稿只给 `app_nodes`（gz-02/gz-03）加了配置，漏掉了 bj-01（Registry 宿主机）。Registry 宿主机本身需要 push 镜像、做 GC、做验证测试，同样需要配置。规划时要列清楚所有访问 Registry 的角色。

- **多云环境中"节点初始状态"是变量，不是常量**：gz-03（火山引擎）预置了只在内部可达的 PyPI 镜像，gz-02（腾讯云）预置了腾讯云镜像，bj-01（京东云）状态又不同。"Ansible 管理节点前，节点的基础配置是一致的"这个假设在多云混用时不成立，必须把 pip 源、apt 源、DNS 等基础配置纳入 ansible 声明式管理，而不是依赖云厂商预置。

- **`community.docker` 模块的适用边界比想象的窄**：`community.docker.docker_login` 在 check mode 下实际发起认证请求，在 SDK 版本升级后产生兼容性问题，并且 `no_log: true` 让调试成本加倍。在"能用 CLI 完成的操作"上，`shell` 模块反而更稳定、更可调试、行为更透明。`community.docker` 模块在 `docker_compose_v2`（已有良好 check mode 支持）场景下表现良好，但 login/pull 类操作不一定适合。

- **pip 的配置优先级是"用户级 > 系统级"**：`/etc/pip.conf` 是系统级配置，但 root 用户的 `~/.pip/pip.conf` 或 `~/.config/pip/pip.conf` 会覆盖它。排查时要用 `pip3 config list -v` 查看所有生效配置的来源，而不是只检查 `/etc/pip.conf`。

- **Jenkins `sh` 步骤写可移植脚本的原则**：除非显式声明 `#!/bin/bash`，否则只使用 POSIX sh 语法。参数默认值用 `${VAR:-default}`，不用 `${VAR:=default}`；不用 `[[...]]`（bash-only），用 `[...]`；JSON payload 用 heredoc 写文件再引用。

- **调试敏感 task 的标准流程**：改 `no_log: false` → 只对出问题的节点 `--limit gz-03` 跑 → 定位根因 → 解决后改回 `no_log: true` → commit。这个顺序可以在不暴露凭据到 git 历史的前提下完成完整调试。

- **排查 community.docker 模块失败的两步定位法**：先用 `ansible <host> -m shell -a 'python3 -c "import docker; c = docker.from_env(); print(c.version())"'` 验证 Python SDK 能否独立连通 Docker daemon——若此步失败则是 SDK 安装或 daemon socket 问题，与 Ansible collection 无关；若此步成功再去看 Ansible 模块报错，缩小到 collection 自身调用逻辑。两步之间结论清晰，不会把 SDK 问题、daemon 问题、collection 问题混在一起排查。

- **check mode diff 是核心价值，handler 报错不是问题**：check mode 中 handler 因为依赖前序任务产生的实际文件系统状态而报错，是正常限制，不需要修复。只需确认模板 diff 符合预期，handler 的 check mode 问题可以忽略。

- **参数化 CD Job 比自动触发 CD 更适合当前阶段**：CI 和 CD 之间保留人工决策点，是"快速交付"和"稳定上线"的合理平衡。`IMAGE_TAG` 参数同时赋予了回滚能力——填旧 tag 即可，无需额外 rollback 逻辑。

- **Jenkins `withCredentials` 的 `****` 遮蔽不等于变量有值**：Jenkins 会把所有被 `withCredentials` 绑定的变量名加入遮蔽列表，即使 credential 存储值为空字符串，输出中也会显示 `****`。排查 credential 注入是否成功，应在 `sh` 步骤中打印变量长度（`echo "len=${#VAR}"`），而不是靠观察 `****` 来判断。但反过来也要注意：`****` 非空也不等于后续 curl/命令就会成功，本次 Phase 6 就因此误诊断过——`$FEISHU_URL` 确实正确注入，但 Groovy 三单引号 `\"` 转义破坏了 JSON body 的 shell 引号结构，问题根本不在 credential 而在 Jenkinsfile 写法。

- **排查 `curl exit code 3 (URL malformed)` 的三步标准流程**：不要先猜 URL 内容是否异常。第一步用独立测试 Job 跑 `printf "%s" "$SECRET" \| wc -c` 和 `printf "%s" "$SECRET" \| od -c \| head`，看长度和字节是否符合预期；第二步把 `-sf` 改成 `-v` 跑 curl，看是否真的建立了 TCP/TLS 连接和正确的 HTTP 路径——如果能建连就不是 URL 问题；第三步回到 Jenkinsfile 看 `set -x` trace 中 `-d` 参数的实际结构，JSON 键值缺引号即是 Groovy/shell 转义层被吃的信号。

- **Jenkinsfile 多行字符串的安全写法**：`sh '''...'''` 三单引号 + shell 变量展开（`${VAR}`）组合最稳定，但遇到要包含双引号的多行内容（JSON、YAML、嵌入式脚本）一律用 heredoc 写到临时文件再引用，不要用 `\"` 内联转义——Groovy 三单引号对 `\"` 的处理不可靠，容易把反斜杠吃掉导致 shell 解析破损。动态计算的变量（`GIT_SHORT_SHA`、`IMAGE_TAG`）只在 `script` 块里用 `env.XXX = sh(...returnStdout: true).trim()` 赋值，不要在 `environment` 块里预声明空字符串占位。

- **GitHub webhook 排障从 Recent Deliveries 的 Response 码开始**：webhook 不触发时第一步不是看 Jenkins 日志、也不是调 Jenkins 插件配置，而是看 GitHub Settings → Webhooks → Recent Deliveries 里最近一次投递的 Response 码。200 才表示 Jenkins 真的收到请求；301/302 说明 GitHub 请求被重定向但 GitHub 不跟随；4xx 说明 Jenkins 拒收（凭据/路径/权限）；5xx 说明 Jenkins 处理时出错。不同响应码对应不同排查路径，不区分会绕远路。

- **Groovy 三单引号 `'''...'''` 的反斜杠转义处理范围比想象的广**：Phase 6 踩了 `\"` 被消耗的坑，Phase 8 又踩了 `\n` 被转成真换行的坑。真相是 Groovy 三单引号是"不做 `${...}` 插值、但做 Java 风格反斜杠转义"的字符串——`\n`、`\t`、`\r`、`\"`、`\\`、八进制 `\NNN`、Unicode `\uXXXX` 都会被处理。规避原则：凡是要让 Groovy 之下的程序（shell/Python/sed/awk/JSON 解析器）看到反斜杠的位置，一律在 Groovy 层写 `\\`；只有 `\\` 逃过 Groovy 处理后能变成一个真正的反斜杠传下去。同理，要安全写多行 JSON 用 heredoc、多行 Python 用 `<<'PYEOF'` heredoc（引号 delimiter 使 shell 也不插值）即可绕开 Groovy 转义。

- **Registry GC 在 BuildKit/OCI 时代的安全策略**：`registry garbage-collect --delete-untagged` 在 BuildKit 推送场景是**危险命令**——BuildKit 默认用 OCI image index 封装多平台镜像 + SLSA attestation，child manifest 没有直接 tag，`--delete-untagged` 会把 child manifest 误判为孤儿删除，连带 layer blob 被 GC，镜像从 Registry 上消失但 tag 仍在，pull 时报 `not found`。安全 GC 策略是两段式：① Pipeline 主动通过 DELETE `/v2/<name>/manifests/<digest>` 软删除超出保留数量的旧 tag（tag 减一）；②裸 `registry garbage-collect`（不带 `--delete-untagged`）回收被软删除 manifest 引用的 blob。这样保留 tag 指向的 image index + 所有 child manifest + 所有 layer blob 完整链路。代价是被软删除 manifest 下的 child manifest 会成为"命名孤儿"（几百字节/个），数量可控可忽略。

- **Jenkins agent 容器的文件系统视图和宿主机/其他容器是完全隔离的**：写 Jenkinsfile 时不要自由假设容器能看到宿主机任意路径。查 Jenkins 容器能访问哪些路径：看 `roles/jenkins/templates/docker-compose.yml.j2` 的 `volumes:` 字段。对于未挂载的路径（如 Registry 数据目录），要读/写只能通过三种方式：① `docker exec <other-container>` （运行在目标容器内部）；② `docker run --rm -v <host_path>:<mount> <util-image> <cmd>`（临时工具容器）；③ 改 `docker-compose.yml.j2` 添加 mount 重部署（持久方案）。Phase 8 选了方式①，因为 Registry 容器一直在跑且自身容器内就能看到自己的数据目录，顺手利用最干净。

- **飞书机器人的职责要在设计阶段就拆分，不要等到踩坑再改**：告警机器人和流水线通知机器人的需求天然不同——告警需要关键词过滤防止噪音滥用，CI/CD 通知消息内容多样（含项目名、环境名、版本号等），无法统一到一个关键词下。从 V1.6 后期的教训来看，如果在 Phase 6 设计飞书通知时就意识到 CI（含 `[ruoyi]`）和 CD（含 `[docker-infra]`）的关键词需求冲突，第一天就应该建两个机器人。两个机器人加入同一群，在群内通过名称和 tag（如「机器人」标签下的显示名）天然区分来源，运维体验反而更清晰。

- **`curl` 退出码 0 不等于业务成功**：飞书 webhook 关键词校验失败返回 HTTP 200 + `{"code":19024,"msg":"Key Words Not Found"}`，`curl -f/-sS` 只判断 HTTP 状态码，无法感知业务层拒绝。凡是依赖 `curl` 调用第三方 API 的通知/webhook 场景，需要在 shell 中额外解析响应 body（如 `python3 -c "import sys,json; d=json.load(sys.stdin); sys.exit(0 if d.get('code')==0 else 1)"`）才能真正感知失败；或者在 Phase 验收时手动检查飞书群是否收到消息，不要只看 Pipeline 颜色。

- **`mysql-replica` role 只管 Compose，不管复制通道状态（已修复，commit `a405def`）**：V1.6 上线后 bj-01 的 MySQL 延迟从库（Slave2）出现 `Replica_IO_Running: No` / `Replica_SQL_Running: No`，`MySQLReplicationDown` 告警持续每 4 小时推送飞书。根因是 `mysql-replica` role 只做"目录 + `.env` + `docker-compose.yml` 渲染"三件事，不检查也不恢复 MySQL 复制线程状态；MySQL 重启或历史 `STOP REPLICA` 操作后，Ansible 无法自动恢复复制，告警永不 RESOLVED。修复方式：`START REPLICA` 报 `ERROR 1872`（applier 元数据仓库不一致），因此先执行 `RESET REPLICA ALL` 清掉损坏的元数据后，再 `CHANGE REPLICATION SOURCE TO ... SOURCE_AUTO_POSITION=1, SOURCE_DELAY=3600; START REPLICA;` 重建复制通道（`SQL_Delay=3600` 保持不变）；同步在 `roles/mysql-replica/tasks/main.yml` 追加"确保 compose stack 运行 → 等待 MySQL 可连接 → 读取 `SHOW REPLICA STATUS\G` → 若线程停止则 `START REPLICA`"四个 task，此后 `ansible-playbook --tags mysql-replica` 幂等运行（`ok=7 changed=0 skipped=1`），线程停止时自动恢复。设计边界：`RESET REPLICA ALL` 属于破坏性操作，**不**纳入自动化；如需重建通道（`SHOW REPLICA STATUS\G` 无输出时），仍须人工操作，防止数据目录未初始化时错误接主库导致数据不一致。
