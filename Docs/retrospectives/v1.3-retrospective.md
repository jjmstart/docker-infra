## 本次演进一句话摘要

将 `/opt/docker` 整体纳入 Git 版本控制 + GitHub Private 仓库，并为四个服务的 `.env` 各建立 `.env.example` 模板，搭起了"配置即代码"的最小管理层；服务拓扑与 V1.2 完全相同，仅管理面变化——这套方案中"各节点独立 `git pull` 同步"的部分在 V1.4 全量切换到 Ansible 时被推翻。

## 重要技术决策解释

**Q：为什么 V1.3 要先单独做"git 化"，而不是 V1.2 → V1.4 一步到位上 Ansible？**

A：Ansible 引入需要同时回答三个问题：控制仓库放在哪台节点、运行目录与代码目录怎么分离、敏感信息怎么从 `.env` 迁移到 vault。这三件事任意一件没想清楚，强行一步到位会让"配置管理"这条线长期处于"边改边改"的不稳定状态。

V1.3 的取舍是"先把版本控制这一层定下来"——只解决"配置变更如何被追踪"这一个问题，不触碰部署链路。这样做的好处是：每次配置变更从此都进入 Git 历史，文档与配置开始捆绑在同一个 commit，即便 V1.4 后续把同步方式整个换掉，Git 历史本身是不会丢的，反而成了 V1.4 推翻"多节点 git pull"决策时可以回溯的依据。

代价是 V1.3 → V1.4 的距离比预期短得多，V1.3 这版的"工作流"几乎没机会被真正使用。事后看，这也是一类典型的过渡版本：本身价值不在"稳定运行"，而在"为下一版准备好基础设施"。

**Q：为什么 V1.3 初版选择 `/opt/docker` 作为 Git 仓库根，后来 V1.4 又改成 `/opt/docker-infra`？**

A：V1.3 的判断是"既然要做配置即代码，那运行目录就直接做成 Git 仓库最简单"——少一层路径、少一份同步逻辑。问题在没考虑到 `/opt/docker` 不只是"配置目录"，它同时承载 MySQL/Redis/Grafana 的 data 目录、Jenkins Home、证书文件、SQL 备份等运行时产物。这些东西虽然能用 `.gitignore` 排除，但仓库根本身就在运行目录里，会让人天然产生"在运行目录里做 `git reset`、`git clean` 是正常操作"的错觉，一旦失误就是直接覆盖线上数据。

V1.4 的修正是把控制仓库挪到 `/opt/docker-infra`，且只放在 bj-01 一台节点上。`/opt/docker` 重新回到纯"运行目录"的身份，由 Ansible 渲染下发。这样"源"和"目标"就明确分开了，排障和审计都更简单。`.cursor/rules/00-project-context.mdc` 里"控制仓库 `/opt/docker-infra` 仅由 Ansible 控制节点持有；各节点线上运行目录统一为 `/opt/docker`"这一条约定，就是从 V1.3 这次失败的取舍里反推出来的。

**Q：为什么 V1.3 选择 `.env.example` + 手动维护，而不是一开始就 ansible-vault？**

A：当时的判断是"单人运维 + 4 节点 × 4 服务规模下，手工维护 `.env` 操作最简单"。这个判断在"维护 N 个 `.env` 的瞬时成本"维度上确实成立——`.env.example` 不需要任何额外工具，新节点拉仓库后照着模板填密码就能跑起来。

但这个方案在"扩展性"维度立刻就崩了：每新增一个节点，就要在该节点上手动重建 4 份 `.env`；每次轮换密码，需要同时改 4 个节点的真实 `.env` 文件；密码本身完全脱离 Git，等于自己手里又多了一套需要离线维护的"密码笔记"。V1.4 引入 ansible-vault 之后，这些密码进入加密的 `vault/secrets.yml`，Jenkins 通过 Credentials 注入 vault 密码，整个链路只有一个"vault 密码"是真正离线的秘密。

事后看 V1.3 的这条选型与其说是"错误"，不如说是"对未来扩展性低估"。`.env.example` 在单人单节点起步阶段是合理的；只是 V1.3 距离需要扩展性的那天太近，导致这条决策几乎来不及兑现其"简单"的好处就被替换。

**Q：为什么"Git tag 替代手动维护历史归档文件"这一条决策被 V1.4 及以后完整沿用？**

A：V1.3 之前的版本是靠在 `architecture/v1.0.md`、`v1.1.md` 头部手写"本版本为历史归档"标注、并在 README 索引里手工维护"当前最新 ✓"标记来管理版本生命周期。这种方式有两个问题：① 每次发布要回头改至少两个文件；② "历史归档"标注是说明性文字，不是机器可读的真相，靠人工保证一致性。

V1.3 引入 `git tag -a arch-vX.Y` 后，每个版本对应的代码与文档状态都被 Git tag 固化，`git show arch-v1.2:Docs/architecture/v1.2.md` 可以无损还原当时的快照。这意味着"历史归档"的语义由 Git 本身承载，文档正文不需要再为"自己是不是历史"操心。

这条决策在 V1.4 之后被进一步标准化（参见 `.cursor/rules/60-git-and-commits.mdc` 的"Tag 规范"段），并成为后续 `Docs/architecture/README.md` 索引中"当前最新"行得以稳定维护的前提——只标当前一行，其余行均委托给 Git tag。V1.3 这版的工作流大部分被推翻，但这一条沉淀下来了。

## 踩坑记录

> **本节性质**：本次为补写，无当时实测踩坑记录。下表列出的是 V1.4 周期内回看 V1.3 时识别到的**设计盲点**（design pitfalls），而非 V1.3 实操中遭遇的运行时错误。每条均能在 `retrospectives/v1.4-retrospective.md` 找到对应的反向修正。

| 盲点描述 | 根因 | 在 V1.4 的修正 | 下次规避手段 |
|----------|------|----------|-------------|
| 把 Git 仓库根放在运行目录 `/opt/docker` | 没意识到运行目录同时承载 data/、证书、Jenkins Home 等运行时产物，做版本控制目标会产生"在运行目录 `git reset` 是正常操作"的错觉 | 控制仓库迁到 `/opt/docker-infra` 并只放在 bj-01；`/opt/docker` 回归纯运行目录身份，由 Ansible 渲染 | 引入新工具前先回答"源/目标边界在哪"，源和目标共用一个路径就是危险信号 |
| 多节点独立 `git pull` 作为同步主路径 | 把"代码同步"和"配置部署"等同看待；忽视了多节点 pull 时机不一致导致的配置漂移、人为 reset/clean 误操作风险 | 用 Ansible `playbooks/site.yml` 替代，控制面（push 触发 Jenkins）与运行面（节点执行）解耦；`--check --diff` 提供预演 | 多节点同步必须有"中心化下发 + 预演 + 幂等"三件套，缺一件都不算成熟方案 |
| `.env.example` + 手动维护 `.env` 作为长期方案 | 用"瞬时操作最简单"作为选型标准，忽视了"扩展性 + 密码轮换 + 离线密码笔记"三类隐性成本 | ansible-vault 加密 `vault/secrets.yml`；Jenkins Credentials 注入 vault 密码；密码本身可进 Git（密文形态） | 选型时同时评估"瞬时成本"和"扩展边际成本"，规模刚好低于复杂方案阈值的状态最危险，因为很快就会越过 |
| Phase 6（多节点 clone）未完整执行就被推翻 | 未识别到 V1.3 是过渡版本，按"完整发布版"标准编排了 Phase 6 | runbook 顶部已补充标注"Phase 6 未执行，已由 v1.4 取代" | 识别到"过渡版本"性质时，明确标注哪些 Phase 是"为下一版铺路"，避免实施时按完整版本投入 |

## 操作心得

- **过渡版本也要打 tag**：`arch-v1.2` tag 是 V1.3 期间打的，目的是把 V1.2 的状态固化进 Git。事后证明这个动作的价值远大于 V1.3 本身——V1.3 的同步方案被推翻了，但 V1.2 → V1.3 之间的 commit + tag 链路把这次"小步快跑"的判断与代价完整保留下来，成了后续讲清楚"V1.3 为什么短命"的唯一证据。
- **"先简后繁分两步走"在距离过近时不划算**：V1.3 → V1.4 间隔太短，V1.3 这版几乎没机会兑现"简单"的好处就被替换。一个经验法则是：如果方案 B 已经能预见在 1-2 周内启动，方案 A 就不要单独发版了，直接合并到 B 里做。V1.3 当时如果意识到 Ansible 已经在视野里，更合理的做法是把"git 化"作为 V1.4 的 Phase 0，不单独发版。
- **runbook 校准必须包含"被取代的 Phase"标注**：`runbooks/v1.2-to-v1.3.md` 顶部对 Phase 6 的标注（"Phase 6 未执行，已由 v1.4 取代"）是事后回填的。它的存在让后续读 runbook 的人不会被"完整 6 个 Phase"的章节结构误导，以为这个工作流真的在生产侧跑过。
- **设计盲点回写要老实标注性质**：本份 retrospective 的"踩坑记录"是事后识别的设计盲点，不是当时遭遇的运行时错误。两者的复盘价值不同——前者沉淀"工程判断的进化"，后者沉淀"线上排障的细节"。混在一起会让事后复盘失真。

## 遗留问题 / 下一步建议

- 同步方案被 V1.4 全量替换：见 `retrospectives/v1.4-retrospective.md`，本目录后续不再单列。
- "源 / 目标边界"这一类设计原则的沉淀：V1.3 的"`/opt/docker` 同时做源和目标"教训已在 `.cursor/rules/00-project-context.mdc` 固化（控制仓库 `/opt/docker-infra` 与运行目录 `/opt/docker` 显式分开），后续新增组件时应主动检查是否存在同类边界混淆。
- 本份 retrospective 是补写的占位，未来如再次出现"过渡版本"性质的发布，应在发布当时就同步起草 retrospective，并显式标注其"过渡版本"身份，而不是事后回填——事后回填会让"踩坑记录"的真实性需要专门说明。
