## 重要技术决策解释

**Q：为什么 Alertmanager 部署在 bj-01，而不是 gz-01？**

A：Alertmanager 的核心职责是对外发送告警通知，依赖公网访问飞书机器人 webhook。gz-01 是广州节点，网络出口稳定但多跳；bj-01 是北京节点，本身是 Ansible 控制节点，已有公网出口，且在 V1.4 架构中就承担了延迟从库、Redis 从等运维支撑角色。将 Alertmanager 和 `prometheus-alert` 放在 bj-01，减少了对 gz-01 的依赖，同时避免"监控节点宕机 → 告警也发不出去"的单点风险。

**Q：为什么最初选用 `bougou/alertmanager-webhook-adapter`，后来又切换到 `prometheus-alert`？**

A：飞书机器人的消息格式与 Alertmanager 的 webhook payload 格式不一致，需要做字段映射和消息渲染。最初选用 `bougou/alertmanager-webhook-adapter` 是为了快速验证链路，但 Phase 6 发现它生成的飞书消息格式与当前飞书自定义机器人 API 不兼容，飞书返回 `the msg does not match with specified msgType`。随后切换到 `feiyu563/prometheus-alert`，使用 `type=fs&tpl=prometheus-fs&fsurl=...` 方式发送飞书 V2 卡片，最终手动测试告警成功送达飞书。

**Q：为什么飞书机器人安全策略选"自定义关键词"而不是"签名校验"？**

A：当前 V1.5 使用的告警转发工具不负责飞书 HMAC 签名校验。飞书签名校验要求在请求头携带时间戳和签名，工具侧需要专门适配。"自定义关键词"方案简单有效——只要消息中包含 `docker-infra`，飞书就会放行。Phase 6 验证中，真实告警因为不包含该关键词被飞书返回 `Key Words Not Found` 拦截；手动测试告警在 description 中包含 `docker-infra` 后发送成功。

**Q：为什么告警规则按 node / mysql / blackbox / monitoring 四个文件拆分？**

A：单一规则文件在规则数量增长后难以维护，按监控对象拆分有助于快速定位和修改。`node.yml` 管主机资源告警，`mysql.yml` 管数据库拓扑告警，`blackbox.yml` 管入口可用性告警，`monitoring.yml` 管监控系统自身的健康告警（自监控）。这四类关注点正交，拆开后每个文件职责单一，后续新增规则时也清楚应该放在哪里。

**Q：`inhibit_rules` 里 NodeDown 抑制其他 node 告警的设计意图是什么？**

A：当一台节点宕机（`NodeDown` firing），Prometheus 会同时失去该节点的所有指标，导致磁盘、内存、CPU 等告警也连带触发。如果不做抑制，一次节点宕机会发出 5~6 条告警，大量冗余通知会干扰排障判断。`inhibit_rules` 的逻辑是：如果同一 `instance` 上已经有 `NodeDown` 在 firing，就抑制该 instance 上的其他资源告警，让值班人员只收到最根本的问题通知。

**Q：为什么 Phase 7 前先修改了 `prometheus-fs` 模板，而不是直接开始演练？**

A：飞书机器人启用了"自定义关键词"安全策略，真实告警消息必须包含 `docker-infra` 才会被放行。Phase 6 已证明手动测试告警只要在 description 中包含关键词即可成功，但 Phase 7 的真实规则告警最初并不稳定包含该关键词。因此演练前先在 `prometheus-alert` 的 `prometheus-fs` 模板里加入 `项目：docker-infra`，让 FIRING 和 RESOLVED 通知都具备统一兜底。该修改已通过一条不手动包含关键词的 `ManualFeishuTemplateKeywordTest` 验证成功，并在 Phase 8 纳入 `roles/alertmanager`，由 Ansible 下发模板文件并写入运行时 SQLite DB。

## 踩坑记录

| 坑点描述 | 根因 | 解决方法 | 下次规避手段 |
|----------|------|----------|-------------|
| bj-01 本机执行 Ansible ad-hoc 或 playbook 报 `Permission denied (publickey)` | bj-01 作为控制节点同时也是被管理节点；为了兼容 Jenkins 容器执行，inventory 中不能固化 `ansible_connection: local`，本机手动执行时 Ansible 会尝试 SSH 连接自己 | 本机手动命令临时加 `--connection=local`；Jenkins 执行仍保持默认 SSH | 控制节点自管理时要区分执行环境：Jenkins 走 SSH，本机人工命令临时覆盖 connection，不把 local 写死进 inventory |
| `ansible all -m ping` 对 gz-01/gz-02/gz-03 长时间挂起，没有返回 | Tailscale SSH 的 ACL 策略为 `check` 模式，Ansible 建立 SSH 连接时 Tailscale 要求打开浏览器二次认证，Ansible 无法交互，永久阻塞 | 在 Tailscale 控制台将 SSH ACL 策略从 `action: "check"` 改为 `action: "accept"` | 自动化工具（Ansible / Jenkins）走的 SSH 连接必须用 `accept` 模式；`check` 模式只适合人工交互会话 |
| `ansible -m shell -a 'docker ps --format "{{.Names}}"'` 报模板语法错误 | Ansible 的 `-a` 参数会经过 Jinja2 渲染，`{{.Names}}` 被误判为 Jinja2 表达式，`.Names` 不是合法的变量名因此报错 | 改用 `docker ps \| grep` 过滤，或改用 `-m raw` 模块绕过 Jinja2 渲染 | 在 Ansible ad-hoc 命令中使用含 `{{` 的 shell 语法时，优先用 `raw` 模块或改写为不依赖 Go template 的等价命令 |
| `become: true` 在 bj-01 本机 local 执行下可能触发 sudo 问题 | `--connection=local` 时 Ansible 以当前本机用户执行，sudo 提权依赖本机 sudo 策略；而 Jenkins 远程执行时使用 SSH 登录 root，不走这个路径 | 本机执行时确保当前用户具备所需 sudo 权限，或在需要时使用 `-K` 输入 become 密码；Jenkins 仍使用远程 root SSH | local 连接只作为人工临时覆盖，不能把本机 sudo 假设带入 Jenkins 自动化路径 |
| check mode 下 alertmanager handler 报 `is not a directory` | check mode 只模拟目录创建，并不实际创建目录；handler 被触发后试图访问尚不存在的目录，因而报错 | 认识到这是 check mode 的预期限制，不需要修复；正式部署时目录会被真实创建，handler 正常执行 | check mode 下 handler 依赖前序任务实际产生的文件系统状态时，错误是正常的；只需确认 diff 符合预期，不要把 check mode handler 报错当成真实部署问题 |
| Prometheus API 中 `activeAlertmanagers` 一直为空 | `prometheus.yml` 使用文件级 bind mount，Ansible `template` 模块通过 atomic rename 写文件后产生新 inode，容器内仍读到旧配置；仅执行 `/-/reload` 无法解决 | 重启 Prometheus 容器，使 Docker 重新绑定最新文件；后续计划将文件级挂载改为目录级挂载 | 对会被 Ansible 原子替换的配置文件，优先使用目录级 bind mount，避免容器长期持有旧 inode |
| `curl http://127.0.0.1:9093` 查询 Alertmanager 返回空或无法解析 JSON | Alertmanager 端口绑定到 `100.118.69.78:9093`，没有监听 `127.0.0.1` | 所有本机测试命令改用 `http://100.118.69.78:9093` | runbook 中涉及绑定到指定 Tailscale IP 的服务时，不要默认使用 localhost |
| `bougou/alertmanager-webhook-adapter` 能收到 Alertmanager webhook，但飞书不出消息 | adapter 返回 500，飞书报 `the msg does not match with specified msgType`，说明消息结构与飞书当前 API 不兼容 | 替换为 `feiyu563/prometheus-alert`，Alertmanager webhook URL 使用 `/prometheusalert?type=fs&tpl=prometheus-fs&fsurl=...` | 选择第三方告警 adapter 时必须做真实通道端到端验证，不能只看容器 healthy |
| `prometheus-alert` 初次改造后仍然报 `access token invalid` | webhook URL 使用了错误参数 `type=feishu`，触发了飞书应用 token 逻辑，而不是自定义机器人 webhook 逻辑 | 改为 `type=fs&tpl=prometheus-fs&fsurl={{ alerting_feishu_webhook_url | urlencode }}` | 对第三方工具参数必须以官方文档为准；`feishu` 与 `fs` 在 PrometheusAlert 中是不同通道 |
| 真实告警被飞书返回 `Key Words Not Found` 拦截 | 飞书机器人启用了自定义关键词安全策略，但真实告警内容未包含 `docker-infra` | 手动测试告警加入 `docker-infra` 后发送成功；随后在 `prometheus-fs` 模板中加入 `项目：docker-infra`，并在 Phase 8 通过 Ansible 持久化 | 使用关键词安全策略时，所有生产告警模板都必须稳定包含关键词，且模板兜底应纳入配置管理 |
| Phase 7 前 `prometheus-alert` 容器内没有普通模板目录 | `prometheus-alert` 的模板来自 `/app/db/PrometheusAlertDB.db` 中的 `prometheus_alert_d_b` 表，而不是文件模板目录 | 先使用 `sqlite3` 更新 `prometheus-fs` 的 `tpl` 字段完成演练；Phase 8 改为由 `roles/alertmanager/files/prometheus-fs.tpl` 下发并幂等写入 DB | 对依赖运行时 DB 的第三方工具，发布前要补齐 Ansible 持久化，避免容器重建或镜像升级后退化 |

## 操作心得

- **Tailscale SSH 的认证机制和传统 SSH 不同**：Tailscale SSH 的 `check` 模式认证的是"设备"而非"SSH 用户"，一台设备完成浏览器认证后，该设备上所有 SSH 用户（包括 root）都暂时通过认证。这解释了为什么 `admin-alex` 浏览器认证后 Ansible 的 `root` 连接也能成功。但这个认证是有期限的，不适合自动化流水线，生产环境中应明确区分"人工 SSH"和"自动化 SSH"的认证策略。
- **控制节点入 inventory 是双刃剑**：把 bj-01 自己也放入 inventory 管理，好处是 Alertmanager role 可以像其他节点一样统一下发，坏处是需要额外处理本地连接和免密 sudo 两个问题。每次新增"控制节点自管理"的 role 时，都要预先确认这两点已配置。
- **check mode 是信心来源，不是万能预演**：check mode 能验证模板渲染、变量替换、文件 diff，但对依赖前序任务实际状态的 handler 无能为力。正确的使用方式是：用 check mode 确认"变更内容是否符合预期"，而不是"部署是否会成功"。
- **规则文件的 Jinja2 转义要在写模板时就想清楚**：Prometheus 规则文件里的 `{{ $labels.instance }}` 是 Go template 语法，在 Ansible Jinja2 模板中需要用 `{{ '{{' }}` 转义。这个问题在写模板时容易遗忘，建议写完模板后立即用 check mode 的 diff 输出核对渲染结果，确认 `{{ $labels.xxx }}` 没有被 Jinja2 吞掉。
- **分阶段 commit 让回滚粒度更细**：本次按 Phase 分四次提交（variables、alertmanager role、monitor-stack、playbook），每个 commit 职责单一。如果 Phase 5 部署失败，回退一个 commit 就能恢复 playbook 到部署前状态，不需要一次性回退所有变更。这比"一次大提交"要安全得多。
- **healthy 不等于端到端可用**：`prometheus-alert` 容器 healthy、Alertmanager 也能成功 POST，但飞书仍可能因为参数、消息格式或关键词策略拒绝消息。通知链路必须以"飞书群真实收到消息"作为验收标准。
- **第三方告警工具的通道名不能靠猜**：PrometheusAlert 中 `type=feishu` 和 `type=fs` 语义不同，前者不是飞书自定义机器人 webhook 通道。集成第三方工具时，URL path、type、tpl、token 参数都要按官方文档逐项验证。
- **飞书关键词策略会影响真实告警，不只影响测试消息**：直连测试和手动测试容易带上 `docker-infra`，但规则触发的真实告警可能不包含该关键词。最终选择在统一的 `prometheus-fs` 模板中加入 `项目：docker-infra`，并由 Ansible 持久化，避免逐条规则重复维护关键词。
- **故障演练要同时验证 FIRING 和 RESOLVED**：只收到故障通知还不能说明闭环完整。Phase 7 中每类演练都等待恢复通知，确认 `send_resolved: true`、Alertmanager resolved 状态和飞书发送链路都正常。
- **silence 验证要看 Alertmanager 状态而不是只看飞书**：Phase 7 中 `ManualFeishuNotificationTest` 告警被 silence 命中后，Alertmanager API 中状态为 `suppressed` 且 `silencedBy` 指向 silence ID，同时 `prometheus-alert` 日志无发送记录。这样才能证明是 Alertmanager 抑制生效，而不是通知链路偶发未发送。
- **运行时修复不等于配置管理完成**：`prometheus-fs` 模板关键词兜底最初是在容器 SQLite DB 中手工修复的，功能上已通过演练；Phase 8 已将模板文件、compose 挂载和 DB 写入纳入 `roles/alertmanager`，让该修复具备可重建性。
- **Go template 文件要进 `files/` 而不是 `templates/`**：`prometheus-fs` 模板包含大量 `{{ ... }}` Go template 语法，如果放进 Ansible `templates/` 会被 Jinja2 解析破坏；本次直接放在 `roles/alertmanager/files/prometheus-fs.tpl` 用 `copy` 模块下发，绕开了二次渲染。后续遇到第三方工具的模板原样下发时都应优先用 `files/`。
- **`flush_handlers` 是表达"先重启再操作"语义的正确手段**：当 task 需要在容器重建后立即执行（例如 `docker exec` 写运行时数据），不要依赖 task 默认顺序，而要在 task 列表中显式插入 `meta: flush_handlers`，这样 handler 行为就成为 role 内部的硬约束，不会因为后续新增 task 顺序变化而失效。
- **本机控制节点的最终幂等确认要分两段做**：当 inventory 中既有远端节点又有控制节点本身时，不要试图用 `--connection=local` 一把覆盖；远端节点保持默认 SSH，本机用 `--limit bj-01 --connection=local`，两段分别确认 `changed=0 failed=0` 才算闭环。

- **Prometheus 配置改用目录级 bind mount**：当前 `prometheus.yml` 文件级挂载导致 Ansible 原子替换后 inode 不一致，需要 `force-recreate` 才能生效；后续把 `roles/monitor-stack/templates/docker-compose.yml.j2` 改为目录级挂载，`/-/reload` 即可直接生效，并能减少演练后清理临时目标时的复杂度。
- **Ansible playbook 的 vault 引入方式可以统一**：当前 `setup_monitor.yml` 中只有 `alertmanager` 和 `node-exporter` play 引用了 `vars_files: vault/secrets.yml`，`monitor-stack` play 没有引用（因为不需要敏感变量）。随着 vault 变量增多，可以考虑在 playbook 顶层统一引入 vault，或者在 `ansible.cfg` 中配置 `vault_password_file`，减少手动 `--ask-vault-pass` 的频率。
- **bj-01 的免密 sudo 范围可以收窄**：当前配置是 `NOPASSWD: ALL`，赋予了 `admin-alex` 全量免密 sudo 权限。安全意识提升后，可以收窄为只允许特定命令（如 `docker`、`ansible-playbook`），降低权限溢出风险。
- **考虑为 Alertmanager 增加持久化存储备份**：当前 Alertmanager 的数据目录 `/opt/docker/monitor/alertmanager/data` 存储了 silence 记录和告警状态，未纳入备份体系。如果 bj-01 重启或数据丢失，silence 配置会丢失。后续可考虑将 silence 配置定期导出并纳入 Git 或备份流程。
- **告警阈值需要基于真实数据调优**：当前阈值（CPU > 85%、磁盘剩余 < 15%、内存可用 < 10% 等）是经验值。Phase 7 已验证基础链路，但资源类阈值仍需要结合日常观察继续调优，避免过多误报或漏报。
- **V1.6 可考虑启用飞书机器人签名校验**：当前安全策略是"自定义关键词 `docker-infra`"，依赖模板兜底；V1.6 可评估更换或增强 adapter 以支持飞书 HMAC 签名校验，进一步提高 webhook 通道防伪能力。
